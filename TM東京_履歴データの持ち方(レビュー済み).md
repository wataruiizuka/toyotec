# TM東京 履歴データの持ち方について

## 1. この資料で整理したいこと

この資料では、TM東京プロジェクトにおける「履歴データをどう持つべきか」について、現時点で確認できている資料をもとに、設計方針を整理します。

今回の対象である `AzurePowerBIDB` には、複数のシステムからデータが集約されており、Power BI、Power Automate、Teams通知など、いくつかの用途で利用されていると考えています。

そのため、単純に「全テーブルに履歴用のカラムを追加する」という進め方は、正直あまり良くないと思っています。

理由はシンプルで、テーブルごとにデータの性質が違うからです。

たとえば、アクセスログのように発生した事実をそのまま積み上げるべきデータもあれば、顧客マスタや車両マスタのように、最新値を見たいが、必要に応じて過去の状態も追いたいデータもあります。さらに、受注やリクエスト、TMS/UMS系のように、ステータスや状態が変わっていく業務データもあります。

なので、この資料では「全テーブル一律」ではなく、**データの種類ごとに履歴の持ち方を分ける**前提で整理します。

---

## 2. 先に結論

今回の履歴データ設計では、以下の方針が現実的だと考えています。

1. ログ/イベント系データは、原則として**追記型**で持つ。
2. マスタ系データは、必要に応じて**SCD Type2**、または**最新テーブル+履歴テーブル**で持つ。
3. 受注、リクエスト、TMS、UMSなど状態が変わる業務データは、**最新状態と履歴を分けて管理**する。
4. Power BIには、履歴テーブルをそのまま見せるのではなく、基本的には**最新ビュー**または**集計済みマート**を提供する。
5. 一時テーブル、検証テーブル、バックアップテーブルは、原則として履歴管理の対象外にする。
6. 大容量ログ系データは、保持期間とアーカイブ方針を決めたうえで、日付単位または月単位で管理する。
7. 個人情報を含むテーブルは、履歴化する前に、マスキング、権限、削除要件を必ず確認する。

特に大事なのは、**Power BIで使うためのデータ**と、**監査・再現・調査のために残す履歴データ**を分けて考えることです。

履歴データをそのままPower BIに見せると、レコードが二重・三重に見えたり、`is_current = 1` の条件漏れで集計値が膨らんだりします。なので、Power BI側には基本的に「利用しやすい形」に整えたビューやマートを出すのが良いと思います。

---

## 3. 設計の前提

この資料は、現時点で確認できている以下の情報をもとに整理しています。

* TM東京インフラ構成図
* AzurePowerBIDBテーブル設計書
* テーブル一覧
* 項目定義
* Index定義
* レコード件数情報

一方で、まだ確認できていないこともあります。

ここが確定しないと、テーブルごとの最終的な履歴方式までは決め切れません。

* 各テーブルの業務用途
* 各テーブルの正式な主キー、または業務キー
* データ連携方式が全量なのか、差分なのか
* 更新データ、削除データ、訂正データの扱い
* Power BIで実際に参照しているテーブル
* Power BIで過去時点の再現が必要かどうか
* データ保持期間
* 月次の増分件数
* 個人情報の取り扱い
* 障害時、再実行時の運用

なので、この資料は「最終設計書」というより、まずは先方と会話するための設計方針のたたき台です。

## 4. 現状の見え方

### 4.1 システム構成

現状の構成を見る限り、複数の外部システム、既存システムから、Storage Account、Data Factory、直接連携などを通じて、Azure SQL DBである `AzurePowerBIDB` にデータが集約されている構成だと理解しています。

Azure SQL DBに入ったデータは、Power BIのデータフロー、モデル、レポートで利用され、一部はPower Automate経由でTeams通知などにも使われている可能性があります。

つまり、`AzurePowerBIDB` は単なるDBというより、以下の役割をまとめて担っているように見えます。

* 複数システムからのデータ連携先
* Power BIの参照元
* Power Automateの参照元
* 業務データの中間保存先
* 一部の履歴データ、ログデータの保存先

この状態で履歴化を考える場合、「どこまでをAzure SQL DBで持つのか」「どこから先はStorageや別基盤に逃がすのか」は、最初に整理しておいた方が良いです。

特に、ログ系や件数の大きいテーブルまで何でもAzure SQL DBに持たせ続けると、将来的に性能面、運用面、コスト面でしんどくなる可能性があります。

---

### 4.2 テーブル構成

テーブル設計書上では、以下の構成が確認できています。

| 項目 | 内容 |
| --- | ---: |
| 対象テーブル数 | 201 |
| 項目定義数 | 6,547 |
| 想定件数合計 | 118,065,436 |
| dboスキーマ | 139テーブル |

特に件数が大きいものとして、MAアクセスログ、MAメール行動ログ、fkt_TBNH系テーブルが見えています。

この規模になると、「履歴を持つかどうか」だけではなく、**どう検索されるか、どのくらい保持するか、Power BIからどう参照されるか**まで含めて考える必要があります。

---

### 4.3 現状の課題

現時点で気になっている課題は以下です。

* テーブルごとの月次増分が不明。
* データ保持期間が不明。
* PKが確認できないテーブルが多い。
* Index定義が記載されているテーブルが限定的。
* ログ系、マスタ系、トランザクション系、一時系が混在している。
* 個人情報を含む可能性があるテーブルが存在する。
* Power BI参照用テーブルと履歴管理用テーブルの役割が分かれていない。
* 全量連携か差分連携かがテーブル単位で整理されていない。

このあたりを曖昧にしたまま履歴設計に入ると、後からかなり苦しくなります。

たとえば、主キーが分からない状態でSCD Type2を作ると、同じ業務データを識別できません。更新日時がない状態で差分更新を組むと、何が変わったのかを安定して判断できません。

なので、まずはテーブル分類とキー確認を先にやるべきです。

## 5. 履歴管理の基本方針

### 5.1 全テーブル共通の考え方

履歴管理の対象にするテーブルでは、最低限、以下を追えるようにしたいです。

* どのシステムから来たデータなのか。
* いつ抽出されたデータなのか。
* いつDBに取り込まれたデータなのか。
* どのバッチで取り込まれたデータなのか。
* 再実行しても二重登録されないか。
* 変更前と変更後を追えるか。
* 削除データを検知できる場合、削除として扱えるか。
* Power BI向けには、最新状態または集計済みの状態で安全に出せるか。

ここで重要なのは、履歴テーブルは「ただ過去データを残す箱」ではないということです。

あとから調査できること、再実行できること、Power BIで誤集計しないこと、これらをセットで考える必要があります。

---

### 5.2 共通管理カラム

履歴管理対象テーブルでは、以下の管理カラムを標準として持たせることを推奨します。

| カラム名 | 内容 | 用途 |
| --- | --- | --- |
| load_batch_id | 取込単位を識別するID | 再実行、障害調査 |
| source_system | 連携元システム名 | データ発生元の識別 |
| source_table | 連携元テーブル名、またはファイル名 | 連携元の追跡 |
| source_extracted_at | 元データを抽出した日時 | 抽出時点の確認 |
| source_updated_at | 元システム側の更新日時 | 差分判定 |
| loaded_at | AzurePowerBIDBへ取り込んだ日時 | 取込管理 |
| business_key_hash | 業務キーから作成したハッシュ | 重複判定、結合 |
| record_hash | 履歴判定対象カラムから作成したハッシュ | 変更検知 |
| is_current | 現在有効なレコードか | 最新データ抽出 |
| valid_from | 有効開始日時 | 過去時点再現 |
| valid_to | 有効終了日時 | 過去時点再現 |
| is_deleted | 削除扱いか | 論理削除管理 |
| deleted_at | 削除検知日時 | 削除履歴管理 |

もちろん、すべてのテーブルに全カラムが必要という意味ではありません。

ログ系であれば `is_current` や `valid_from` / `valid_to` は不要なことが多いです。逆に、マスタ系や状態管理系であれば、`business_key_hash`、`record_hash`、`is_current`、`valid_from`、`valid_to` はかなり重要になります。

ポイントは、テーブルごとにバラバラの管理カラムを足すのではなく、**標準ルールを決めたうえで、必要なものだけ採用する**ことです。

## 6. データ種別ごとの履歴の持ち方

## 6.1 ログ/イベント系データ

### 対象例

* MAアクセスログ
* MAメール行動ログ
* アクセスログデータ
* Robot Chat通知履歴
* Power Automate実行履歴
* Teams通知履歴

### 推奨方式

ログ/イベント系データは、原則として**追記型**で持つのが良いです。

この種類のデータは、「何かが発生した」という事実そのものが重要です。なので、基本的には上書き更新せず、発生したレコードをそのまま追加していく方式が向いています。

たとえば、アクセスログやメールクリックログは、過去のログが更新されるというより、イベントが増えていくデータです。この場合、SCD Type2のような履歴管理を入れると、むしろ設計が重くなります。

### 推奨カラム

| カラム | 内容 |
| --- | --- |
| event_id | イベント識別子 |
| event_at | イベント発生日時 |
| source_system | 連携元 |
| loaded_at | 取込日時 |
| load_batch_id | 取込バッチID |
| record_hash | 重複判定用ハッシュ |

### 注意点

MAアクセスログのような大容量テーブルは、日付単位または月単位で管理する前提にした方が良いです。

また、IPアドレス、URL、ユーザー識別子などが含まれる場合、個人情報または準個人情報に近い扱いが必要になる可能性があります。

そのため、以下は事前に確認すべきです。

* 何年分保持するのか。
* DB上に明細を持ち続ける必要があるのか。
* 長期分はStorage Accountなどにアーカイブしてよいか。
* Power BIでは明細を見るのか、集計値だけでよいのか。
* 個人を特定できる項目をマスキングする必要があるか。

### Power BI利用方針

Power BIでは、ログ明細を直接参照させるより、集計済みのテーブルを作る方が良いです。

たとえば、以下のようなマートを用意します。

* 日別アクセス数
* 月別アクセス数
* ユーザー別行動回数
* メール開封数
* メールクリック数
* 施策別反応集計

ログ明細をそのままPower BIで扱うと、データ量が大きくなりやすく、レポート表示も重くなります。なので、分析に必要な粒度まで落としてからPower BIに渡す方が安全です。

---

## 6.2 マスタ系データ

### 対象例

* 顧客マスタ
* 店舗マスタ
* 車両マスタ
* 商品/車名マスタ
* 社員マスタ
* コード区分マスタ
* Power BIメタ情報

### 推奨方式

マスタ系データは、**SCD Type2**、または**最新テーブル+履歴テーブル分離型**を推奨します。

マスタは、最新値だけ見ればよいケースもありますが、過去時点の状態が必要になることもあります。

たとえば、顧客ランク、店舗の所属、社員の部署、車両情報、商品カテゴリなどが変更された場合、「今はどうなっているか」だけでなく、「当時はどうだったか」を見たいケースがあります。

この場合、単純に上書きしてしまうと、過去の状態を再現できません。

### SCD Type2の考え方

SCD Type2では、変更が発生したときに既存レコードを上書きしません。

古いレコードの有効終了日を更新し、新しいレコードを追加します。

例：

| 顧客ID | 会員ランク | valid_from | valid_to | is_current |
| --- | --- | --- | --- | ---: |
| C001 | Silver | 2026-04-01 | 2026-05-31 | 0 |
| C001 | Gold | 2026-06-01 | 9999-12-31 | 1 |

この形にしておくと、最新値も取れますし、過去時点の状態も再現できます。

### 推奨カラム

| カラム | 内容 |
| --- | --- |
| business_key | 業務キー |
| business_key_hash | 業務キーのハッシュ |
| record_hash | 変更検知用ハッシュ |
| valid_from | 有効開始日時 |
| valid_to | 有効終了日時 |
| is_current | 現在有効フラグ |
| source_updated_at | 元システム更新日時 |
| loaded_at | 取込日時 |
| load_batch_id | 取込バッチID |
| is_deleted | 削除フラグ |

### 注意点

SCD Type2を採用するには、以下が必要です。

* 主キー、または業務キーが明確であること。
* 更新日時、または変更検知できる項目があること。
* どの項目の変更を履歴化するか決めること。
* 削除されたデータをどう扱うか決めること。
* Power BIで最新だけ見るのか、過去状態も見るのか決めること。

すべてのマスタをSCD Type2にする必要はありません。

過去状態の再現が不要なマスタであれば、最新テーブルだけでも十分です。逆に、分析や監査で過去状態が必要なマスタは、SCD Type2にした方が後から困りにくいです。

## 6.3 受注/リクエスト/予約系データ

### 対象例

* order_リクエスト情報
* order_受注情報
* order_新車受注
* order_中古車受注
* reserveInfo
* reserveCar

### 推奨方式

受注、リクエスト、予約系データは、業務の性質に応じて方式を分けるべきです。

大きく分けると、以下の2パターンです。

1. 発生事実として蓄積すればよいデータ  
   → 追記型

2. ステータスや内容が更新されるデータ  
   → 最新テーブル+履歴テーブル、またはSCD Type2

受注やリクエストは、登録されたあとに状態が変わる可能性があります。

たとえば、以下のような状態変化が考えられます。

* 受付
* 商談中
* 成約
* キャンセル
* 納車済み
* 失注

このように状態が変わる場合、最新状態だけ分かればよいのか、状態の変化まで追いたいのかで設計が変わります。

### 推奨設計

| テーブル | 用途 |
| --- | --- |
| current_order_request | 現在のリクエスト状態 |
| history_order_request | リクエスト状態の変更履歴 |
| event_order_request_status | 状態変更イベント履歴 |

個人的には、Power BIでよく見るのは `current_` または集計済みマート、状態変化を追うときだけ `history_` や `event_` を見る設計が良いと思います。

### 注意点

`order_リクエスト情報` のようなテーブルには、氏名、住所、電話番号、生年月日などの個人情報が含まれる可能性があります。

履歴化するということは、過去の住所や過去の電話番号も残る可能性があるということです。

そのため、以下は必ず確認した方が良いです。

* 過去住所や過去電話番号を残してよいか。
* Power BIで個人情報を表示する必要があるか。
* Power BI上ではマスキングすべきか。
* データ削除依頼があった場合、履歴データからも削除する必要があるか。
* 個人情報を含む履歴テーブルへのアクセス権限をどうするか。

ここを曖昧にしたまま履歴化すると、後から運用やコンプライアンス面で問題になりやすいです。

## 6.4 TMS/UMS系データ

### 対象例

* tms_tbc_VehicleInfo
* tms_tbc_TransportVehiclePlan
* tms_tbc_FowordRequest
* ums_tbu_Vehicles
* ums_tbu_VehicleDetails
* ums_tbu_VehicleProgressStatus
* ums_tbu_ProcessPlanResults

### 推奨方式

TMS/UMS系は、車両、輸送計画、進捗、処理結果など、状態変化を持つデータが多いと考えています。

そのため、状態管理系は**SCD Type2**、または**状態遷移履歴**で持つのが良いです。

### 推奨設計

| データ内容 | 推奨方式 |
| --- | --- |
| 車両基本情報 | 最新+SCD2 |
| 輸送計画 | 計画変更履歴 |
| 車両進捗 | 状態遷移履歴 |
| 処理結果 | 追記型 |

特に進捗系は、最新状態だけではなく、「いつ、どの状態に変わったか」を見たい可能性があります。

その場合は、SCD Type2よりも、状態変更イベントとして追記していく方が自然なこともあります。

### 注意点

`tmstmp`、`umstmp` のような一時テーブルと思われるものは、原則として履歴管理対象外でよいと思います。

ただし、名前は一時テーブルっぽくても、実際には本番連携の着地点として使われている可能性もあります。

なので、テーブル名だけで判断せず、以下を確認した方が良いです。

* Power BIで参照されているか。
* 本番業務で使われているか。
* 一時的な加工用なのか。
* 障害時の復旧に必要なのか。

---

## 6.5 Power BIメタ情報

### 対象例

* report_meta.t_reports
* report_meta.t_report_views
* report_meta.t_workspaces
* report_meta.t_exclude_users

### 推奨方式

Power BIメタ情報は、**最新テーブル+変更履歴**で持つのが良いと思います。

レポート名、ワークスペース、参照状況、除外ユーザーなどは、運用管理に使われる可能性があります。

ただし、閲覧ログとレポート定義情報は性質が違います。

* 閲覧ログ  
  → 追記型

* レポート定義、ワークスペース情報  
  → 最新+変更履歴、またはSCD Type2

### 推奨設計

| テーブル | 用途 |
| --- | --- |
| current_report_meta | 最新のレポート/ワークスペース情報 |
| history_report_meta | レポート名や所属変更の履歴 |
| fact_report_view | レポート閲覧実績 |

閲覧実績は分析用のファクトとして扱い、レポート定義やワークスペースはマスタとして扱うイメージです。

---

## 6.6 一時/検証/バックアップ系テーブル

### 対象例

* tmp
* temp
* test
* BK
* 前回
* reserveInfoTest
* t_service_daily_report_test
* tmstmp
* umstmp

### 推奨方式

一時、検証、バックアップ系テーブルは、原則として履歴管理対象外で良いと思います。

ここまで履歴化すると、正直かなり運用が重くなります。

### 対象外にした方がよい理由

* 不要なデータ量が増える。
* 正式データと検証データが混在する。
* Power BI集計に誤って使われる可能性がある。
* 運用対象が増えすぎる。
* 設計が複雑になる。
* 障害時の切り分けが難しくなる。

### 例外

ただし、以下に該当する場合は、例外的に履歴管理対象として検討します。

* 実質的に本番連携の着地点になっている。
* Power BIで参照されている。
* 障害時の復旧に必要。
* 監査目的で保持を求められている。
* 一時テーブルという名前だが、実態としては業務テーブルになっている。

なので、名前だけで機械的に除外するのではなく、最終的には利用実態を確認して判断するのが安全です。

## 7. 推奨する論理レイヤー

AzurePowerBIDB内、または周辺のStorage Account/Fabric環境も含めて、以下のような論理レイヤーに分けて考えるのが良いです。

```text
Raw層
  ↓
Staging層
  ↓
Current層
  ↓
History層
  ↓
Mart層
  ↓
Power BI
```

ここで大事なのは、すべての物理テーブルを必ずこの名前で作るという話ではありません。

考え方として、以下の役割を分けたいという意味です。

* 元データを残す場所
* 加工前後を調整する場所
* 最新状態を持つ場所
* 履歴を持つ場所
* Power BI向けに使いやすくする場所

---

## 7.1 Raw層

### 目的

Raw層は、元システムから取得したデータを、できるだけ加工せずに保存する層です。

### 役割

* 再処理用の原本を残す。
* 連携元との差異調査に使う。
* 障害時に再ロードできるようにする。
* ファイル連携の場合、元ファイルを保存する。
* 「DBに入った後に何かおかしくなった」場合の戻り先にする。

### 推奨

大量ログやファイル連携データは、Azure SQL DBだけでなく、Storage Account側にも原本を保存することを推奨します。

理由は、Azure SQL DBにすべての明細を長期保存し続けると、性能面でもコスト面でも重くなる可能性があるためです。

特にログ系は、短期はDB、長期はStorageにアーカイブするような考え方が現実的です。

---

## 7.2 Staging層

### 目的

Staging層は、取り込んだデータを一時的に保持し、型変換、重複排除、キー生成、差分比較を行う層です。

### 役割

* 文字列日付を日付型に変換する。
* 業務キーを生成する。
* `record_hash` を生成する。
* `business_key_hash` を生成する。
* 全量データの場合、前回との差分を検出する。
* 不正データを検出する。
* 本テーブルへ入れる前の品質チェックを行う。

履歴管理では、このStaging層がかなり重要です。

いきなり本テーブルへ入れるのではなく、Stagingで整えてからCurrent/Historyに反映することで、再実行や障害時の切り分けがしやすくなります。

---

## 7.3 Current層

### 目的

Current層は、現在有効なデータだけを保持する層です。

### 役割

* Power BIで高速に最新状態を参照する。
* 業務ユーザーが分かりやすく使える形にする。
* 履歴データの重複を避けて集計する。
* `is_current = 1` の条件漏れによる誤集計を防ぐ。

### 推奨

Power BIは、原則としてCurrent層またはMart層を参照させるのが良いです。

History層を直接見せると、履歴レコードが複数存在するため、集計が膨らみやすくなります。

---

## 7.4 History層

### 目的

History層は、過去の状態や変更履歴を保持する層です。

### 役割

* 過去時点の状態を再現する。
* 監査、証跡として変更履歴を残す。
* データの変更前後を追跡する。
* 削除されたデータも必要に応じて保持する。
* 障害調査時に、いつ何が変わったか確認する。

History層は非常に便利ですが、Power BIの一般利用者にそのまま見せるものではないと思っています。

使うとしても、過去時点分析や監査用の専用ビューを用意した方が安全です。

---

## 7.5 Mart層

### 目的

Mart層は、Power BI用に、集計済み、または分析しやすい形に整えた層です。

### 役割

* 日次集計
* 月次集計
* KPI集計
* レポート専用データセット
* 大容量ログのサマリ
* Power BIで扱いやすい粒度への整形

### 推奨

大容量ログや履歴テーブルをPower BIで直接参照させるのではなく、Mart層で集計したデータを参照させることを推奨します。

Power BI側で重い加工や大量スキャンをさせるより、DB側で必要な粒度に整えてから渡す方が、レポートの性能も運用も安定しやすいです。

---

## 8. 代表的な物理設計パターン

## 8.1 追記型テーブル

### 用途

* アクセスログ
* メール行動ログ
* 通知履歴
* 操作ログ
* 処理実績
* イベントデータ

### 設計イメージ

```sql
CREATE TABLE hist_event_xxx (
    event_key              NVARCHAR(200) NULL,
    event_at               DATETIME2(3)  NULL,
    source_system          NVARCHAR(100) NOT NULL,
    source_table           NVARCHAR(200) NOT NULL,
    source_extracted_at    DATETIME2(3)  NULL,
    loaded_at              DATETIME2(3)  NOT NULL,
    load_batch_id          NVARCHAR(100) NOT NULL,
    record_hash            CHAR(64)      NULL,

    -- 元テーブルの項目
    -- ...
);
```

### 設計ポイント

* 原則として更新しない。
* 同じイベントが再送される可能性がある場合は、`event_key` または `record_hash` で重複排除する。
* 大容量データは `event_at` または `loaded_at` で日次/月次管理する。
* Power BI用には日次/月次集計テーブルを作る。
* 長期保管が必要な場合は、DBではなくStorageへのアーカイブも検討する。

---

## 8.2 SCD Type2履歴テーブル

### 用途

* 顧客属性
* 店舗属性
* 車両属性
* 商品/車名マスタ
* 社員マスタ
* Power BIメタ情報
* 状態を持つ業務データ

### 設計イメージ

```sql
CREATE TABLE hist_master_xxx (
    history_id             BIGINT IDENTITY(1,1) NOT NULL,
    business_key_hash      CHAR(64)      NOT NULL,
    record_hash            CHAR(64)      NOT NULL,

    valid_from             DATETIME2(3)  NOT NULL,
    valid_to               DATETIME2(3)  NOT NULL,
    is_current             BIT           NOT NULL,
    is_deleted             BIT           NOT NULL,

    source_system          NVARCHAR(100) NOT NULL,
    source_table           NVARCHAR(200) NOT NULL,
    source_updated_at      DATETIME2(3)  NULL,
    source_extracted_at    DATETIME2(3)  NULL,
    loaded_at              DATETIME2(3)  NOT NULL,
    load_batch_id          NVARCHAR(100) NOT NULL,

    -- 元テーブルの項目
    -- ...
);
```

### 設計ポイント

* `business_key_hash` で同一業務データを識別する。
* `record_hash` で変更有無を判定する。
* 変更があった場合、古いレコードの `valid_to` を更新し、新しいレコードを追加する。
* 最新レコードは `is_current = 1` で判定する。
* 削除検知できる場合は、`is_deleted = 1` の履歴を残す。
* Power BIには、基本的に `is_current = 1 AND is_deleted = 0` のビューを提供する。

---

## 8.3 最新テーブル+履歴テーブル分離型

### 用途

* Power BIで最新状態を高速参照したいテーブル
* 変更履歴も保持したいテーブル
* TMS/UMSの状態管理系
* 受注/リクエスト系

### 設計イメージ

```text
cur_xxx
  → 現在有効な1件のみ保持

hist_xxx
  → 過去を含めたすべての変更履歴を保持
```

### メリット

* Power BIは `cur_xxx` を見ればよいため、集計がシンプルになる。
* 履歴分析が必要な場合は `hist_xxx` を参照できる。
* 履歴の重複集計を防ぎやすい。
* 業務利用と履歴調査の役割を分けやすい。

### 注意点

`cur_xxx` と `hist_xxx` の整合性を保つ必要があります。

たとえば、履歴側では更新されているのに、最新テーブルが更新されていない、という状態になると、Power BIの値と履歴の値がズレます。

そのため、同じバッチ内でCurrentとHistoryを更新する、または更新順序とリカバリ手順を明確にする必要があります。

---

## 8.4 スナップショット型

### 用途

* 月末時点の状態を残したいテーブル
* 全量連携しかできないテーブル
* 差分判定が難しいテーブル
* Power BIの過去表示値を固定したいテーブル

### 設計イメージ

```sql
CREATE TABLE snapshot_xxx (
    snapshot_date          DATE          NOT NULL,
    source_system          NVARCHAR(100) NOT NULL,
    source_table           NVARCHAR(200) NOT NULL,
    loaded_at              DATETIME2(3)  NOT NULL,
    load_batch_id          NVARCHAR(100) NOT NULL,

    -- 元テーブルの項目
    -- ...
);
```

### 注意点

スナップショット型は便利ですが、データ量が増えやすいです。

そのため、全テーブルに適用するのは避けた方が良いです。

特にMAアクセスログのような大容量テーブルに対して、毎日または毎月スナップショットを取ると、すぐにデータ量が膨らみます。

スナップショット型は、あくまで「過去時点の状態を固定して見たい」テーブルに限定して使うべきです。

---

## 9. 全量連携/差分連携別の設計

## 9.1 全量連携の場合

毎回全件が連携される場合は、以下の処理が必要です。

1. 今回分をStaging層へロードする。
2. 業務キーを作成する。
3. `record_hash` を作成する。
4. 前回のCurrent層と比較する。
5. 新規レコードを追加する。
6. 変更レコードを履歴化する。
7. 今回データに存在しないキーを削除候補として扱う。
8. Current層を最新状態に更新する。

### 注意点

全量連携では、前回存在していて今回存在しないデータを検知できます。

ただし、それが本当に削除なのか、連携漏れなのかは別問題です。

たとえば、連携元の抽出条件が変わっただけ、または一時的に連携漏れが発生しただけでも、DB側から見ると「消えた」ように見えます。

そのため、全量連携で削除検知をする場合は、以下を決める必要があります。

* 1回消えたら削除扱いにするのか。
* 複数回連続で消えた場合に削除扱いにするのか。
* 削除ではなく「欠損候補」として扱うのか。
* 削除扱いにしたデータを後から復活できるようにするのか。

---

## 9.2 差分連携の場合

差分データだけが連携される場合は、以下の処理が必要です。

1. 差分データをStaging層へロードする。
2. Insert/Update/Deleteの区分を確認する。
3. 新規は追加する。
4. 更新は履歴化する。
5. 削除は `is_deleted = 1` として履歴化する。
6. Current層を更新する。

### 注意点

差分連携でDelete情報が来ない場合、削除検知はできません。

この場合は、以下のどちらかを選ぶ必要があります。

* 削除は管理対象外にする。
* 定期的に全量照合を行い、削除を補完する。
* 連携元側で削除フラグを持ってもらう。

差分連携は効率が良いですが、Deleteや訂正データの扱いが曖昧だと、履歴としては不完全になります。

---

## 9.3 新規追加のみの場合

新規追加分しか連携されない場合、更新や削除を正しく管理することはできません。

この場合は、以下のどちらかを選ぶ必要があります。

* 新規追加のみを履歴対象にする。
* 定期的に全量データを取得し、更新/削除を補完する。

ここを無理に履歴管理しようとすると、実態と合わない履歴になります。

なので、新規追加のみの連携であれば、「このデータでは更新/削除は追えない」と明示したうえで設計する方が良いです。

---

## 10. Power BI向け設計

Power BIでは、履歴テーブルをそのまま直接参照させない方が良いです。

理由は、履歴テーブルには同じ業務キーのレコードが複数存在するためです。

たとえば、顧客マスタをSCD Type2で持つ場合、1人の顧客に対して過去分と現在分の複数行が存在します。その状態でPower BI側のフィルタが漏れると、人数や売上が二重集計される可能性があります。

### 推奨方針

* 最新値を見せるレポートはCurrent層を参照する。
* 過去時点を見せるレポートはHistory層を条件付きで参照する。
* 大容量ログはMart層の集計済みテーブルを参照する。
* 履歴テーブルと最新テーブルをPower BI上で混在させない。
* `is_current = 1` のフィルタ漏れによる二重集計を防ぐ。
* Power BI用のビューを作成し、利用者にはビューを公開する。

### ビュー例

```sql
CREATE VIEW vw_current_customer AS
SELECT *
FROM hist_customer
WHERE is_current = 1
  AND is_deleted = 0;
```

### 集計マート例

```sql
CREATE TABLE mart_ma_access_daily (
    access_date     DATE NOT NULL,
    page_url        NVARCHAR(3000) NULL,
    access_count    BIGINT NOT NULL,
    user_count      BIGINT NULL,
    loaded_at       DATETIME2(3) NOT NULL
);
```

Power BIの利用者には、なるべく「どれを見ればよいか」が分かりやすい形で提供するべきです。

履歴テーブルの仕様を理解している前提で使ってもらう設計は、長期運用では事故りやすいと思います。

---

## 11. 運用設計

## 11.1 取込ログ管理

すべての取込処理について、以下のような管理テーブルを作ることを推奨します。

```sql
CREATE TABLE etl_load_batch (
    load_batch_id       NVARCHAR(100) NOT NULL,
    source_system       NVARCHAR(100) NOT NULL,
    source_table        NVARCHAR(200) NOT NULL,
    target_table        NVARCHAR(200) NOT NULL,
    load_started_at     DATETIME2(3)  NOT NULL,
    load_finished_at    DATETIME2(3)  NULL,
    status              NVARCHAR(20)  NOT NULL,
    input_count         BIGINT        NULL,
    insert_count        BIGINT        NULL,
    update_count        BIGINT        NULL,
    delete_count        BIGINT        NULL,
    error_count         BIGINT        NULL,
    error_message       NVARCHAR(MAX) NULL
);
```

### 目的

* どの処理が成功したか分かる。
* 件数差異を確認できる。
* 障害時に再実行判断ができる。
* Power BIの数値変動の原因を追跡できる。
* 連携元、連携先、処理時刻を後から確認できる。

この管理テーブルがないと、障害が起きたときに「どこまで入ったのか」「何件入ったのか」「前回と何が違うのか」が分かりにくくなります。

履歴管理をするなら、取込ログもセットで設計した方が良いです。

---

## 11.2 データ品質チェック

以下のチェックは標準化した方が良いです。

| チェック | 内容 |
| --- | --- |
| 件数チェック | 前回比で異常増減がないか |
| NULLチェック | 必須キーがNULLでないか |
| 重複チェック | 業務キーが重複していないか |
| 型チェック | 日付/数値に変換できるか |
| 参照整合性チェック | マスタに存在しないコードがないか |
| 日付範囲チェック | 未来日/異常過去日がないか |

特に履歴管理では、キーのNULLや重複があると致命的です。

業務キーが不安定な状態で履歴を作ると、同じデータなのに別データとして扱われたり、別データなのに同じデータとして扱われたりします。

なので、履歴テーブルに入れる前に、Staging層で最低限の品質チェックを入れるべきです。

---

## 11.3 再実行設計

取込処理は、同じ日付、同じバッチを再実行しても、データが二重登録されないように設計する必要があります。

この性質を冪等性と呼びます。

### 推奨方針

* `load_batch_id` を必ず付与する。
* Staging層に一度ロードしてから本テーブルへ反映する。
* `business_key_hash` と `record_hash` で重複を判定する。
* 失敗時は対象バッチだけ再実行できるようにする。
* 再実行時の件数差異をログに残す。
* 再実行してもCurrentとHistoryの整合性が崩れないようにする。

再実行設計がない履歴管理は、かなり危険です。

履歴データは積み上がるため、一度二重登録が起きると、Power BIの集計値にも影響します。あとから原因を追うのも大変になります。

---

## 12. 保持期間/アーカイブ方針

テーブルごとに、保持期間を決める必要があります。

| 区分 | 推奨保持方針 |
| --- | --- |
| 最新テーブル | 常時保持 |
| マスタ履歴 | 業務要件に応じて3年/5年/7年など |
| 受注/リクエスト履歴 | 業務/監査要件に応じて保持 |
| アクセスログ | 短期はDB、長期はアーカイブ推奨 |
| メール行動ログ | 分析期間に応じて保持 |
| 通知履歴 | 監査要件に応じて保持 |
| 一時/検証テーブル | 原則短期削除 |

特にMAアクセスログは約6,289万件存在するため、無制限にAzure SQL DB上で保持し続ける設計は避けた方が良いです。

短期分析に必要な期間はDBに残し、長期保管が必要な分はStorage Accountなどにアーカイブする方が現実的だと思います。

また、保持期間は技術だけで決めるものではありません。

以下の観点で先方確認が必要です。

* 業務上、何年分見る必要があるか。
* 監査上、何年分残す必要があるか。
* 個人情報を含む場合、どこまで残してよいか。
* Power BIで過去年を参照する必要があるか。
* 削除依頼が来た場合に履歴まで削除する必要があるか。

---

## 13. テーブル分類案

現時点の設計書から見ると、仮の分類は以下です。

| 分類 | 対象例 | 推奨方式 |
| --- | --- | --- |
| アクセス/メールログ | MAアクセスログ、MAメール行動ログ | 追記型+集計マート |
| 基幹連携系 | fkt_、TBBA、TBBC、TBNH、TBV系 | 要確認。トランザクションは追記、マスタはSCD2 |
| 受注/リクエスト系 | order_リクエスト情報、order_受注情報 | 最新+履歴またはSCD2 |
| 車両/輸送/TMS/UMS系 | tms_、ums_ | 状態遷移履歴またはSCD2 |
| Power BIメタ情報 | report_meta系 | 最新+変更履歴 |
| 通知履歴系 | robot_chat.*notify_his | 追記型 |
| 一時/検証/バックアップ系 | tmp、temp、test、BK、前回 | 原則対象外 |

これはあくまでテーブル名と現時点の情報から見た仮分類です。

最終的には、以下を確認したうえで確定する必要があります。

* 本当にその用途で使われているか。
* Power BIで参照されているか。
* 更新が発生するか。
* 削除が発生するか。
* 過去状態を残す必要があるか。
* 個人情報を含むか。

## 14. 実装優先順位

最初から201テーブルすべてを履歴化しようとすると、たぶんかなり重いです。

なので、段階的に進めるのが良いと思います。

### Phase1: 対象整理

まずは、テーブルを分類します。

* 正式テーブル
* 一時テーブル
* 検証テーブル
* バックアップテーブル
* Power BI参照テーブル
* 業務上重要なテーブル
* 履歴化対象外のテーブル

ここで履歴化対象外を明確にすることがかなり重要です。

対象外を決めないと、すべてが対象になってしまい、設計も実装も運用も重くなります。

### Phase2: キー/更新仕様確認

次に、履歴化対象候補について、以下を確認します。

* 主キー
* 業務キー
* 更新日時
* 削除フラグ
* 全量連携か差分連携か
* 更新データが来るか
* 削除データが来るか
* 過去訂正が発生するか

ここが決まらないと、SCD Type2にするのか、追記型にするのか、スナップショット型にするのかを決められません。

### Phase3: 履歴方式決定

テーブルごとに、以下の方式を決めます。

* 追記型
* SCD Type2
* スナップショット型
* 最新+履歴分離型
* 履歴対象外

この段階で、Power BIに見せるテーブル/ビューも合わせて決めるべきです。

履歴方式だけ決めて、Power BI側の見せ方を後回しにすると、あとからレポート側で苦しくなります。

### Phase4: 共通基盤実装

履歴管理に必要な共通部品を作ります。

* `load_batch` 管理テーブル
* Staging層
* Current層
* History層
* Power BI用ビュー
* 品質チェック
* 再実行設計

ここは最初にちゃんと作った方が良いです。

テーブルごとに個別実装すると、あとから保守が大変になります。

### Phase5: 大容量テーブル対応

最後に、大容量テーブルは個別に設計します。

* MAアクセスログ
* MAメール行動ログ
* fkt_TBNH系
* TMS/UMS大容量テーブル

大容量テーブルは、通常の履歴テーブルと同じ考え方で扱うと重くなりやすいです。

保持期間、日付管理、Index、集計マート、アーカイブをセットで考える必要があります。

## 15. 先方に確認すべき内容

最終設計前に、最低限以下は確認した方が良いです。

1. 各テーブルの業務用途。
2. Power BIで参照しているテーブル。
3. 履歴化対象としたいテーブル。
4. 各テーブルの主キー/業務キー。
5. 全量連携か差分連携か。
6. 更新データが来るか。
7. 削除データが来るか。
8. 過去データ訂正が発生するか。
9. 履歴を何年保持するか。
10. 月次増分件数。
11. 個人情報を含むテーブル。
12. 過去時点の状態を再現する必要性。
13. Power BIで過去時点レポートを再現する必要性。
14. 一時/検証/バックアップテーブルの扱い。
15. 障害時の再実行運用。
16. 長期アーカイブ先としてStorage Accountを使ってよいか。
17. Power BI利用者に公開してよい項目、非公開にすべき項目。
18. 個人情報削除依頼が来た場合の履歴データの扱い。

この確認をせずに実装へ進むと、後から「この履歴は残してはいけなかった」「この過去状態は見たかった」「Power BIの集計値が合わない」といった問題が出やすいです。

## 16. 最終的な提案方針

TM東京の履歴データ設計では、以下を基本方針として提案するのが良いと思います。

* AzurePowerBIDB内の全テーブルを一律に履歴化しない。
* テーブル種別ごとに、履歴の持ち方を分ける。
* ログ/イベント系は追記型で管理する。
* マスタ/状態管理系はSCD Type2、または最新+履歴で管理する。
* Power BI向けには、履歴テーブルではなく、最新ビューまたは集計マートを提供する。
* 大容量ログは保持期間を明確にし、長期保管はアーカイブを検討する。
* 個人情報を含むテーブルは、履歴化前にマスキング、権限、削除要件を確認する。
* 一時/検証/バックアップテーブルは原則として履歴対象外にする。
* `load_batch_id`、`loaded_at`、`source_system`、`business_key_hash`、`record_hash` などの共通管理カラムを標準化する。
* まずはPower BIで利用中、かつ業務上重要なテーブルから優先して履歴設計を進める。

これにより、将来的なデータ増加、Power BIの性能、過去時点の再現性、監査対応、障害時の再実行に対応しやすい、現実的な履歴データ基盤を作れると考えています。

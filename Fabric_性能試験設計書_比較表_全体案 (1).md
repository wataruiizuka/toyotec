# Fabric連携方式別 性能試験設計書 比較表（全体案）

## 0. 本資料の位置づけ

本資料は、PMから依頼されている以下3つのデータ参照ルートについて、性能試験設計書を作成するためのたたき台です。

- 現行の Kudu → ゴールドテーブル → BI
- 現行の Iceberg → ゴールドテーブル → BI
- 新規の Iceberg → Fabric → BI

先方が正式な資料を作成する前提のため、本資料では「資料化しやすい粒度」で、比較観点・試験項目・測定項目・前提条件・確認事項を整理します。

なお、現時点で未確定の情報は断定せず、想定または確認事項として記載しています。

---

## 1. 今回の性能試験の目的

今回の性能試験では、BIがデータを表示するまでの性能について、現行構成と新規Fabric構成を比較します。

目的は、単純に「どの構成が一番速いか」を見ることだけではありません。

実際には、以下を総合的に確認します。

| 観点 | 確認内容 |
|---|---|
| 表示速度 | BIレポートの初回表示、フィルタ変更、ページ切替がどれくらい速いか |
| 安定性 | タイムアウト、エラー、処理失敗が発生しないか |
| データ鮮度 | BIで見えるデータがどれくらい最新状態に近いか |
| 運用性 | ゴールドテーブル作成、更新、監視、再実行が現実的に運用できるか |
| コスト・負荷 | 既存基盤やFabric側にどれくらい負荷がかかるか |

今回の試験結果をもとに、FabricをBI向けデータ参照基盤として採用する価値があるかを判断します。

---

## 2. 比較対象の3ルート

今回比較するルートは以下の3つです。

| No | 比較ルート | 位置づけ | 主な確認ポイント |
|---:|---|---|---|
| 1 | 現行 Kudu → ゴールドテーブル → BI | 現行構成1 | 直近・カレントデータ系を起点にした場合のBI表示性能を確認する |
| 2 | 現行 Iceberg → ゴールドテーブル → BI | 現行構成2 | 履歴・トランザクションデータを起点に、現行方式でBI表示した場合の性能を確認する |
| 3 | 新規 Iceberg → Fabric → BI | 新規構成 | Fabricを経由した場合に、BI表示性能・運用性・拡張性が改善するか確認する |

表記については、資料上は `Iceberg`、`Fabric` に統一するのが望ましいです。

---

## 3. 各ルートの説明

### 3.1 現行 Kudu → ゴールドテーブル → BI

このルートは、Kudu側のデータをもとに、BIで使いやすい形のゴールドテーブルを作成し、BIで表示する構成です。

Kuduは直近・カレントデータの検索や絞り込みに向いているため、最新性の高いデータを扱う用途で使われている想定です。

ただし、FabricからKuduを直接参照する前提ではないため、今回の比較では「現行構成としての基準値」を取得する位置づけになります。

#### 確認したいこと

- 現行のKudu起点ルートで、BI表示にどれくらい時間がかかっているか
- ゴールドテーブル作成にどれくらい時間がかかっているか
- BI表示時にKudu側または中間処理側がボトルネックになっていないか
- 直近データを使う場合の現行性能基準として妥当か

---

### 3.2 現行 Iceberg → ゴールドテーブル → BI

このルートは、Iceberg側の履歴・トランザクションデータをもとに、現行方式でゴールドテーブルを作成し、BIで表示する構成です。

今回のFabric検証ではIcebergが主対象になるため、このルートは新規Fabric構成と比較するうえで重要な現行比較対象になります。

#### 確認したいこと

- 現行方式でIcebergからゴールドテーブルを作成した場合の処理時間
- ゴールドテーブルをBIで参照した場合の表示速度
- データ量が増えた場合に現行方式で耐えられるか
- Fabricを使わなくても十分な性能が出るのか

---

### 3.3 新規 Iceberg → Fabric → BI

このルートは、IcebergデータをFabricから参照し、BIで利用する新規構成です。

想定される実装方式には、主に以下があります。

| 方式 | 内容 | 備考 |
|---|---|---|
| Shortcut方式 | IcebergデータをFabric側にコピーせず、OneLake Shortcutで参照する | データ二重持ちを抑えられるが、性能は実測が必要 |
| 物理コピー方式 | IcebergデータをFabric側へコピーし、LakehouseまたはWarehouseで利用する | BI向けに最適化しやすいが、コピー処理とストレージコストが発生する |
| Gold化方式 | Fabric内でBI向けの集計済みテーブルを作成する | 表示速度は上げやすいが、加工処理と運用設計が必要 |

今回のPM依頼では「Iceberg → Fabric → BI」と表現されているため、まずはFabricを経由したBI表示性能を確認する形で整理します。

ただし、Fabric内でShortcut参照するのか、物理コピーするのか、Goldテーブルを作るのかによって性能特性が変わるため、試験前に方式を明確にする必要があります。

## 5. 性能試験で揃えるべき共通条件

3ルートを公平に比較するため、以下の条件はできる限り揃える必要があります。

| 項目 | 揃える内容 |
|---|---|
| 対象BIレポート | 同じレポート、同じページ、同じビジュアルを使う |
| 対象データ範囲 | 同じ期間、同じ会社、同じ店舗、同じ商品条件にする |
| 集計ロジック | 売上、件数、ユーザー数などの計算式を一致させる |
| データ更新タイミング | 可能な限り同じ時点のデータで比較する |
| フィルタ条件 | 初期表示、期間指定、店舗指定などを固定する |
| 実行回数 | 各ケースを複数回実行し、平均・最大・p95を見る |
| キャッシュ条件 | キャッシュあり・なしを分けて記録する |
| 実行時間帯 | 他処理の影響を避けるため、同じ時間帯で実施する |
| 同時実行条件 | 単独実行と複数ユーザー同時実行を分ける |
| 環境情報 | CPU、メモリ、Fabric Capacity、BI設定などを記録する |

特に重要なのは、同じデータ・同じ集計条件で比較することです。
数値が一致しない状態で速度比較をしても、正しい比較になりません。

| 分類 | 測定項目 | 内容 |

## 8. 判定基準の置き方

現時点で具体的な目標秒数が未確定の場合は、以下のように段階的に判定するのが現実的です。

| 判定区分 | 考え方 |
|---|---|
| 数値一致 | まず3ルートで集計結果が一致していることを確認する |
| 現行同等 | 新規Fabric構成が現行Iceberg → Gold → BIと同等以上の表示速度であること |
| 改善あり | Fabric構成が現行より明確に速い、または運用負荷が下がること |
| 要改善 | 表示速度は改善しないが、Gold化や物理コピーで改善余地があること |
| 不採用候補 | 速度・安定性・運用性のいずれも現行より悪化し、改善策も見えないこと |

最終的な合格基準は、先方またはPMと以下を決める必要があります。

- BI初回表示は何秒以内を目標にするか
- フィルタ変更は何秒以内を目標にするか
- エラー率は何％以下を許容するか
- データ反映遅延はどこまで許容するか
- 同時利用者数を何人想定するか

## 9. 試験データの考え方

試験データは、実運用に近い条件で用意する必要があります。

| 項目 | 必要な情報 |
|---|---|
| 対象期間 | 例：1日、1か月、3か月、1年など |
| データ件数 | 明細件数、集計後件数、マスタ件数 |
| データ容量 | テーブルサイズ、ファイルサイズ |
| データ粒度 | 明細、日次、月次、店舗別、商品別など |
| データ分布 | 特定店舗・特定商品・特定期間に偏りがないか |
| 更新頻度 | 日次、時間単位、リアルタイムに近い更新か |
| 機密情報 | 個人情報や機密項目が含まれるか |

データ量だけでなく、データの偏りも性能に影響します。
そのため、可能であれば実業務に近い分布のデータで試験する必要があります。

## 13. PMへ報告する場合の説明文

PMへの報告では、以下のように説明できます。

今回の性能試験では、以下3つのデータ参照ルートを比較対象として整理します。

1つ目は、現行のKuduを起点にゴールドテーブルを作成し、BIで参照するルートです。
これは直近・カレントデータ系の現行基準値を把握するための比較対象です。

2つ目は、現行のIcebergを起点にゴールドテーブルを作成し、BIで参照するルートです。
今回のFabric検証ではIcebergが主対象になるため、新規Fabric構成と比較するうえで重要な現行基準値になります。

3つ目は、IcebergをFabric経由でBIに連携する新規ルートです。
このルートでは、FabricのShortcut、Lakehouse、Warehouse、Direct Lakeなどの活用により、BI表示性能や運用性が改善するかを確認します。

比較では、BIの初回表示時間、フィルタ変更時間、ページ切替時間、Gold作成・更新時間、データ反映遅延、エラー率、リソース使用率を測定します。

ただし、性能比較の前提として、3ルートで同じデータ範囲・同じ集計ロジック・同じBIレポートを使い、集計値が一致していることを確認する必要があります。

また、Fabric構成については、Shortcutで参照するのか、Fabric側へ物理コピーするのか、Fabric内でGold化するのかによって性能が変わります。
そのため、試験前にFabric側の実装方式を確認する必要があります。

最終的には、単純な表示速度だけでなく、データ鮮度、運用負荷、コスト、安定性を含めて、QCDの観点で評価する方針が望ましいです。

| 6 | Power BI接続方式は何か | Import、DirectQuery、Direct Lakeで表示性能が変わるため |
| 7 | 目標レスポンスタイムは何秒か | 合格判定に必要なため |
| 8 | 想定同時利用者数は何人か | ピーク時試験の条件になるため |
| 9 | Fabric Capacityのサイズは何か | Fabric側の性能評価に必須のため |
| 10 | 権限・セキュリティ条件はどうするか | Fabric側で再設計が必要になる可能性があるため |

---

## 15. まとめ

PM依頼に対しては、3つのルートを単純に並べるだけではなく、以下の構成で性能試験設計書を作るのがよいです。

- 比較対象の3ルートを明確にする
- 同じBIレポート、同じデータ範囲、同じ集計ロジックで比較する
- まず集計値が一致することを確認する
- そのうえで、BI表示時間、操作時の応答時間、Gold作成時間、データ反映遅延、エラー率、リソース使用率を測定する
- Fabric構成は、Shortcut、物理コピー、Gold化のどの方式かを明確にする
- 最終判断は速度だけでなく、QCD、つまり品質・コスト・導入運用のしやすさで行う

今回の性能試験は、Fabricが現行構成より必ず速いことを前提にするのではなく、現行Kuduルート、現行Icebergルート、新規Fabricルートを同じ条件で比較し、どの構成がBI用途に最も適しているかを判断するための検証として設計するのが適切です。

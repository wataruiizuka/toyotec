# テスト試験項目確認資料  
## 経路別 BI性能比較

> この資料は、Fabricを実装する際の「テスト試験項目」を確認するためのたたき台です。  
> レビュー結果を踏まえ、試験実施前に確定すべき前提条件、測定定義、記録ID、判定基準案を追加しています。  
> なお、合格基準の秒数や同時実行数などは、実環境・業務要件に依存するため、本資料では仮置きせず、関係者間で合意する前提とします。

---

## 1. 試験目的

本資料は、性能試験を実施する前に、どのテスト試験項目を確認するかを整理するための資料です。

今回の目的は、BI帳票を表示する際に、現行方式とFabric経由方式でどのような観点を測定すべきかを明確にすることです。

比較対象は、以下の3経路です。

- 現行：参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI
- 現行：Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI
- 新規：Goldレイヤーのテーブル（Iceberg） → Fabric → BI

本資料では、性能試験に必要なテスト項目、測定定義、記録項目、判定基準案を整理します。  
ただし、Fabric経由を採用するかどうかの最終判断は、本資料では行いません。  
実測結果、コスト、運用性、障害時の切り戻し方法などを整理したうえで、別途判断する想定です。

---

## 2. 試験対象・前提条件

本試験では、3つの経路で同じBI帳票、同じ対象期間、同じ集計条件を使って比較する想定です。

性能比較を正しく行うためには、まず3経路で表示されるデータが同じ意味を持っている必要があります。  
そのため、表示速度を測る前に、件数、対象期間、主要指標、NULL件数、重複状況、データスナップショットなどを確認します。

### 2.1 比較対象経路

| 区分 | 経路 | 確認したいこと |
|---|---|---|
| 現行1 | 参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI | 参照用DBを起点とした現行経路のBI性能を確認する |
| 現行2 | Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI | Bronzeを起点とした現行Iceberg経路のBI性能を確認する |
| 新規 | Goldレイヤーのテーブル（Iceberg） → Fabric → BI | Fabricを経由した場合のBI性能、安定性、Capacity影響を確認する |

### 2.2 主な前提条件

| 項目 | 内容 |
|---|---|
| 対象帳票 | 同一帳票、または同等条件の帳票を使用する |
| 対象期間 | 3経路で同じ期間に揃える |
| 集計条件 | フィルタ条件、集計粒度、表示項目を揃える |
| 測定方法 | 体感ではなく、秒数・件数・エラー有無などの数値で記録する |
| 測定回数 | 1回だけではなく、複数回実行してばらつきを確認する |
| キャッシュ条件 | 初回表示と再表示を分けて確認する |
| Fabric確認項目 | Capacity使用率、スロットリング、処理待ちの有無を確認する |
| 接続方式 | Fabric接続方式、Power BI接続方式を試験前に確定する |
| 権限条件 | RLS/OLSなど、本番相当の権限条件を試験前に確認する |
| データ同一性 | 同じデータスナップショットで比較できる状態にする |
| Iceberg物理状態 | ファイル数、パーティション、スナップショット、コンパクション状態を記録する |

### 2.3 注意点

今回の比較では、3経路すべてが完全に同じ処理段階を比較しているわけではありません。

現行1は、参照用DB（Kudu）を起点にしてGoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
現行2は、BronzeレイヤーのIcebergテーブルを起点にしてGoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
新規経路は、すでに作成されたGoldレイヤーのIcebergテーブルをFabricに連携し、Fabric経由でBIから参照する流れです。

そのため、結果を比較する際は、単純な表示時間だけで判断しないようにします。  
どの処理段階が比較対象に含まれているか、どこで時間がかかっているか、どのデータ状態を参照しているかを合わせて確認します。

## 3. 試験開始前に確定する項目

性能試験を始める前に、以下の項目を関係者間で確定します。  
ここが曖昧なまま試験を始めると、試験後に「結果をどう判断するか」が分からなくなる可能性があります。

| No | 確定項目 | 内容 | 未確定の場合のリスク |
|---:|---|---|---|
| 1 | 合格基準の数値 | 初回表示、フィルタ変更、クエリ応答、同時実行などについて、何秒以内なら許容するかを決める | 結果が出ても合否判断ができない |
| 2 | Fabric接続方式 | FabricでLakehouse、Warehouse、Shortcut、Direct Lake、DirectQueryなど、どの方式を使うかを明確にする | Fabricの性能評価が曖昧になる |
| 3 | Power BI接続方式 | Import、DirectQuery、Direct Lakeなど、BI側の接続方式を明確にする | BI側の接続方式差が性能差として混ざる |
| 4 | 同時実行数 | 何人相当の同時利用を想定するかを決める | 本番利用時の負荷を再現できない |
| 5 | 業務SLO | 業務上、表示時間やデータ反映時間としてどの程度を求めるかを決める | 「業務上問題ないか」を判断できない |
| 6 | データ同一性の開始ゲート | 3経路で件数、期間、主要指標、checksumなどが一致しているかを確認する | 違うデータを使って速度比較してしまう |
| 7 | 対象帳票・ページ | どのBI帳票、どのページを対象にするかを決める | 試験者によって測る対象が変わる |
| 8 | 代表フィルタ条件 | 期間、店舗、商品など、代表的なフィルタ条件を固定する | フィルタの重さが毎回変わる |
| 9 | タイムアウト秒数 | 何秒以上をタイムアウト扱いにするかを決める | エラー判定が試験者によって変わる |
| 10 | 試行回数 | 各試験を何回実行するかを決める | 平均やp95の信頼性が下がる |

## 4. 優先度の整理

元資料では優先度を「高」「中」で整理していましたが、レビュー結果を踏まえ、試験実施の順番が分かりやすいように P0 / P1 / P2 / P3 に整理します。

| 優先度 | 位置づけ | 対象項目 |
|---|---|---|
| P0 | 試験開始前に必ず確認する開始ゲート | データ正確性、データ量・スキャン量、スナップショット固定、接続方式、合格基準 |
| P1 | MVP試験で必ず実施する項目 | 初回表示、再表示、フィルタ変更、クエリ応答、エラー・タイムアウト、同時実行性能、Fabric Capacity確認 |
| P2 | 原因分析のために必要な項目 | 繰り返し実行安定性、リソース使用量、キャッシュ影響 |
| P3 | 第2段階、または業務要件次第で実施する項目 | ページ切替、データ反映時間、コスト、監視、切り戻し |

※ データ反映時間は、業務上の鮮度要件が強い場合は P3 ではなく P1 として扱います。

## 6. 測定定義

測定者によって値がぶれないように、主要な測定項目の開始点と終了点を定義します。

### 6.1 表示時間の定義

| 項目 | 定義 |
|---|---|
| open_start_at | BI帳票を開く操作を開始した時刻 |
| first_visual_at | 最初のグラフ、表、カードなどのビジュアルが表示された時刻 |
| all_visuals_rendered_at | 対象ページ内のすべてのビジュアルが描画完了した時刻 |
| total_render_seconds | open_start_at から all_visuals_rendered_at までの秒数 |

初回表示の評価では、原則として `total_render_seconds` を使用します。  
ただし、利用者体感を確認するために、`first_visual_at` までの時間も参考値として記録します。

### 6.2 クエリ応答時間の定義

| 項目 | 定義 |
|---|---|
| query_start_at | BI、またはFabric側でクエリが開始された時刻 |
| query_end_at | クエリが完了した時刻 |
| queue_ms | 実行待ちにかかった時間 |
| engine_ms | 実際のクエリ処理にかかった時間 |
| scan_bytes | クエリで読み取ったデータ量 |
| query_id | 実行されたクエリを追跡するためのID |

クエリ応答時間は、BIの画面表示時間とは分けて記録します。  
これにより、画面描画が遅いのか、クエリが遅いのかを切り分けやすくします。

### 6.3 データ反映時間の定義

| 項目 | 定義 |
|---|---|
| source_commit_at | 元データが更新された時刻 |
| gold_ready_at | Goldレイヤーのテーブルに反映された時刻 |
| fabric_ready_at | Fabric側で参照可能になった時刻 |
| bi_queryable_at | BIから更新後データを確認できるようになった時刻 |

データ反映時間は、業務上の鮮度要件がある場合に重点的に確認します。

### 6.4 キャッシュ条件の定義

| cache_profile | 内容 |
|---|---|
| cold | キャッシュが効いていない、または初回実行に近い状態 |
| warm | 同じ条件で再実行し、キャッシュが効いている可能性がある状態 |
| unknown | キャッシュ状態を明確に制御できない状態 |

キャッシュ影響は独立項目として見るだけでなく、各試験結果の属性として記録します。

## 7. 代表ワークロード

高負荷集計や同時実行性能を確認する際は、試験条件が毎回変わらないように、代表ワークロードを事前に定義します。

| workload_id | ワークロード名 | 内容 | 目的 |
|---|---|---|---|
| W1 | 標準帳票表示 | 通常利用に近い期間・粒度で帳票を表示する | 基本性能を確認する |
| W2 | 広範囲集計 | 対象期間を広げて集計する | 大量データ処理時の性能を確認する |
| W3 | 高カーディナリティ集計 | 店舗別、商品別、日別など細かい粒度で集計する | 重い集計条件での性能を確認する |
| W4 | フィルタ高選択 | 絞り込み後の件数が少ない条件で確認する | 軽いフィルタ条件の性能を見る |
| W5 | フィルタ低選択 | 絞り込み後の件数が多い条件で確認する | 重いフィルタ条件の性能を見る |

※ 実際の対象期間、店舗、商品、集計粒度は、試験前に関係者で確定します。

## 8. 試験結果の記録項目

試験結果は、後から比較しやすいように、同じ形式で記録します。

単に「速かった」「遅かった」と書くのではなく、どの試験ケースで、どの経路で、どの条件で、何秒かかったのかを数値で残します。

### 8.1 共通ID

| 記録項目 | 内容 |
|---|---|
| test_case_id | 試験項目を識別するID。例：TC-001 |
| workload_id | 試験条件を識別するID。例：W1、W2 |
| run_id | 1回ごとの試験実行を識別するID |
| query_id | 実行されたクエリを追跡するID |
| snapshot_id | 使用したデータスナップショットを識別するID |
| model_version | Power BIモデル、またはセマンティックモデルのバージョン |
| sql_dax_hash | 使用したSQL、DAX、またはクエリ定義のハッシュ値 |

### 8.2 基本記録項目

| 記録項目 | 内容 |
|---|---|
| 試験日 | 試験を実施した日付 |
| 実施者 | 試験を実施した担当者 |
| 試験項目 | 初回表示、フィルタ変更、クエリ応答など |
| 対象経路 | 参照用DB（Kudu）経由、Bronze（Iceberg）経由、Fabric経由 |
| 対象帳票 | 試験に使用したBI帳票名 |
| 対象ページ | 試験に使用したBIページ名 |
| 対象期間 | BIで指定した集計期間 |
| フィルタ条件 | 店舗、商品、会社、期間などの条件 |
| 実行回数 | 同じ条件で何回実行したか |
| 実行順序 | 経路ごとの実行順。偏りを避けるため記録する |
| cache_profile | cold、warm、unknown のいずれか |
| 表示時間 | BI画面が表示されるまでの時間 |
| クエリ時間 | データ基盤側、またはFabric側のクエリ実行時間 |
| エラー有無 | エラーや表示失敗が発生したか |
| タイムアウト有無 | タイムアウトが発生したか |
| 件数・主要指標 | 3経路で比較する件数、売上、ユーザー数など |
| リソース使用量 | CPU、メモリ、I/O、Fabric Capacity使用率など |
| スキャン量 | 処理対象となったデータ量 |
| Gold作成・更新条件 | Goldレイヤーのテーブルがいつ、どの条件で作成されたか |
| Fabric接続方式 | Fabric経由でBIに接続する方式 |
| Power BI接続方式 | Import、DirectQuery、Direct Lakeなど |
| RLS/OLS条件 | 本番相当の権限条件になっているか |
| 備考 | 気づいた点、異常値、再実行が必要な内容 |

### 8.3 Iceberg物理状態の記録項目

| 記録項目 | 内容 |
|---|---|
| row_count | 対象データの行数 |
| data_bytes | 対象データの容量 |
| scan_bytes | 実際にスキャンしたデータ量 |
| file_count | 対象テーブルのファイル数 |
| partition_info | パーティション条件 |
| snapshot_id | IcebergのスナップショットID |
| compaction_status | コンパクション状態 |
| min_date / max_date | 対象データの最小日付・最大日付 |

## 9. エラー・タイムアウト分類

エラーが発生した場合、単に「エラーあり」と記録するだけでは原因分析が難しくなります。  
そのため、どの層でエラーが発生したかを分類して記録します。

| 記録項目 | 内容 |
|---|---|
| error_code | エラーコード |
| error_message | エラーメッセージ |
| error_layer | エラー発生箇所。BI、認証、接続、Fabric、Capacity、クエリ、データ不整合など |
| timeout_seconds | タイムアウト判定となる秒数 |
| retry_result | 再実行した場合に成功したか |
| reproducible | 同じ条件で再現するか |
| impact | 試験結果への影響 |
| action | 対応方針 |

## 10. 判定基準案

判定基準は、試験前に関係者間で数値を決める必要があります。  
現時点では勝手に数値を置かず、以下のように埋める形式にします。

| 評価項目 | 判定基準案 |
|---|---|
| データ正確性 | 行数、対象期間、主キー一意性、NULL件数、主要指標、checksum が一致していること。差分を許容する場合は事前承認を得る |
| 初回表示 | cold条件の p95 が `<N秒以内>`、かつ現行比の劣化が `<X%以内>` |
| 再表示 | warm条件の p95 が `<N秒以内>`、初回表示との差分を記録する |
| フィルタ変更 | 代表フィルタごとに p95 が `<N秒以内>`、タイムアウトが発生しないこと |
| クエリ応答 | queue_ms、engine_ms、scan_bytes を分けて記録し、ボトルネック層を特定できること |
| エラー・タイムアウト | エラー率が `<X%以下>`、タイムアウト件数が `<N件以下>` |
| 高負荷集計 | W2、W3の条件で p95 が `<N秒以内>`、または遅延理由を説明できること |
| 同時実行性能 | `<Nユーザー>` 同時実行時に p95 が `<N秒以内>`、エラー率が `<X%以下>` |
| Fabric Capacity | スロットリング回数が `<N回以下>`、処理待ち時間が `<N秒以下>`、Capacity使用率が事前合意した範囲内であること |
| データ反映時間 | source_commit_at から bi_queryable_at までの p95 が `<N分以内>` |
| 繰り返し実行安定性 | p50、p95、p99、最大値を記録し、外れ値の理由を説明できること |
| リソース使用量 | CPU、メモリ、I/O、Capacity使用率を記録し、遅延理由の説明に利用できること |
| データ量・スキャン量 | 3経路の row_count、data_bytes、scan_bytes、file_count、snapshot_id が比較可能な状態であること |
| 接続方式 | 試験中にFabric接続方式、Power BI接続方式が変わっていないこと |
| RLS/OLS条件 | 本番相当の権限条件、または試験用に合意された権限条件で実施していること |

## 12. 今後の整理事項

本資料は、テスト試験項目を確認するための資料です。  
ただし、レビュー結果を踏まえ、試験実施前に確定すべき内容を追加しました。

今後、性能試験を実施する前に、以下を関係者間で決定する必要があります。

- 合格基準の数値
- Fabric接続方式
- Power BI接続方式
- 対象帳票、対象ページ
- 代表フィルタ条件
- 同時実行数
- 業務SLO
- データ同一性の開始ゲート
- 試行回数
- タイムアウト秒数
- RLS/OLS条件
- Iceberg物理状態の取得方法
- Fabric Capacityメトリクスの取得方法
- 実行順序と外れ値の扱い

試験結果に対する課題・考察や、Fabric経由を採用するかどうかの判断は、本資料では扱いません。  
試験実施後に、実測結果、性能差の理由、エラー原因、Capacity影響、コスト、運用性を整理したうえで、別途判断する想定です。

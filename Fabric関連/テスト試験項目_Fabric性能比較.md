# テスト試験項目  
## 経路別 BI性能比較

## 1. 試験目的

本試験の目的は、BI帳票を表示する際に、現行方式とFabric経由方式でどの程度の性能差が出るかを確認することです。

比較対象は、以下の3経路です。

- 現行：参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI
- 現行：Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI
- 新規：Goldレイヤーのテーブル（Iceberg） → Fabric → BI

今回の試験では、単純に「BI画面の表示が速いか」だけを見るのではなく、フィルタ操作時の反応、クエリ応答時間、エラーやタイムアウトの有無、データの正確性、データ反映までの時間なども確認します。

また、新規経路ではFabricを経由するため、Fabric Capacityの使用状況やスロットリングの有無も確認します。  
これにより、Fabric経由の構成が、現行方式と比較して実運用に耐えられるかを判断するための材料を整理します。

---

## 2. 試験対象・前提条件

本試験では、3つの経路で同じBI帳票、同じ対象期間、同じ集計条件を使って比較します。

性能比較を正しく行うためには、まず3経路で表示されるデータが同じ意味を持っている必要があります。  
そのため、性能測定の前に、件数、対象期間、主要指標、NULL件数、重複状況などを確認します。

主な前提条件は以下です。

| 項目 | 内容 |
|---|---|
| 比較対象 | 参照用DB（Kudu）経由、Bronze（Iceberg）経由、Fabric経由の3パターン |
| 現行経路1 | 参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI |
| 現行経路2 | Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI |
| 新規経路 | Goldレイヤーのテーブル（Iceberg） → Fabric → BI |
| 対象帳票 | 同一帳票、または同等条件の帳票を使用する |
| 対象期間 | 3経路で同じ期間に揃える |
| 集計条件 | フィルタ条件、集計粒度、表示項目を揃える |
| 測定方法 | 体感ではなく、秒数・件数・エラー有無などの数値で記録する |
| 測定回数 | 1回だけではなく、複数回実行してばらつきを確認する |
| キャッシュ影響 | 初回表示と再表示を分けて確認する |
| Fabric確認項目 | Capacity使用率、スロットリング、処理待ちの有無を確認する |

今回の比較で注意すべき点は、3経路すべてが同じ処理段階を比較しているわけではないことです。

1つ目の現行経路は、参照用DBであるKuduを起点にして、GoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
2つ目の現行経路は、BronzeレイヤーのIcebergテーブルを起点にして、GoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
3つ目の新規経路は、すでに作成されたGoldレイヤーのIcebergテーブルをFabricに連携し、Fabric経由でBIから参照する流れです。

そのため、試験結果を見る際は、単純な表示速度だけではなく、どの処理段階で時間がかかっているのかも整理する必要があります。

## 3. 試験項目

本試験では、以下の15項目をテスト試験項目として設計します。

最初に実施すべき項目は、BI利用者の体感速度や利用可否に直結する項目です。  
次に実施する項目は、安定性、ばらつき、リソース状況、データ量の妥当性など、性能結果の理由を説明するために必要な項目です。

| No | 試験項目 | 確認内容 | 主な測定内容 | 優先度 |
|---:|---|---|---|---|
| 1 | 初回表示 | BI帳票を最初に開いたとき、画面が表示されるまでの時間を確認する | 初回表示時間、エラー有無 | 高 |
| 2 | 再表示 | 同じ帳票を再度表示したときの時間を確認する | 再表示時間、初回との差分 | 高 |
| 3 | フィルタ変更 | 期間、店舗、商品などの条件を変更した後、画面が再描画されるまでの時間を確認する | 再描画時間、クエリ時間 | 高 |
| 4 | クエリ応答 | BIからデータ基盤へ問い合わせたときの応答時間を確認する | SQL実行時間、待機時間、スキャン量 | 高 |
| 5 | データ正確性 | 3経路で表示される件数や主要指標が一致しているか確認する | 件数、売上、ユーザー数、NULL件数、重複件数 | 高 |
| 6 | エラー・タイムアウト | 操作中に表示失敗、クエリエラー、タイムアウトが発生しないか確認する | エラー件数、タイムアウト件数、発生条件 | 高 |
| 7 | 高負荷集計 | 広い期間や細かい粒度で重い集計を行い、性能上の限界を確認する | クエリ時間、表示時間、タイムアウト有無 | 高 |
| 8 | 同時実行性能 | 複数ユーザー、または複数セッションで同時にBIを操作した場合の性能を確認する | 応答時間、エラー率、リソース使用量 | 高 |
| 9 | Fabric Capacity確認 | Fabric経由で処理した場合に、Capacity不足やスロットリングが発生しないか確認する | Capacity使用率、スロットリング有無、処理待ち | 高 |
| 10 | ページ切替 | BI内の別ページへ切り替えたときの表示時間を確認する | ページ切替時間、エラー有無 | 中 |
| 11 | データ反映時間 | 元データ更新後、BIに反映されるまでの時間を確認する | 更新時刻、反映時刻、反映遅延 | 中 |
| 12 | 繰り返し実行安定性 | 同じ操作を複数回実行し、結果のばらつきを確認する | 平均、最大、最小、p95、ばらつき | 中 |
| 13 | リソース使用量 | 各経路でどの程度リソースを使っているか確認する | CPU、メモリ、I/O、Fabric Capacity使用率など | 中 |
| 14 | キャッシュ影響 | 初回実行と2回目以降で性能差がどの程度あるか確認する | 初回時間、再実行時間、差分 | 中 |
| 15 | データ量・スキャン量 | 各経路で処理しているデータ件数や容量が妥当か確認する | 件数、容量、スキャン量、対象期間 | 中 |

## 4. 試験結果の記録項目

試験結果は、後から比較しやすいように、同じ形式で記録します。

特に重要なのは、単に「速かった」「遅かった」と書くのではなく、どの経路で、どの条件で、何秒かかったのかを数値で残すことです。

| 記録項目 | 内容 |
|---|---|
| 試験日 | 試験を実施した日付 |
| 実施者 | 試験を実施した担当者 |
| 試験項目 | 初回表示、フィルタ変更、クエリ応答など |
| 対象経路 | 参照用DB（Kudu）経由、Bronze（Iceberg）経由、Fabric経由 |
| 対象帳票 | 試験に使用したBI帳票名 |
| 対象期間 | BIで指定した集計期間 |
| フィルタ条件 | 店舗、商品、会社、期間などの条件 |
| 実行回数 | 同じ条件で何回実行したか |
| 表示時間 | BI画面が表示されるまでの時間 |
| クエリ時間 | データ基盤側、またはFabric側のクエリ実行時間 |
| エラー有無 | エラーや表示失敗が発生したか |
| タイムアウト有無 | タイムアウトが発生したか |
| 件数・主要指標 | 3経路で比較する件数、売上、ユーザー数など |
| リソース使用量 | CPU、メモリ、I/O、Fabric Capacity使用率など |
| スキャン量 | 処理対象となったデータ量 |
| Gold作成・更新条件 | Goldレイヤーのテーブルがいつ、どの条件で作成されたか |
| Fabric接続方式 | Fabric経由でBIに接続する方式 |
| 備考 | 気づいた点、異常値、再実行が必要な内容 |

## 5. 判定基準

判定基準は、試験前にできるだけ決めておくことが望ましいです。  
基準がないまま結果を見ると、「どこまでなら許容できるのか」が判断しにくくなるためです。

現時点では、以下の考え方を判定基準案とします。

| 評価項目 | 判定基準案 |
|---|---|
| データ正確性 | 3経路で主要指標が一致していること。差分がある場合は、原因を説明できること |
| 初回表示 | 現行方式と同等、または改善していること |
| 再表示 | 初回表示より極端に遅くならないこと |
| フィルタ変更 | 利用者が操作に支障を感じない範囲で再描画されること |
| クエリ応答 | 現行方式と同等、または改善していること |
| 高負荷集計 | 重い条件でもタイムアウトせず、原因分析できる状態であること |
| エラー・タイムアウト | 原則としてエラー、タイムアウトが発生しないこと |
| 同時実行性能 | 同時操作時に極端な性能劣化やエラーが発生しないこと |
| Fabric Capacity | Capacity不足、スロットリング、処理待ちが頻発しないこと |
| データ反映時間 | 業務上必要な鮮度要件を満たしていること |
| 繰り返し実行安定性 | 実行ごとのばらつきが大きすぎないこと |
| リソース使用量 | 特定の経路だけ過度にリソースを消費していないこと |
| データ量・スキャン量 | 3経路で比較条件として妥当なデータ量になっていること |
| Goldレイヤー条件 | 3経路で参照しているGoldレイヤーのデータ条件や更新状態が比較可能であること |
| Fabric経由の妥当性 | Fabricを経由することで、性能・運用性・拡張性の面で現行方式と比較する価値があること |

## 7. 課題・考察

試験結果を確認する際は、数値だけではなく、なぜその結果になったのかを整理します。

特にFabric経由が速かった場合でも、遅かった場合でも、その理由を説明できる状態にしておくことが重要です。

確認すべき主な観点は以下です。

- 参照用DB（Kudu）経由とBronze（Iceberg）経由で、Goldレイヤー作成後のBI性能に差があるか
- Goldレイヤーのテーブル条件が3経路で揃っているか
- Fabric経由が現行方式より速い、または遅い理由は何か
- BI側の処理がボトルネックになっていないか
- データ基盤側のクエリが重くなっていないか
- GoldレイヤーとFabric側で参照条件が揃っているか
- キャッシュの影響で速く見えていないか
- Fabric Capacityが不足していないか
- スロットリングや処理待ちが発生していないか
- 3経路でデータ件数や対象期間に差がないか
- 同時実行時に極端な性能劣化が発生していないか
- データ反映時間が業務要件を満たしているか
- 本番環境でも同じ性能が期待できるか

今回の比較では、新規経路が「Goldレイヤーのテーブル（Iceberg） → Fabric → BI」であるため、Fabricそのものの性能だけでなく、GoldレイヤーからFabricへ連携する部分の影響も確認する必要があります。

また、現行2経路については、どちらも最終的にはGoldレイヤーのIcebergテーブルをBIから参照する形になります。  
そのため、差分が出た場合は、BI表示部分だけではなく、Goldレイヤー作成前のデータ起点やGold作成処理の違いも確認する必要があります。

## 8. 結論・判断材料

本試験の結果をもとに、Fabric経由を今後も検討すべきかを判断します。

判断時には、表示速度だけではなく、データ正確性、安定性、同時実行時の性能、データ反映時間、運用性、Capacity消費の妥当性をあわせて確認します。

最終的には、以下の観点で整理します。

| 判断観点 | 確認内容 |
|---|---|
| 性能 | Fabric経由が現行方式と比べて同等以上か |
| 正確性 | 3経路で主要な集計値が一致しているか |
| 安定性 | エラーやタイムアウトが発生しないか |
| 同時実行性 | 複数ユーザー利用時にも性能が大きく落ちないか |
| 鮮度 | データ更新後、BIに必要なタイミングで反映されるか |
| 運用性 | 障害時に原因調査や再実行が現実的にできるか |
| Fabric Capacity | Fabric経由でCapacity不足やスロットリングが発生しないか |
| コスト妥当性 | Fabric Capacityの消費が性能改善に見合っているか |
| 継続検討可否 | 追加検証に進むべきか、本番適用を検討できるか |

本試験では、Fabric経由を採用するかどうかを最終決定するのではなく、現行方式と比較したうえで、継続検討に値するかを判断するための材料を整理します。

結果としては、以下のいずれかの形でまとめるのが望ましいです。

- Fabric経由は現行方式と同等以上の性能があり、継続検討可能
- Fabric経由は一部条件で性能劣化があるため、追加検証が必要
- Fabric経由はデータ正確性や安定性に課題があり、構成見直しが必要
- 現時点では判断材料が不足しており、対象帳票や接続方式を追加して再検証が必要

以上の観点で整理することで、単なる速度比較ではなく、Fabric導入判断に使える性能試験結果としてまとめることができます。

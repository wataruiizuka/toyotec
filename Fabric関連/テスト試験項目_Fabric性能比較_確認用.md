# テスト試験項目確認資料  
## 経路別 BI性能比較

> この資料は、Fabricを実装する際の「**テスト試験項目**」のたたき台になります。
> 項目については過不足がないかを関係者とともに確認後、決定したほうが良いかと思われます。
> 確認内容としては「テスト項目」と「判定基準」は話し合ったほうが良いかと思われます。
---

## 1. 試験目的

本資料は、性能試験を実施する前に、**どのテスト試験項目を確認するか**を整理するための資料です。

今回の目的は、BI帳票を表示する際に、現行方式とFabric経由方式でどのような観点を測定すべきかを明確にすることです。

比較対象は、以下の3経路です。

- 現行：参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI
- 現行：Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI
- 新規：Goldレイヤーのテーブル（Iceberg） → Fabric → BI

本資料では、まだ試験結果の考察やFabric採用判断までは行いません。  
まずは、実際に性能試験を行う際に必要となるテスト項目、記録項目、判定基準案を整理します。

---

## 2. 試験対象・前提条件

本試験では、3つの経路で同じBI帳票、同じ対象期間、同じ集計条件を使って比較する想定です。

性能比較を正しく行うためには、まず3経路で表示されるデータが同じ意味を持っている必要があります。  
そのため、表示速度を測る前に、件数、対象期間、主要指標、NULL件数、重複状況などを確認します。

### 2.1 比較対象経路

| 区分 | 経路 | 確認したいこと |
|---|---|---|
| 現行1 | 参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI | 参照用DBを起点とした現行経路のBI性能を確認する |
| 現行2 | Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI | Bronzeを起点とした現行Iceberg経路のBI性能を確認する |
| 新規 | Goldレイヤーのテーブル（Iceberg） → Fabric → BI | Fabricを経由した場合のBI性能、安定性、Capacity影響を確認する |

### 2.2 主な前提条件

| 項目 | 内容 |
|---|---|
| 対象帳票 | 同一帳票、または同等条件の帳票を使用する |
| 対象期間 | 3経路で同じ期間に揃える |
| 集計条件 | フィルタ条件、集計粒度、表示項目を揃える |
| 測定方法 | 体感ではなく、秒数・件数・エラー有無などの数値で記録する |
| 測定回数 | 1回だけではなく、複数回実行してばらつきを確認する |
| キャッシュ影響 | 初回表示と再表示を分けて確認する |
| Fabric確認項目 | Capacity使用率、スロットリング、処理待ちの有無を確認する |

### 2.3 注意点

今回の比較では、3経路すべてが完全に同じ処理段階を比較しているわけではありません。

現行1は、参照用DB（Kudu）を起点にしてGoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
現行2は、BronzeレイヤーのIcebergテーブルを起点にしてGoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
新規経路は、すでに作成されたGoldレイヤーのIcebergテーブルをFabricに連携し、Fabric経由でBIから参照する流れです。

そのため、結果を比較する際は、BI表示時間だけでなく、どの処理段階が比較対象に含まれているかを意識する必要があります。

## 3. テスト試験項目一覧

本資料では、テスト試験項目として以下の15項目を整理します。

「最優先」は、BI利用者の体感速度や利用可否に直結するため、初期の性能試験で優先して確認したい項目です。  
「次に実施」は、性能結果のばらつきや原因分析、運用面の確認に必要な項目です。

| No | 試験項目 | 確認内容 | 主な測定内容 | 優先度 |
|---:|---|---|---|---|
| 1 | 初回表示 | BI帳票を最初に開いたとき、画面が表示されるまでの時間を確認する | 初回表示時間、エラー有無 | 最優先 |
| 2 | 再表示 | 同じ帳票を再度表示したときの時間を確認する | 再表示時間、初回との差分 | 最優先 |
| 3 | フィルタ変更 | 期間、店舗、商品などの条件を変更した後、画面が再描画されるまでの時間を確認する | 再描画時間、クエリ時間 | 最優先 |
| 4 | クエリ応答 | BIからデータ基盤、またはFabricへ問い合わせたときの応答時間を確認する | SQL実行時間、待機時間、スキャン量 | 最優先 |
| 5 | データ正確性 | 3経路で表示される件数や主要指標が一致しているか確認する | 件数、売上、ユーザー数、NULL件数、重複件数 | 最優先 |
| 6 | エラー・タイムアウト | 操作中に表示失敗、クエリエラー、タイムアウトが発生しないか確認する | エラー件数、タイムアウト件数、発生条件 | 最優先 |
| 7 | 高負荷集計 | 広い期間や細かい粒度で重い集計を行い、性能上の限界を確認する | クエリ時間、表示時間、タイムアウト有無 | 最優先 |
| 8 | 同時実行性能 | 複数ユーザー、または複数セッションで同時にBIを操作した場合の性能を確認する | 応答時間、エラー率、リソース使用量 | 最優先 |
| 9 | Fabric Capacity確認 | Fabric経由で処理した場合に、Capacity不足やスロットリングが発生しないか確認する | Capacity使用率、スロットリング有無、処理待ち | 最優先 |
| 10 | ページ切替 | BI内の別ページへ切り替えたときの表示時間を確認する | ページ切替時間、エラー有無 | 次に実施 |
| 11 | データ反映時間 | 元データ更新後、BIに反映されるまでの時間を確認する | 更新時刻、反映時刻、反映遅延 | 次に実施 |
| 12 | 繰り返し実行安定性 | 同じ操作を複数回実行し、結果のばらつきを確認する | 平均、最大、最小、p95、ばらつき | 次に実施 |
| 13 | リソース使用量 | 各経路でどの程度リソースを使っているか確認する | CPU、メモリ、I/O、Fabric Capacity使用率など | 次に実施 |
| 14 | キャッシュ影響 | 初回実行と2回目以降で性能差がどの程度あるか確認する | 初回時間、再実行時間、差分 | 次に実施 |
| 15 | データ量・スキャン量 | 各経路で処理しているデータ件数や容量が妥当か確認する | 件数、容量、スキャン量、対象期間 | 次に実施 |

## 4. テスト試験項目の詳細

### 4.1 初回表示

BI帳票を最初に開いたときに、画面が表示されるまでの時間を確認します。

初回表示は、利用者が最初に体感する速度であり、BI利用時の印象に大きく影響します。  
また、初回表示ではキャッシュが効いていない可能性があるため、再表示とは分けて測定します。

確認する内容は以下です。

- 帳票を開いてから表示完了までの時間
- 表示中にエラーが発生しないか
- 初回だけ極端に遅くならないか
- 経路ごとに表示時間に大きな差がないか

---

### 4.2 再表示

同じBI帳票を再度表示した場合の時間を確認します。

再表示では、Power BIやデータ基盤側のキャッシュが効く可能性があります。  
初回表示と再表示を比較することで、キャッシュの影響を把握できます。

確認する内容は以下です。

- 同じ条件で再表示したときの表示時間
- 初回表示との差分
- キャッシュにより極端に速く見えていないか
- 再表示でもエラーが発生しないか

---

### 4.3 フィルタ変更

期間、店舗、商品などのフィルタ条件を変更した後、画面が再描画されるまでの時間を確認します。

実際のBI利用では、利用者が条件を変えながら分析することが多いため、フィルタ変更時の速度は重要です。

確認する内容は以下です。

- フィルタ変更後の再描画時間
- フィルタ変更後に発行されるクエリ時間
- 表示内容が正しく更新されるか
- 条件変更時にタイムアウトが発生しないか

---

### 4.4 クエリ応答

BIからデータ基盤、またはFabricへ問い合わせたときの応答時間を確認します。

画面表示が遅い場合、原因がBI側にあるのか、データ基盤側にあるのか、Fabric側にあるのかを切り分ける必要があります。  
そのため、BI画面の表示時間だけでなく、裏側のクエリ応答時間も記録します。

確認する内容は以下です。

- SQLまたはクエリの実行時間
- 待機時間
- スキャン量
- クエリがタイムアウトしないか
- 経路ごとの応答差がどこで発生しているか

---

### 4.5 データ正確性

3経路で表示される件数や主要指標が一致しているか確認します。

性能を比較する前に、まず同じ条件で同じ結果が出ることを確認する必要があります。  
数値が一致していない状態で速度だけを比較しても、正しい評価にはなりません。

確認する内容は以下です。

- 対象件数が一致しているか
- 対象期間の最小日付、最大日付が一致しているか
- 売上、件数、ユーザー数などの主要指標が一致しているか
- NULL件数に想定外の差がないか
- 重複件数に想定外の差がないか

---

### 4.6 エラー・タイムアウト

操作中に表示失敗、クエリエラー、タイムアウトが発生しないか確認します。

どれだけ表示速度が速くても、エラーが頻発する場合は実運用には向きません。  
特に高負荷条件や同時実行時に、安定して動作するかを確認します。

確認する内容は以下です。

- BI表示エラーの有無
- クエリエラーの有無
- タイムアウトの有無
- エラーが発生した条件
- 再実行すれば成功するのか、毎回失敗するのか

---

### 4.7 高負荷集計

広い期間や細かい粒度で重い集計を行い、性能上の限界を確認します。

通常条件では問題がなくても、対象期間を広げたり、明細に近い粒度で集計したりすると、急に遅くなることがあります。  
そのため、あえて負荷の高い条件でも確認します。

確認する内容は以下です。

- 広い期間を指定した場合の表示時間
- 店舗別、商品別、日別など細かい粒度での集計時間
- 明細に近いデータを扱った場合のクエリ時間
- 高負荷時のタイムアウト有無
- 経路ごとの性能差

---

### 4.8 同時実行性能

複数ユーザー、または複数セッションで同時にBIを操作した場合の性能を確認します。

1人で操作した場合は問題がなくても、複数人が同時に利用すると処理待ちや性能劣化が発生する可能性があります。  
本番利用を想定する場合は、同時実行時の確認が重要です。

確認する内容は以下です。

- 1ユーザー実行時の表示時間
- 5ユーザー相当で同時実行した場合の表示時間
- 10ユーザー以上で同時実行した場合の表示時間
- 同時実行時のエラー率
- Fabric Capacity使用率や処理待ちの有無

---

### 4.9 Fabric Capacity確認

Fabric経由で処理した場合に、Capacity不足やスロットリングが発生しないか確認します。

Fabricを経由する場合、性能はFabric Capacityの状態に影響を受けます。  
そのため、BI表示時間だけでなく、Fabric側のCapacity使用状況もあわせて確認します。

確認する内容は以下です。

- Fabric Capacity使用率
- スロットリングの有無
- 処理待ちの有無
- 同時実行時にCapacity不足が発生しないか
- Fabric経由が遅い場合、Capacityが原因になっていないか

---

### 4.10 ページ切替

BI内の別ページへ切り替えたときの表示時間を確認します。

複数ページ構成の帳票では、ページ切替時にもクエリが発行される可能性があります。  
そのため、初回表示やフィルタ変更だけでなく、ページ切替時の性能も確認します。

確認する内容は以下です。

- ページ切替にかかる時間
- ページ切替時にエラーが発生しないか
- ページによって極端な差がないか
- 重いビジュアルを含むページで遅延が発生しないか

---

### 4.11 データ反映時間

元データが更新された後、BIに反映されるまでの時間を確認します。

BI帳票で最新データを見る必要がある場合、データ反映時間は重要です。  
特に、現行経路とFabric経由で反映タイミングが異なる場合は、業務利用上の影響を確認する必要があります。

確認する内容は以下です。

- 元データの更新時刻
- Goldレイヤーへの反映時刻
- Fabric側への反映時刻
- BI上で確認できるようになる時刻
- 経路ごとの反映遅延

---

### 4.12 繰り返し実行安定性

同じ操作を複数回実行し、結果のばらつきを確認します。

性能試験では、1回だけの結果で判断するのは危険です。  
一時的な環境負荷により、たまたま速い、または遅い結果になる可能性があります。

確認する内容は以下です。

- 複数回実行したときの平均時間
- 最大時間
- 最小時間
- p95
- 実行ごとのばらつき
- 特定の回だけ極端に遅くなっていないか

---

### 4.13 リソース使用量

各経路でどの程度リソースを使っているか確認します。

表示やクエリが遅い場合、原因を判断するためにはリソース使用状況を見る必要があります。  
特にFabric経由では、Capacity使用率やスロットリングの確認が重要です。

確認する内容は以下です。

- CPU使用率
- メモリ使用量
- I/O使用量
- Fabric Capacity使用率
- ネットワーク影響
- 特定の経路だけ過度にリソースを消費していないか

---

### 4.14 キャッシュ影響

初回実行と2回目以降で性能差がどの程度あるか確認します。

BIやデータ基盤では、同じ処理を再実行するとキャッシュにより速くなる場合があります。  
そのため、キャッシュが効いている状態だけで判断しないようにします。

確認する内容は以下です。

- 初回実行時間
- 2回目以降の実行時間
- 初回と再実行の差分
- キャッシュにより速く見えていないか
- キャッシュが効かない条件でも許容できる性能か

---

### 4.15 データ量・スキャン量

各経路で処理しているデータ件数や容量が妥当か確認します。

3経路で処理対象のデータ量が違っている場合、性能比較が不公平になります。  
そのため、表示時間やクエリ時間とあわせて、処理対象の件数やスキャン量を記録します。

確認する内容は以下です。

- 対象件数
- データ容量
- スキャン量
- 対象期間
- 経路ごとに処理対象が揃っているか
- スキャン量が多すぎる場合、SQLやテーブル設計の見直しが必要か

---

## 5. 試験結果の記録項目

試験結果は、後から比較しやすいように、同じ形式で記録します。

単に「速かった」「遅かった」と書くのではなく、どの経路で、どの条件で、何秒かかったのかを数値で残します。

| 記録項目 | 内容 |
|---|---|
| 試験日 | 試験を実施した日付 |
| 実施者 | 試験を実施した担当者 |
| 試験項目 | 初回表示、フィルタ変更、クエリ応答など |
| 対象経路 | 参照用DB（Kudu）経由、Bronze（Iceberg）経由、Fabric経由 |
| 対象帳票 | 試験に使用したBI帳票名 |
| 対象期間 | BIで指定した集計期間 |
| フィルタ条件 | 店舗、商品、会社、期間などの条件 |
| 実行回数 | 同じ条件で何回実行したか |
| 表示時間 | BI画面が表示されるまでの時間 |
| クエリ時間 | データ基盤側、またはFabric側のクエリ実行時間 |
| エラー有無 | エラーや表示失敗が発生したか |
| タイムアウト有無 | タイムアウトが発生したか |
| 件数・主要指標 | 3経路で比較する件数、売上、ユーザー数など |
| リソース使用量 | CPU、メモリ、I/O、Fabric Capacity使用率など |
| スキャン量 | 処理対象となったデータ量 |
| Gold作成・更新条件 | Goldレイヤーのテーブルがいつ、どの条件で作成されたか |
| Fabric接続方式 | Fabric経由でBIに接続する方式 |
| 備考 | 気づいた点、異常値、再実行が必要な内容 |

---

## 6. 判定基準案

判定基準は、試験前にできるだけ決めておくことが望ましいです。  
基準がないまま結果を見ると、「どこまでなら許容できるのか」が判断しにくくなるためです。

現時点では、以下の考え方を判定基準案とします。

| 評価項目 | 判定基準案 |
|---|---|
| データ正確性 | 3経路で主要指標が一致していること。差分がある場合は、原因を説明できること |
| 初回表示 | 現行方式と同等、または改善していること |
| 再表示 | 初回表示より極端に遅くならないこと |
| フィルタ変更 | 利用者が操作に支障を感じない範囲で再描画されること |
| クエリ応答 | 現行方式と同等、または改善していること |
| 高負荷集計 | 重い条件でもタイムアウトせず、原因分析できる状態であること |
| エラー・タイムアウト | 原則としてエラー、タイムアウトが発生しないこと |
| 同時実行性能 | 同時操作時に極端な性能劣化やエラーが発生しないこと |
| Fabric Capacity | Capacity不足、スロットリング、処理待ちが頻発しないこと |
| データ反映時間 | 業務上必要な鮮度要件を満たしていること |
| 繰り返し実行安定性 | 実行ごとのばらつきが大きすぎないこと |
| リソース使用量 | 特定の経路だけ過度にリソースを消費していないこと |
| データ量・スキャン量 | 3経路で比較条件として妥当なデータ量になっていること |
| Goldレイヤー条件 | 3経路で参照しているGoldレイヤーのデータ条件や更新状態が比較可能であること |
| Fabric経由の確認 | Fabricを経由した場合に、性能・安定性・Capacity面で大きな懸念がないこと |

# テスト試験項目確認資料  
## 経路別 BI性能比較

> この資料は、Fabricを実装する際の「テスト試験項目」を確認するためのたたき台です。  
> 現時点では、試験を実施するための最終手順書ではなく、関係者間で「何を試験するか」「何を記録するか」「どの基準で見るか」を確認するための資料です。  
> 実際に性能試験を行う前に、合格基準の数値、Fabric接続方式、同時実行数、データ同一性の確認条件を確定する必要があります。

---

## 1. 試験目的

本資料の目的は、性能試験を実施する前に、どのテスト試験項目を確認するかを整理することです。

今回の性能試験では、BI帳票を表示する際に、現行方式とFabric経由方式でどのような観点を測定すべきかを明確にします。

比較対象は、以下の3経路です。

- 現行：参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI
- 現行：Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI
- 新規：Goldレイヤーのテーブル（Iceberg） → Fabric → BI

本資料では、実際に性能試験を行う際に必要となるテスト項目、記録項目、測定時の注意点、判定基準案を整理します。

なお、この資料だけでFabric経由を採用するかどうかを判断するものではありません。  
Fabric経由の採用判断は、実測結果、コスト、運用性、障害時の切り戻し方針などを整理したうえで、別途判断する想定です。

---

## 2. 試験対象・前提条件

本試験では、3つの経路で同じBI帳票、同じ対象期間、同じ集計条件を使って比較する想定です。

性能比較を正しく行うためには、まず3経路で表示されるデータが同じ意味を持っている必要があります。  
そのため、表示速度を測る前に、件数、対象期間、主要指標、NULL件数、重複状況などを確認します。

---

### 2.1 比較対象経路

| 区分 | 経路 | 確認したいこと |
|---|---|---|
| 現行1 | 参照用DB（Kudu） → Goldレイヤーのテーブル（Iceberg） → BI | 参照用DBを起点とした現行経路のBI性能を確認する |
| 現行2 | Bronzeレイヤーのテーブル（Iceberg） → Goldレイヤーのテーブル（Iceberg） → BI | Bronzeを起点とした現行Iceberg経路のBI性能を確認する |
| 新規 | Goldレイヤーのテーブル（Iceberg） → Fabric → BI | Fabricを経由した場合のBI性能、安定性、Capacity影響を確認する |

---

### 2.2 主な前提条件

| 項目 | 内容 |
|---|---|
| 対象帳票 | 同一帳票、または同等条件の帳票を使用する |
| 対象期間 | 3経路で同じ期間に揃える |
| 集計条件 | フィルタ条件、集計粒度、表示項目を揃える |
| 測定方法 | 体感ではなく、秒数・件数・エラー有無などの数値で記録する |
| 測定回数 | 1回だけではなく、複数回実行してばらつきを確認する |
| キャッシュ影響 | 初回表示と再表示を分けて確認する |
| Fabric確認項目 | Capacity使用率、スロットリング、処理待ちの有無を確認する |
| 接続方式 | Fabric接続方式、Power BI接続方式を事前に明確にする |
| 権限条件 | 本番相当のユーザー権限、RLS/OLSの有無を確認する |
| データ状態 | 3経路で同じデータ時点・同じ対象範囲を比較する |

---

### 2.3 試験前に確定すべき事項

実際に性能試験を行う前に、以下の内容を関係者間で確定する必要があります。

| No | 確定事項 | 理由 |
|---:|---|---|
| 1 | 合格基準の数値 | 「何秒以内なら合格か」を決めないと、結果の判断がぶれるため |
| 2 | Fabric接続方式 | Direct Lake、DirectQuery、Importなどにより性能が変わるため |
| 3 | Power BI側の接続方式 | BI側の方式によって表示速度やキャッシュ条件が変わるため |
| 4 | 同時実行数 | 何人同時利用を想定するかで負荷条件が変わるため |
| 5 | 業務SLO | 業務上、許容できる表示時間や反映時間を決めるため |
| 6 | データ同一性の開始条件 | データが一致していない状態で速度比較をしても意味がないため |
| 7 | 測定対象帳票 | 帳票によってクエリ量や表示時間が変わるため |
| 8 | 代表フィルタ条件 | 軽い条件・重い条件で性能が変わるため |
| 9 | 測定回数 | 1回だけではなく、ばらつきを見る必要があるため |
| 10 | タイムアウト秒数 | エラー判定の基準を揃えるため |

---

### 2.4 注意点

今回の比較では、3経路すべてが完全に同じ処理段階を比較しているわけではありません。

現行1は、参照用DB（Kudu）を起点にしてGoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
現行2は、BronzeレイヤーのIcebergテーブルを起点にしてGoldレイヤーのIcebergテーブルを作成し、BIから参照する流れです。  
新規経路は、すでに作成されたGoldレイヤーのIcebergテーブルをFabricに連携し、Fabric経由でBIから参照する流れです。

そのため、結果を比較する際は、単純に「Fabricが速い」「Fabricが遅い」と判断するのではなく、どの処理段階が比較対象に含まれているかを意識する必要があります。

## 4. テスト試験項目の詳細

### 4.1 データ正確性

3経路で表示される件数や主要指標が一致しているか確認します。

性能を比較する前に、まず同じ条件で同じ結果が出ることを確認する必要があります。  
数値が一致していない状態で速度だけを比較しても、正しい評価にはなりません。

確認する内容は以下です。

- 対象件数が一致しているか
- 対象期間の最小日付、最大日付が一致しているか
- 売上、件数、ユーザー数などの主要指標が一致しているか
- NULL件数に想定外の差がないか
- 重複件数に想定外の差がないか
- 必要に応じてchecksumでデータ差分を確認できるか

---

### 4.2 データ量・スキャン量

各経路で処理しているデータ件数や容量が妥当か確認します。

3経路で処理対象のデータ量が違っている場合、性能比較が不公平になります。  
そのため、表示時間やクエリ時間とあわせて、処理対象の件数やスキャン量を記録します。

確認する内容は以下です。

- 対象件数
- データ容量
- スキャン量
- 対象期間
- Icebergのsnapshot_id
- file_count
- partition条件
- 経路ごとに処理対象が揃っているか

---

### 4.3 接続方式確認

FabricおよびPower BIの接続方式を確認します。

Fabric経由の性能は、接続方式によって大きく変わります。  
そのため、性能測定前に、どの方式でBIから参照するのかを明確にします。

確認する内容は以下です。

- Fabric側の接続方式
- Power BI側の接続モード
- Direct Lake、DirectQuery、Importなどの利用有無
- Gatewayの有無
- Semantic Modelの構成
- RLS/OLSの有無
- 本番相当の権限条件で試験するか

---

### 4.4 初回表示

BI帳票を最初に開いたときに、画面が表示されるまでの時間を確認します。

初回表示は、利用者が最初に体感する速度であり、BI利用時の印象に大きく影響します。  
また、初回表示ではキャッシュが効いていない可能性があるため、再表示とは分けて測定します。

確認する内容は以下です。

- 帳票を開いてから表示完了までの時間
- 最初のビジュアルが表示されるまでの時間
- 全ビジュアルが描画されるまでの時間
- 表示中にエラーが発生しないか
- 初回だけ極端に遅くならないか

---

### 4.5 再表示

同じBI帳票を再度表示した場合の時間を確認します。

再表示では、Power BIやデータ基盤側のキャッシュが効く可能性があります。  
初回表示と再表示を比較することで、キャッシュの影響を把握できます。

確認する内容は以下です。

- 同じ条件で再表示したときの表示時間
- 初回表示との差分
- キャッシュにより極端に速く見えていないか
- 再表示でもエラーが発生しないか
- cache_profileがcoldかwarmか

---

### 4.6 フィルタ変更

期間、店舗、商品などのフィルタ条件を変更した後、画面が再描画されるまでの時間を確認します。

実際のBI利用では、利用者が条件を変えながら分析することが多いため、フィルタ変更時の速度は重要です。

確認する内容は以下です。

- フィルタ変更後の再描画時間
- フィルタ変更後に発行されるクエリ時間
- 表示内容が正しく更新されるか
- 条件変更時にタイムアウトが発生しないか
- 高選択、中選択、低選択のフィルタ条件で差がないか

---

### 4.7 クエリ応答

BIからデータ基盤、またはFabricへ問い合わせたときの応答時間を確認します。

画面表示が遅い場合、原因がBI側にあるのか、データ基盤側にあるのか、Fabric側にあるのかを切り分ける必要があります。  
そのため、BI画面の表示時間だけでなく、裏側のクエリ応答時間も記録します。

確認する内容は以下です。

- SQLまたはクエリの実行時間
- 待機時間
- エンジン実行時間
- スキャン量
- クエリがタイムアウトしないか
- query_idを記録できるか
- 経路ごとの応答差がどこで発生しているか

---

### 4.8 エラー・タイムアウト

操作中に表示失敗、クエリエラー、タイムアウトが発生しないか確認します。

どれだけ表示速度が速くても、エラーが頻発する場合は実運用には向きません。  
特に高負荷条件や同時実行時に、安定して動作するかを確認します。

確認する内容は以下です。

- BI表示エラーの有無
- クエリエラーの有無
- タイムアウトの有無
- エラーが発生した条件
- エラーが発生した層
- 再実行すれば成功するのか
- 毎回失敗するのか
- タイムアウト秒数

---

### 4.9 高負荷集計

広い期間や細かい粒度で重い集計を行い、性能上の限界を確認します。

通常条件では問題がなくても、対象期間を広げたり、明細に近い粒度で集計したりすると、急に遅くなることがあります。  
そのため、あえて負荷の高い条件でも確認します。

確認する内容は以下です。

- 標準帳票での表示時間
- 広範囲集計での表示時間
- 高カーディナリティ明細でのクエリ時間
- 店舗別、商品別、日別など細かい粒度での集計時間
- 高負荷時のタイムアウト有無
- 経路ごとの性能差

---

### 4.10 同時実行性能

複数ユーザー、または複数セッションで同時にBIを操作した場合の性能を確認します。

1人で操作した場合は問題がなくても、複数人が同時に利用すると処理待ちや性能劣化が発生する可能性があります。  
本番利用を想定する場合は、同時実行時の確認が重要です。

確認する内容は以下です。

- 1ユーザー実行時の表示時間
- 5ユーザー相当で同時実行した場合の表示時間
- 10ユーザー以上で同時実行した場合の表示時間
- 実行間隔
- 持続時間
- 同時実行時のエラー率
- Fabric Capacity使用率や処理待ちの有無

---

### 4.11 Fabric Capacity確認

Fabric経由で処理した場合に、Capacity不足やスロットリングが発生しないか確認します。

Fabricを経由する場合、性能はFabric Capacityの状態に影響を受けます。  
そのため、BI表示時間だけでなく、Fabric側のCapacity使用状況もあわせて確認します。

確認する内容は以下です。

- Fabric Capacity使用率
- スロットリングの有無
- 処理待ちの有無
- 同時実行時にCapacity不足が発生しないか
- 試験中にバックグラウンド処理が動いていないか
- Fabric経由が遅い場合、Capacityが原因になっていないか

---

### 4.12 繰り返し実行安定性

同じ操作を複数回実行し、結果のばらつきを確認します。

性能試験では、1回だけの結果で判断するのは危険です。  
一時的な環境負荷により、たまたま速い、または遅い結果になる可能性があります。

確認する内容は以下です。

- 複数回実行したときの平均時間
- 最大時間
- 最小時間
- p50
- p95
- p99
- 実行ごとのばらつき
- 特定の回だけ極端に遅くなっていないか
- 実行順序による偏りがないか

---

### 4.13 リソース使用量

各経路でどの程度リソースを使っているか確認します。

表示やクエリが遅い場合、原因を判断するためにはリソース使用状況を見る必要があります。  
特にFabric経由では、Capacity使用率やスロットリングの確認が重要です。

確認する内容は以下です。

- CPU使用率
- メモリ使用量
- I/O使用量
- Fabric Capacity使用率
- ネットワーク影響
- 特定の経路だけ過度にリソースを消費していないか

---

### 4.14 キャッシュ影響

初回実行と2回目以降で性能差がどの程度あるか確認します。

BIやデータ基盤では、同じ処理を再実行するとキャッシュにより速くなる場合があります。  
そのため、キャッシュが効いている状態だけで判断しないようにします。

確認する内容は以下です。

- 初回実行時間
- 2回目以降の実行時間
- 初回と再実行の差分
- cache_profileがcoldかwarmか
- キャッシュにより速く見えていないか
- キャッシュが効かない条件でも許容できる性能か

---

### 4.15 ページ切替

BI内の別ページへ切り替えたときの表示時間を確認します。

複数ページ構成の帳票では、ページ切替時にもクエリが発行される可能性があります。  
ただし、初回のMVP試験では、すべてのページを対象にするのではなく、代表的な重いページに絞って確認する方が現実的です。

確認する内容は以下です。

- ページ切替にかかる時間
- ページ切替時にエラーが発生しないか
- ページによって極端な差がないか
- 重いビジュアルを含むページで遅延が発生しないか

---

### 4.16 データ反映時間

元データが更新された後、BIに反映されるまでの時間を確認します。

BI帳票で最新データを見る必要がある場合、データ反映時間は重要です。  
一方で、鮮度要件が明確でない場合は、初回MVP試験では優先度を下げてもよい項目です。

確認する内容は以下です。

- 元データの更新時刻
- Goldレイヤーへの反映時刻
- Fabric側への反映時刻
- BI上で確認できるようになる時刻
- 経路ごとの反映遅延
- 業務上、何分以内の反映が必要か

---

## 5. 測定定義

試験結果のばらつきを防ぐため、測定時の定義を事前に決めておきます。

---

### 5.1 表示時間の定義

| 項目 | 内容 |
|---|---|
| open_start_at | BI帳票を開き始めた時刻 |
| first_visual_at | 最初のグラフや表が表示された時刻 |
| all_visuals_rendered_at | すべてのビジュアルが描画された時刻 |
| total_display_time | open_start_at から all_visuals_rendered_at までの時間 |

「表示完了」は、原則として **すべてのビジュアルが描画された時点** とします。

---

### 5.2 クエリ時間の定義

| 項目 | 内容 |
|---|---|
| query_start_at | クエリが開始された時刻 |
| query_end_at | クエリが終了した時刻 |
| queue_ms | 実行待ち時間 |
| engine_ms | 実際の処理時間 |
| scan_bytes | 読み取ったデータ量 |
| query_id | 対象クエリを後から追跡するためのID |

クエリ時間は、単に合計時間だけでなく、待機時間、実行時間、スキャン量をできるだけ分けて記録します。

---

### 5.3 キャッシュ条件の定義

| cache_profile | 内容 |
|---|---|
| cold | キャッシュが効いていない、または初回に近い状態 |
| warm | 直前に同じ、または近い条件で実行しており、キャッシュが効いている可能性がある状態 |
| unknown | キャッシュ状態を判断できない状態 |

キャッシュ影響は独立した試験項目として見るだけでなく、各試験結果に `cache_profile` を付与して記録します。

---

### 5.4 同時実行条件の定義

| 項目 | 内容 |
|---|---|
| concurrent_users | 同時実行ユーザー数 |
| ramp_up_time | 同時実行数まで増やす時間 |
| steady_duration | 一定負荷をかける時間 |
| operation_mix | 初回表示、フィルタ変更、ページ切替などの操作割合 |
| interval_seconds | 各操作の実行間隔 |

同時実行性能では、単に「5人」「10人」とするだけでなく、どの操作を、どの間隔で、どのくらいの時間実行するかを決めます。

## 7. 判定基準案

判定基準は、試験前にできるだけ数値で決めておくことが望ましいです。  
基準が曖昧なままだと、結果を見た後に「速い」「遅い」の判断が人によって変わってしまいます。

現時点では、以下のように、関係者で数値を埋める形式とします。

| 評価項目 | 判定基準案 |
|---|---|
| データ正確性 | 行数、主キー一意性、対象期間、主要指標、NULL件数、checksumが一致していること。差分がある場合は事前承認された許容差内であること |
| 初回表示 | cold条件のp95が `<N秒以内>`、かつ現行比の劣化が `<X%以内>` であること |
| 再表示 | warm条件のp95が `<N秒以内>` であること |
| フィルタ変更 | 代表フィルタごとのp95が `<N秒以内>` であること |
| クエリ応答 | queue_ms、engine_ms、scan_bytesを分解して記録し、ボトルネック層を説明できること |
| エラー・タイムアウト | タイムアウト0件、またはエラー率が `<X%以下>` であること |
| 高負荷集計 | 代表ワークロードでp95が `<N秒以内>`、タイムアウトが発生しないこと |
| 同時実行性能 | `<N人>` 同時実行時にp95が `<N秒以内>`、エラー率が `<X%以下>` であること |
| Fabric Capacity | スロットリング回数、処理待ち時間、Capacity使用率が事前定義した範囲内であること |
| 繰り返し実行安定性 | p50、p95、p99を記録し、特定の実行だけ極端に遅くならないこと |
| リソース使用量 | 特定の経路だけ過度なCPU、メモリ、I/O、Capacity使用が発生していないこと |
| キャッシュ影響 | cold/warmの差分を説明できること。warm条件だけで良し悪しを判断しないこと |
| データ量・スキャン量 | 3経路の対象行数、scan_bytes、file_count、snapshot_idが比較可能であること |
| データ反映時間 | source更新からBI問い合わせ可能までのp95が `<N分以内>` であること。鮮度要件がない場合は参考扱いとする |

## 9. 今後の整理事項

本資料は、あくまでテスト試験項目を確認するための資料です。  
そのため、試験結果に対する課題・考察や、Fabric経由を採用するかどうかの判断は、本資料では扱いません。

試験実施前には、以下を整理する必要があります。

- 15項目をP0/P1/P2/P3の優先順位で確定する

- Power BI側の接続方式を確定する

# 0. この資料の位置づけ

本資料は、PMから依頼された以下3つのデータ参照・集計・BI表示の流れについて、性能試験設計書として比較できるように整理したものです。

```text
1. 現行の Kudu    → ゴールドテーブル → BI
2. 現行の Iceberg → ゴールドテーブル → BI
3. 新規の Iceberg → Fabric → BI
```

目的は、先方が正式な資料を作成する際に、**この内容をもとに性能試験設計書・比較表・検証計画へ落とし込める粒度**で整理することです。

なお、現時点で未確定の前提情報があるため、数値目標や環境条件は決め打ちせず、**「要確認」「仮置き」「想定」**として明示します。

# 2. 性能試験の目的

## 2.1 目的

今回の性能試験の目的は、**BIで同一または同等の帳票を表示する際に、3つのデータ経路でどの程度の性能差が出るかを数値で比較すること**です。

特に確認したいことは以下です。

| 観点 | 確認したいこと |
|---|---|
| 表示速度 | BIレポートの初回表示、フィルタ変更、ページ遷移がどの程度速いか |
| 安定性 | タイムアウトやエラーが発生しないか |
| データ鮮度 | データがどのタイミングでBIに反映されるか |
| 処理コスト | ゴールドテーブル作成やFabric処理にどの程度の負荷がかかるか |
| 運用性 | ジョブ監視、再実行、障害対応が現実的か |
| 将来性 | Fabric導入により、Power BI連携・AI活用・データマネジメントに広げられるか |

---

## 2.2 今回の検証で明らかにしたいこと

今回の検証では、以下の3つを切り分けて評価します。

| 切り分けたい論点 | 内容 |
|---|---|
| データソース差 | Kudu起点とIceberg起点で、ゴールドテーブル作成やBI表示に差が出るか |
| ゴールドテーブル効果 | 事前集計・整形済みデータをBIに渡すことで、表示速度が安定するか |
| Fabric効果 | IcebergデータをFabric経由で参照・加工・BI表示することで、性能や運用性が改善するか |

# 3. 比較対象の3パターン

## 3.1 比較対象一覧

| No | 比較パターン | データの流れ | 主な確認ポイント |
|---:|---|---|---|
| 1 | 現行Kudu → Gold → BI | Kuduのデータを現行処理でGold化し、BIから参照 | 現行構成の基準値を取得する |
| 2 | 現行Iceberg → Gold → BI | Icebergのデータを現行処理でGold化し、BIから参照 | Iceberg起点の現行性能を確認する |
| 3 | 新規Iceberg → Fabric → BI | IcebergのデータをFabric経由で参照・加工し、BIから参照 | Fabric導入時の性能・運用性を確認する |

---

## 3.2 各パターンの概要

### パターン1：現行Kudu → ゴールドテーブル → BI

```text
Kudu
  ↓
現行の集計・加工処理
  ↓
ゴールドテーブル
  ↓
BI
```

| 項目 | 内容 |
|---|---|
| 位置づけ | 現行構成の基準値 |
| 期待する役割 | 現在のBI表示性能・Gold作成性能を把握する |
| 主な測定対象 | Gold作成時間、BI初回表示時間、フィルタ反映時間、エラー率 |
| 注意点 | Kuduは最新・カレントデータ向けのため、Iceberg/Fabricと同一条件で比較できるようデータ期間・件数を合わせる必要がある |

---

### パターン2：現行Iceberg → ゴールドテーブル → BI

```text
Iceberg
  ↓
現行の集計・加工処理
  ↓
ゴールドテーブル
  ↓
BI
```

| 項目 | 内容 |
|---|---|
| 位置づけ | Iceberg起点の現行性能確認 |
| 期待する役割 | Kudu起点ではなくIceberg起点にした場合の差分を把握する |
| 主な測定対象 | Iceberg読込時間、Gold作成時間、BI初回表示時間、フィルタ反映時間 |
| 注意点 | Kuduと同じ粒度・期間・件数・集計ロジックで比較する必要がある |

---

### パターン3：新規Iceberg → Fabric → BI

```text
Iceberg
  ↓
Fabric
  ↓
Lakehouse / Warehouse / Gold相当テーブル
  ↓
BI
```

| 項目 | 内容 |
|---|---|
| 位置づけ | Fabric導入時の新構成候補 |
| 期待する役割 | IcebergデータをFabric経由で扱った場合のBI性能・運用性を確認する |
| 主な測定対象 | Fabric接続時間、Fabric内処理時間、BI表示時間、Capacity使用量 |
| 注意点 | Fabricの方式がShortcutなのか物理コピーなのか、Direct Lake / Import / DirectQueryのどれを使うのかを明確にする必要がある |

# 5. 試験方針

## 5.1 基本方針

今回の性能試験では、以下の方針を採用します。

| 方針 | 内容 |
|---|---|
| 同一条件で比較する | 同じ帳票、同じ期間、同じ抽出条件、同じ集計ロジックで比較する |
| 体感ではなく数値で評価する | 秒数、件数、リソース使用率、エラー率で記録する |
| ベースラインを先に取る | まず現行Kudu→Gold→BIの値を取得する |
| データ品質を確認してから速度を見る | 件数・集計値が一致しない状態で性能比較しない |
| 不明点は決めつけない | 環境・方式・合格基準が未確定の場合は要確認とする |

---

## 5.2 試験の進め方

```text
Step 1：対象帳票・対象データを決める
Step 2：3パターンで同じデータ条件を揃える
Step 3：データ件数・集計値が一致するか確認する
Step 4：各パターンで単体の処理時間を測る
Step 5：BI表示時間を測る
Step 6：必要に応じて同時実行・ピーク条件を測る
Step 7：結果を比較表にまとめる
Step 8：QCD観点で採用可否・次アクションを整理する
```

# 9. 試験ケース一覧

## TC01：データ件数・集計値一致確認

| 項目 | 内容 |
|---|---|
| 目的 | 3パターンで比較対象のデータが同等であることを確認する |
| 対象 | Kudu、Iceberg、Fabric上のテーブル |
| 実施内容 | 件数、対象期間、主要集計値を比較する |
| 測定値 | COUNT、MIN/MAX日付、売上合計、件数合計など |
| 合格基準 | 同一条件で許容差分内に収まること。許容差分は要確認 |
| 注意点 | データが一致しない場合、性能比較に進まない |

---

## TC02：Goldテーブル作成時間確認

| 項目 | 内容 |
|---|---|
| 目的 | ソースデータからBI向けGoldテーブルを作成する処理時間を比較する |

| 測定値 | 処理時間、入力件数、出力件数、失敗有無 |
| 合格基準 | 要確認。例：日次バッチ窓内に完了すること |
| 注意点 | Gold定義を同等にしないと比較できない |

# 21. まとめ

今回の性能試験では、以下3パターンを比較対象とします。

```text
1. 現行Kudu    → ゴールドテーブル → BI
2. 現行Iceberg → ゴールドテーブル → BI
3. 新規Iceberg → Fabric → BI
```

重要なのは、**Fabricが速いかどうかだけを見るのではなく、Kudu起点・Iceberg起点・Fabric経由の差分を公平に比較すること**です。

そのためには、以下を必ず揃える必要があります。

```text
・同じBI帳票
・同じデータ期間
・同じ抽出条件
・同じ集計ロジック
・同じGold定義
・同じ測定方法
・同じ判定基準
```

現時点で未確定の情報については決めつけず、PM・先方・MS社・ベンダーへ確認する前提で進めます。

最初のアクションとしては、**対象BI帳票を1つ選び、その帳票に必要なKudu/IcebergテーブルとGold定義を確認すること**が最優先です。

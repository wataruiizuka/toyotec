## 1. 結論サマリー

この資料は、性能試験のたたき台としてはかなり良いです。特に、初回表示・再表示・フィルタ変更・クエリ応答・データ正確性・同時実行・Fabric Capacity・キャッシュ・スキャン量まで入っており、単なる「画面が速い/遅い」の確認で終わらせない姿勢があります。

一方で、このまま試験に入るのは危険です。理由は、各項目が「何を測るか」は書かれているものの、「どう測るか」「何秒なら合格か」「どの層が原因かをどう切り分けるか」がまだ弱いためです。試験後に、Fabricが悪いのか、BIモデルが悪いのか、Icebergの物理設計が悪いのか、キャッシュ差なのかを判断しにくくなります。

MVPとしては、まず15項目を全部同じ重さでやるのではなく、`データ正確性`、`初回/再表示`、`フィルタ変更`、`クエリ応答`、`同時実行`、`Fabric Capacity`、`データ量・スキャン量` に絞って開始ゲートを作るのが現実的です。そのうえで、ページ切替やデータ反映時間は第2段階に回してよいです。

## 2. 目的・前提・MVP範囲

| 観点 | 今回の整理 |
|---|---|
| 目的 | 現行経路とFabric経由のBI性能を、体感ではなく数値で比較できる試験項目に整える |
| 前提 | 実測値、実データ、Fabric接続方式、Power BIモデル定義、Capacity SKU、同時利用者数は未確認 |
| MVP範囲 | 3経路で同じ意味のデータを使い、代表帳票の表示・操作・クエリ・負荷・Capacity影響を比較する |
| スケール時の拡張余地 | 試験ログを `run_id` 単位で蓄積し、将来の回帰試験、Capacity見積り、SLO監視、コスト評価に流用する |
| 非スコープ | 実測、Fabric構成の公式仕様確認、BIレポート改修、SQL/DAXチューニング、APIキー利用 |

## 3. 全体評価

| 区分 | 評価 | コメント |
|---|---|---|
| 良い点 | 試験項目の網羅性が高い | BI表示、クエリ、データ正確性、同時実行、Capacity、キャッシュ、スキャン量を押さえており、性能試験の入口として妥当です。 |
| 良い点 | データ正確性を性能測定前に置いている | 数値が合わない状態で速度比較しても意味がないため、この順序は正しいです。 |
| 良い点 | 初回表示と再表示を分けている | キャッシュ影響を意識している点は実務上かなり重要です。 |
| 良い点 | Fabric Capacityを独立項目にしている | Fabric経由の性能はCapacity状態の影響を受けるため、BI表示時間だけを見ない方針は良いです。 |
| 悪い点 | 合格基準が曖昧 | 「現行方式と同等」「極端に遅くならない」「業務上必要」では、試験後に合否を決めきれません。 |
| 悪い点 | 測定定義が不足 | 表示完了、クエリ開始、クエリ完了、レンダリング完了、待機時間の定義がないため、測定者によって値がぶれます。 |
| 悪い点 | 経路比較の同等性が弱い | Kudu起点、Bronze起点、Gold→Fabric経由で処理段階が異なるため、単純な速度比較にすると誤判定しやすいです。 |
| 悪い点 | 原因切り分けに必要な共通IDがない | `run_id`、`test_case_id`、`query_id`、`snapshot_id` がないと、BI・Fabric・Iceberg・基盤ログを後からつなげません。 |
| 不足 | セキュリティ/RLS条件 | 本番相当のユーザーロール、RLS/OLS、権限差が性能に与える影響を試験条件に入れるべきです。 |
| 不足 | Iceberg物理状態 | ファイル数、ファイルサイズ、partition、snapshot、compaction状態が違うと性能差が混入します。 |
| 不足 | コスト・運用性 | Fabric経由が速くても、Capacity費用、運用監視、障害時切り戻しが重ければ採用判断は変わります。 |
| 不足 | 負荷モデル | 1/5/10ユーザーだけでは本番ピークや操作比率を表せません。ユーザー数、操作シナリオ、実行間隔、持続時間が必要です。 |

## 4. 試験項目別レビュー

| No | 試験項目 | 評価 | 良い点 | 懸念・不足 | 修正・追加案 |
|---:|---|---|---|---|---|
| 1 | 初回表示 | 必須 | 利用者の体感に直結し、最初に見るべき指標です。 | 「表示完了」の定義が曖昧です。画面枠表示、ビジュアル描画完了、全クエリ完了のどれかで値が変わります。 | `open_start_at`、`first_visual_at`、`all_visuals_rendered_at` を分けて記録する。Cold条件を明示する。 |
| 2 | 再表示 | 必須 | キャッシュ効果を確認でき、初回表示との差分が見えます。 | 再表示が速い場合、どのキャッシュが効いたのか分かりません。 | BI、セマンティックモデル、Fabric、ストレージなど、制御可能なキャッシュ層と制御不能な層を記録する。 |
| 3 | フィルタ変更 | 必須 | 実利用に近い操作で、帳票の使いやすさに直結します。 | フィルタ値の選択性が未定義です。高選択・低選択で負荷が大きく変わります。 | 代表フィルタを `高選択/中選択/低選択` に分け、値と期待行数を固定する。 |
| 4 | クエリ応答 | 必須 | BI側とデータ基盤側の切り分けに必要です。 | SQL/DAX、待機、エンジン実行、スキャン、レンダリングが混ざる可能性があります。 | `queue_ms`、`engine_ms`、`scan_bytes`、`render_ms` を分けて記録する。クエリIDを保存する。 |
| 5 | データ正確性 | 開始ゲート | 性能比較前に数値一致を見る設計は正しいです。 | 「差分が説明できればよい」だと、比較条件が緩くなります。 | 主比較では行数、期間、主キー一意性、NULL、主要指標、checksumを開始ゲートにする。差分許容は事前承認制にする。 |
| 6 | エラー・タイムアウト | 必須 | 実運用可否を判断するうえで重要です。 | エラー分類が不足しています。BI、認証、接続、Capacity、クエリ、データ不整合で対応が違います。 | `error_code`、`error_layer`、`retry_result`、`reproducible` を記録する。タイムアウト秒数も固定する。 |
| 7 | 高負荷集計 | 必須 | 性能限界を把握する観点として必要です。 | 「広い期間」「細かい粒度」が曖昧で再現性が弱いです。 | W1標準、W2広範囲集計、W3高カーディナリティ明細のように代表ワークロード化する。 |
| 8 | 同時実行性能 | 必須 | 本番利用を考えるなら避けられない項目です。 | 1/5/10ユーザーだけでは負荷モデルとして弱いです。実行間隔、操作比率、持続時間がありません。 | ランプアップ、定常負荷、ランプダウンを定義する。最低でも定常15分、可能なら30分以上を測る。 |
| 9 | Fabric Capacity確認 | 必須 | Fabric経由固有のボトルネックを見に行っている点は良いです。 | Capacity使用率だけでは原因を断定できません。処理待ち、スロットリング、同時実行、バックグラウンド処理を合わせて見る必要があります。 | 試験時刻ごとにCapacity状態、待機、スロットリング、バックグラウンド負荷を記録する。現行経路とは別軸で評価する。 |
| 10 | ページ切替 | 有用 | 複数ページ帳票では実利用に近い確認です。 | 初回表示やフィルタ変更と重複する可能性があります。ページごとの重さも未定義です。 | MVPでは代表的な重いページ1つに絞る。全ページ試験は第2段階でよい。 |
| 11 | データ反映時間 | 条件付き必須 | 鮮度が業務要件なら重要です。 | 開始点と終了点が曖昧です。元データ更新、Gold更新、Fabric反映、BIで問い合わせ可能のどれを測るか決める必要があります。 | `source_commit_at`、`gold_ready_at`、`fabric_ready_at`、`bi_queryable_at` を分ける。業務鮮度SLAがない場合は優先度を下げる。 |
| 12 | 繰り返し実行安定性 | 必須 | 平均だけでなくp95を見る点は良いです。 | サンプル数、外れ値扱い、実行順序が未定義です。 | 各条件30回以上を目安にし、経路順はラウンドロビンまたはランダム化する。p50/p95/p99を保存する。 |
| 13 | リソース使用量 | 必須 | 原因分析に必要な観点です。 | 経路ごとに取れるメトリクスが違うため、単純比較しづらいです。 | 共通指標はCPU/メモリ/I/O/scan、Fabric固有指標は別列にする。測定粒度と収集元を固定する。 |
| 14 | キャッシュ影響 | 必須だが統合推奨 | 初回/再実行の差分を見る発想は正しいです。 | 1・2と重複しています。独立項目にすると記録が散らばります。 | `cache_profile = cold/warm/unknown` として全試験項目に付与する。項目14は横断属性に変える。 |
| 15 | データ量・スキャン量 | 開始ゲート | 比較の公平性とボトルネック分析に効きます。 | 件数・容量だけでは不足です。Icebergではファイル数やpartition状態が効きます。 | `row_count`、`data_bytes`、`scan_bytes`、`file_count`、`partition`、`snapshot_id` を記録する。 |

## 5. 追加した方がよい試験・記録項目

| 追加観点 | 追加理由 | MVPでの扱い | 具体的な記録例 |
|---|---|---|---|
| 試験ケースID設計 | 後から結果を追跡できるようにするため | 必須 | `test_case_id`、`workload_id`、`run_id` |
| データスナップショット固定 | 経路間で同じデータを比較するため | 必須 | `snapshot_id`、対象期間、行数、checksum |
| SQL/DAX/モデル定義の固定 | クエリ差による性能差を排除するため | 必須 | SQL/DAXハッシュ、BIページ、ビジュアルID、モデルバージョン |
| 接続方式の明記 | Fabric経由の性能は接続方式に左右されるため | 必須 | Fabric接続方式、BI接続モード、更新方式 |
| RLS/権限条件 | 本番と異なる権限条件だと性能・キャッシュ条件が変わるため | 必須 | user_role、RLS有無、権限パターン |
| バックグラウンド負荷 | 本番では更新処理や他ユーザー処理と共存するため | Phase 2 | 更新処理有無、他ワークロード有無、実行時間帯 |
| コスト指標 | 採用判断は速度だけでなく費用対効果も見るため | Phase 2 | 1,000クエリあたりコスト、月額Capacity見積り、運用工数 |
| 障害時の切り戻し | Fabric経由で問題が出た場合の運用判断に必要 | Phase 2 | 現行経路への戻し条件、戻し時間、影響範囲 |
| 監視・アラート | 本番移行後に劣化を検知するため | Phase 2 | p95閾値、エラー率、スロットリング、鮮度遅延 |

## 6. 判定基準の修正案

現行資料の判定基準は方向性としては良いですが、試験実施前に数値化しないと使えません。最低限、以下のように空欄を埋める形式にした方がよいです。

| 評価項目 | 現行案の弱さ | 修正案 |
|---|---|---|
| データ正確性 | 「主要指標が一致」「原因を説明できる」だけでは緩い | 行数、主キー一意性、期間、NULL、主要指標、checksumが一致。許容差は指標ごとに事前承認 |
| 初回表示 | 「現行方式と同等、または改善」だけでは曖昧 | Cold p95 が `<N秒以内>`、かつ現行比劣化 `<X%以内>` |
| 再表示 | 「極端に遅くならない」が曖昧 | Warm p95 が `<N秒以内>`、Coldとの差分を記録 |
| フィルタ変更 | 「支障を感じない範囲」が主観的 | 代表フィルタごとに p95 `<N秒以内>`、タイムアウト0件 |
| クエリ応答 | 画面表示とクエリ時間の関係が曖昧 | `queue_ms`、`engine_ms`、`scan_bytes` を分解し、ボトルネック層を特定可能にする |
| 同時実行性能 | 同時操作時の「極端な劣化」が曖昧 | ピーク想定同時数で p95 `<N秒以内>`、エラー率 `<X%以下>`、スロットリング条件を満たす |
| Fabric Capacity | 「頻発しない」が曖昧 | 試験中のスロットリング回数、待機時間、Capacity使用率の上限を事前定義 |
| データ反映時間 | 業務鮮度要件が未定義 | source更新からBI問い合わせ可能まで p95 `<N分以内>` |
| リソース使用量 | 「過度に消費」が曖昧 | 経路別のCPU/メモリ/I/O/Capacity使用率を記録し、ボトルネック説明に使う。単独の合否条件にはしすぎない |
| データ量・スキャン量 | 妥当性の判断軸が曖昧 | 3経路の対象行数、scan bytes、file count、snapshotが比較可能であることを開始条件にする |

## 7. 優先順位の見直し案

| 優先 | 対象 | 理由 |
|---|---|---|
| P0: 試験開始前に必須 | データ正確性、データ量・スキャン量、スナップショット固定、接続方式、合格基準 | ここが曖昧だと性能比較の結論が信用できません。 |
| P1: MVP試験で必須 | 初回表示、再表示、フィルタ変更、クエリ応答、エラー・タイムアウト、同時実行、Fabric Capacity | 利用者体感、安定性、Fabric固有リスクに直結します。 |
| P2: 原因分析で必須 | 繰り返し実行安定性、リソース使用量、キャッシュ影響 | 結果のばらつきや遅延理由を説明するために必要です。 |
| P3: 第2段階でよい | ページ切替、データ反映時間、コスト、監視、切り戻し | 業務要件次第で重要ですが、初回MVPでは範囲を絞った方が早いです。鮮度要件が強い場合はデータ反映時間をP1に上げます。 |

## 8. プロ目線での率直な意見

| 論点 | 意見 |
|---|---|
| この資料をそのまま使えるか | たたき台としては使えます。ただし、試験実施手順書としてはまだ不足です。 |
| 一番良い点 | 性能だけでなく、正確性、安定性、Capacity、キャッシュ、スキャン量まで見ようとしている点です。これは良い方向です。 |
| 一番危ない点 | 「現行と同等」「業務上必要」などの曖昧な判定基準です。結果が出た後に解釈が割れます。 |
| 技術的に一番不足している点 | BI画面、セマンティックモデル、Fabric、Iceberg、基盤リソースをつなぐ相関IDと分解計測です。ここがないと原因分析が浅くなります。 |
| PMO目線の懸念 | 関係者合意が必要な値が多いです。SLO、同時実行数、鮮度、許容劣化率、Capacity条件は試験前に合意してください。 |
| データエンジニア目線の懸念 | Icebergの物理状態を記録していない点です。ファイル数、partition、compaction状態が違うと、経路差ではなくテーブル状態差を測ってしまいます。 |
| QA目線の懸念 | 再現性が不足しています。フィルタ値、実行順序、試行回数、外れ値扱い、タイムアウト秒数を固定すべきです。 |
| Fabric採用判断への使い方 | この資料だけで採用/不採用は判断しない方がよいです。MVP試験結果、コスト、運用、障害時切り戻しを合わせて判断してください。 |

## 9. 次アクション

1. 15項目をそのまま実行する前に、P0/P1/P2/P3へ並べ替える。
2. `run_id`、`test_case_id`、`workload_id`、`snapshot_id`、`query_id` を持つ記録フォーマットに修正する。
3. 合格基準を `<N秒>`、`<X%>`、`<同時ユーザー数>`、`<鮮度N分>` のように数値で埋める。
4. 代表ワークロードを3つに絞る。例: 標準帳票、広範囲集計、高カーディナリティ明細。
5. Fabric接続方式、Capacity条件、BIモデル/RLS条件、Iceberg物理状態を試験前提として固定する。

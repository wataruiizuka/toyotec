# Hue上でのテーブル確認結果

## 1.確認の目的

今回、Hue上で以下の内容を確認しました。

- 対象のIcebergテーブルがAzure Storage上に配置されているか
- 格納先となるストレージアカウント、コンテナ、パス

まず、共有されていたテーブル一覧をもとに、検証環境の以下の2テーブルについて`SHOW CREATE TABLE`を実行しました。

- `ai21rep_ve_test_kudu.tbv0201m`
- `ai21rep_ve_test_kudu.ice_tbv0201m`

---

## 2.確認結果の概要

今回確認した2テーブルについては、どちらもIcebergテーブルではありませんでした。

確認結果は以下のとおりです。

| テーブル名 | 実際の形式 | 主な格納先・管理先 |
|---|---|---|
| `tbv0201m` | Parquet | Azure Storage |
| `ice_tbv0201m` | Kudu | Kuduクラスタ |

`ice_tbv0201m`はテーブル名の先頭に`ice_`が付いていますが、テーブル定義上はIcebergではなくKuduテーブルでした。

---

## 3.`tbv0201m`の確認結果

`ai21rep_ve_test_kudu.tbv0201m`では、テーブル定義に以下の記載がありました。

```sql
STORED AS PARQUET
```

このため、`tbv0201m`はIcebergテーブルではなく、Parquet形式のテーブルです。

また、`LOCATION`には以下のパスが設定されていました。

```text
abfs://data@sttoyotecdapveje.dfs.core.windows.net/warehouse/tablespace/managed/hive/ai21rep_ve_test_kudu.db/tbv0201m
```

この内容から、格納先は以下のとおりです。

| 確認項目 | 確認結果 |
|---|---|
| ストレージアカウント | `sttoyotecdapveje` |
| コンテナ | `data` |
| パス | `warehouse/tablespace/managed/hive/ai21rep_ve_test_kudu.db/tbv0201m` |
| テーブル形式 | Parquet |
| Icebergテーブルか | いいえ |

したがって、`tbv0201m`はAzure Storage上に配置されていますが、Icebergテーブルではありません。

また、以下のプロパティが設定されていました。

```text
transactional=true
transactional_properties=insert_only
```

このため、Hiveのinsert-onlyトランザクションテーブルとして管理されていることも確認できました。

---

## 4.`ice_tbv0201m`の確認結果

続いて、同じ`tbv0201m`という名称を含み、先頭に`ice_`が付いた以下のテーブルを確認しました。

```text
ai21rep_ve_test_kudu.ice_tbv0201m
```

テーブル定義には以下の記載がありました。

```sql
CREATE EXTERNAL TABLE
```

```sql
STORED AS KUDU
```

このため、`ice_tbv0201m`はIcebergテーブルではなく、External Kuduテーブルです。

テーブル名に`ice_`が付いていても、実際のテーブル形式がIcebergであるとは限らないことが分かりました。

### Kuduの接続先

テーブル定義には、以下の3台のKudu Masterが設定されていました。

```text
cdp-dap-kudu-ve-je-master10.toyotecd.kj09-szpi.a0.cloudera.site:7051
cdp-dap-kudu-ve-je-master20.toyotecd.kj09-szpi.a0.cloudera.site:7051
cdp-dap-kudu-ve-je-master30.toyotecd.kj09-szpi.a0.cloudera.site:7051
```

確認できたKuduクラスタIDは以下です。

```text
e242e27faf994eb297de913aa5003d48
```

これらはAzure Storageのパスではなく、ImpalaからKuduクラスタへ接続するための情報です。

Kuduテーブルは、Azure Storage上のParquetファイルを`LOCATION`で直接参照する形式ではなく、Kuduが実データを管理します。そのため、今回のテーブル定義にはAzure Storageのストレージアカウント、コンテナ、格納パスは表示されていません。

### 主キー

`ice_tbv0201m`には、以下の3カラムを組み合わせた主キーが設定されています。

```text
cd_hansya
cd_kaisya
cd_tenpo
```

そのため、この3カラムの組み合わせで1行を一意に識別する設計になっています。

### 削除時の注意点

以下のプロパティも設定されていました。

```text
TRANSLATED_TO_EXTERNAL=TRUE
external.table.purge=TRUE
```

特に`external.table.purge=TRUE`が設定されているため、`DROP TABLE`を実行した場合、Hive Metastore上のテーブル定義だけでなく、Kudu側の実データも削除される可能性があります。

そのため、以下のような削除操作は実施しないよう注意が必要です。

```sql
DROP TABLE ai21rep_ve_test_kudu.ice_tbv0201m;
```

## 5.2つのテーブルの比較

今回確認した2テーブルを比較すると、以下の違いがあります。

| 確認項目 | `tbv0201m` | `ice_tbv0201m` |
|---|---|---|
| テーブル形式 | Parquet | Kudu |
| Icebergテーブルか | いいえ | いいえ |
| テーブル種別 | Managedテーブルと考えられる | External Kuduテーブル |
| Azure StorageのLOCATION | あり | なし |
| ストレージアカウント | `sttoyotecdapveje` | 該当なし |
| コンテナ | `data` | 該当なし |
| 主キー | 定義なし | 3カラムで定義 |
| 主なデータ管理先 | Azure Storage | Kuduクラスタ |

カラム構成はほぼ同じですが、実際の格納形式と管理方法は異なっています。

また、両方のテーブルに以下のカラムが存在しています。

```text
iceberg_md5
kudu_md5
```

ただし、カラム名に`iceberg`や`kudu`が含まれていることは、テーブル自体の形式を示すものではありません。

テーブル形式は、`SHOW CREATE TABLE`の以下の部分で判断する必要があります。

```text
STORED AS ICEBERG
STORED AS PARQUET
STORED AS KUDU
```

## 6.現時点で確認できていないこと

今回の確認結果だけでは、以下の内容までは判断できません。

- `tbv0201m`と`ice_tbv0201m`のどちらが正本か
- 2テーブル間でデータが同期されているか
- どの処理でデータが連携されているか
- 2テーブルの件数やデータ内容が一致しているか
- `iceberg_md5`と`kudu_md5`が具体的に何のために使用されているか
- 別の名称でIcebergテーブルが存在しているか
- Kudu Tablet Serverの実データが配置されている物理ディスクや基盤上の構成

したがって、「2つのテーブルが同じデータを保持している」「ParquetからKuduへ同期している」といったことは、現時点では断定していません。

## 7.報告内容

今回、検証環境にある`tbv0201m`と、先頭に`ice_`が付いた`ice_tbv0201m`について、Hue上でテーブル定義を確認しました。

確認した結果、`tbv0201m`はIcebergテーブルではなく、Azure Storage上に配置されたParquet形式のテーブルでした。格納先は、ストレージアカウントが`sttoyotecdapveje`、コンテナが`data`、パスが`warehouse/tablespace/managed/hive/ai21rep_ve_test_kudu.db/tbv0201m`です。

一方、`ice_tbv0201m`は、名前に`ice_`が付いていますが、実際にはIcebergではなくKuduテーブルでした。このテーブルにはAzure Storageの`LOCATION`は設定されておらず、3台のKudu Masterで構成されたKuduクラスタ側で管理されています。

そのため、今回確認した2テーブルの中には、Iceberg形式のテーブルはありませんでした。

ただし、今回確認したのは対象候補の一部です。別の名称でIcebergテーブルが存在している可能性や、ParquetテーブルとKuduテーブル間の連携方法については、引き続き確認が必要です。

---

## 8.まとめ

- `tbv0201m`は、Azure Storage上に配置されたParquetテーブル
- `ice_tbv0201m`は、Kuduクラスタで管理されるKuduテーブル
- `ice_tbv0201m`は、名前に`ice_`が付いているがIcebergではない
- 今回確認した2テーブルの中には、Icebergテーブルは存在しない
- `tbv0201m`のストレージアカウント、コンテナ、パスは確認できた
- `ice_tbv0201m`にはAzure Storageの`LOCATION`がなく、ストアカ・コンテナ・パスという形では確認できない
- 2テーブル間の同期関係や正本については、現時点では未確認
- `ice_tbv0201m`は`external.table.purge=TRUE`のため、`DROP TABLE`を実行しないよう注意が必要

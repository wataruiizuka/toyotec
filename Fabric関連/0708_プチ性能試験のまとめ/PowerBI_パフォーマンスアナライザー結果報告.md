# Power BI パフォーマンスアナライザー結果報告  
## Kudu / Iceberg / Fabric 経路別 表示性能比較

---

## 1. 目的

今回の目的は、Power BI帳票画面において、同一内容の集計を3つの経路で表示した場合に、表示性能にどの程度の差が出るかを確認することです。

比較対象の経路は以下の3つです。

| 経路 | 対象テーブル |
|---|---|
| Kudu経路 | `ai21rep_ve_stg_kudu.tbba001g_fabric_test` |
| Iceberg経路 | `ai21rep_ve_stg.tbba001g_fabric_test_iceberg` |
| Fabric経路 | `bronze_tbba001g` |

今回は初回検証として、対象テーブルを1つに絞ったスモールスタートで確認しています。

なお、Kudu経路とIceberg経路の元テーブルは約330万件規模ですが、Fabric経路のテーブルは約3.5万件です。  
そのため、今回の比較では、Kudu経路とIceberg経路についても、Fabric経路のデータ件数に合わせた比較用テーブルを使用しています。

したがって、今回の検証は、約330万件の元テーブル全体を対象にした性能比較ではなく、**約3.5万件程度に条件を揃えたうえで、Power BI帳票画面上の表示性能を比較する検証**です。

測定には、Power BIのパフォーマンスアナライザー結果を使用しました。

---

## 2. 帳票の集計式

今回測定したPower BI帳票は、以下の4つのカードビジュアルで構成されています。

| 帳票/対象ビジュアル | 集計式 | 集計内容 |
|---|---|---|
| レコード件数 | `COUNTROWS('各テーブル名')` | 対象テーブルの行数を数える |
| 販社数 | `DISTINCTCOUNT('各テーブル名'[cd_hansya])` | 販社コードのユニーク数を数える |
| 会社数 | `DISTINCTCOUNT('各テーブル名'[cd_kaisya])` | 会社コードのユニーク数を数える |
| 担当者数 | `DISTINCTCOUNT('各テーブル名'[cd_hanstaff])` | 担当者コードのユニーク数を数える |

今回のビジュアルはいずれもカード形式です。  
そのため、表示時間に差が出る場合は、カードの描画そのものよりも、DAXクエリやデータ取得処理の影響が大きい可能性があります。

---

## 3. 結果

### 3.1 経路別の平均結果

Power BIパフォーマンスアナライザー結果を経路別に平均すると、以下の結果でした。

| 経路 | 平均合計時間 | 最小 | 最大 | 平均DAXクエリ | 平均直接クエリ | 平均ビジュアル表示 | 平均その他 |
|---|---:|---:|---:|---:|---:|---:|---:|
| Kudu | 11,438 ms | 2,656 ms | 20,258 ms | 11,252 ms | 10,995 ms | 38 ms | 114 ms |
| Iceberg | 11,403 ms | 2,596 ms | 20,190 ms | 11,272 ms | 10,992 ms | 23 ms | 109 ms |
| Fabric | 1,313 ms | 1,295 ms | 1,351 ms | 140 ms | 表示なし | 26 ms | 1,147 ms |

今回の測定範囲では、**Fabric経路が最も短い表示時間**となりました。

Kudu経路とIceberg経路は平均で約11.4秒でしたが、Fabric経路は平均で約1.3秒でした。  
特に、Kudu経路とIceberg経路ではDAXクエリおよび直接クエリの時間が大きく、Fabric経路との差が明確に出ています。

### 3.2 ビジュアル別の結果

| 帳票/対象ビジュアル | Kudu | Iceberg | Fabric | 傾向 |
|---|---:|---:|---:|---|
| レコード件数 | 20,183 ms | 20,151 ms | 1,351 ms | Fabricが大幅に速い |
| 会社数 | 2,656 ms | 2,675 ms | 1,295 ms | Fabricが速い |
| 販社数 | 20,258 ms | 20,190 ms | 1,300 ms | Fabricが大幅に速い |
| 担当者数 | 2,656 ms | 2,596 ms | 1,304 ms | Fabricが速い |

特に差が大きかったのは、**レコード件数**と**販社数**です。

この2つは、Kudu経路・Iceberg経路では約20秒かかっている一方、Fabric経路では約1.3秒に収まっています。

### 3.3 Fabric経路による短縮率

| 帳票/対象ビジュアル | Kudu比 | Iceberg比 |
|---|---:|---:|
| レコード件数 | 約93.3%短縮 | 約93.3%短縮 |
| 会社数 | 約51.2%短縮 | 約51.6%短縮 |
| 販社数 | 約93.6%短縮 | 約93.6%短縮 |
| 担当者数 | 約50.9%短縮 | 約49.8%短縮 |

今回の範囲では、Fabric経路は、重いビジュアルでは約93%短縮、比較的軽いビジュアルでも約50%短縮という結果でした。

## 4. 結果からの考察

### 4.1 今回の条件ではFabric経路が最も速い

今回の測定では、Fabric経路が4つすべてのビジュアルで最も速い結果となりました。

特に、レコード件数と販社数では、Kudu経路・Iceberg経路が約20秒かかっているのに対し、Fabric経路は約1.3秒です。  
利用者の体感としても、かなり大きな差になると考えられます。

そのため、今回のスモールスタート検証の範囲では、**同程度のデータ件数に揃えた場合、Power BI帳票表示性能はFabric経路が最も良い結果だった**と整理できます。

### 4.2 Kudu経路とIceberg経路はほぼ同等

Kudu経路とIceberg経路の結果は、全体的に非常に近い値でした。

| 帳票/対象ビジュアル | Kudu | Iceberg | 差分 |
|---|---:|---:|---:|
| レコード件数 | 20,183 ms | 20,151 ms | Icebergが32 ms速い |
| 会社数 | 2,656 ms | 2,675 ms | Kuduが19 ms速い |
| 販社数 | 20,258 ms | 20,190 ms | Icebergが68 ms速い |
| 担当者数 | 2,656 ms | 2,596 ms | Icebergが60 ms速い |

この程度の差分は非常に小さいため、今回の結果だけで「Kuduの方が速い」「Icebergの方が速い」と判断するのは適切ではないと考えます。

今回の範囲では、**Kudu経路とIceberg経路はほぼ同等の性能**と見るのが妥当です。

### 4.3 Kudu/Iceberg経路はクエリ処理に時間がかかっている

Kudu経路とIceberg経路では、合計時間の大部分をDAXクエリおよび直接クエリが占めています。

例として、Kudu経路のレコード件数は以下の通りです。

| 項目 | 時間 |
|---|---:|
| 合計時間 | 20,183 ms |
| DAXクエリ | 19,996 ms |
| 直接クエリ | 19,716 ms |
| ビジュアル表示 | 43 ms |
| その他 | 144 ms |

ビジュアル表示自体は43 msと短いため、Power BIのカード描画が遅いというより、**データ取得や集計処理に時間がかかっている**と考えられます。

つまり、Kudu経路・Iceberg経路の主なボトルネックは、Power BI画面の描画ではなく、DAXクエリやDirectQuery側の処理にある可能性が高いです。

### 4.4 レコード件数と販社数が特に重い

Kudu経路・Iceberg経路では、以下の2つが約20秒かかっています。

- レコード件数：`COUNTROWS('各テーブル名')`
- 販社数：`DISTINCTCOUNT('各テーブル名'[cd_hansya])`

一方で、以下の2つは約2.6秒です。

- 会社数：`DISTINCTCOUNT('各テーブル名'[cd_kaisya])`
- 担当者数：`DISTINCTCOUNT('各テーブル名'[cd_hanstaff])`

この結果から、単純に「DISTINCTCOUNTだから遅い」とは言えません。  
販社数・会社数・担当者数はいずれもDISTINCTCOUNTですが、販社数だけが大きく遅くなっています。

考えられる要因としては、以下があります。

- `COUNTROWS` がデータソース側で全件確認に近い処理になっている可能性
- `cd_hansya` のDISTINCTCOUNTが重いSQLに変換されている可能性
- 対象列のデータ分布や統計情報の影響
- Power BIから発行されるクエリの実行計画の違い
- Kudu/Iceberg側での読み取りや集計処理の効率差

ただし、現時点では原因を断定できません。  
実際に発行されたDAXやデータソース側のSQL、実行計画を確認することで、より正確な原因分析が可能になります。

### 4.5 Fabric経路はDAXクエリが非常に短い

Fabric経路では、4つすべてのビジュアルでDAXクエリが140 msでした。

| 帳票/対象ビジュアル | Fabric 合計時間 | Fabric DAXクエリ |
|---|---:|---:|
| レコード件数 | 1,351 ms | 140 ms |
| 会社数 | 1,295 ms | 140 ms |
| 販社数 | 1,300 ms | 140 ms |
| 担当者数 | 1,304 ms | 140 ms |

Kudu/Iceberg経路ではDAXクエリが約2.4秒〜20秒かかっているため、Fabric経路との差は大きいです。

今回の表示性能差は、Fabric経路でDAXクエリ時間が短く収まっていることが大きく影響していると考えられます。

### 4.6 Fabric経路では「その他」の割合が大きい

Fabric経路では、DAXクエリは短い一方で、「その他」が約1.1秒あります。

| Fabric平均 | 時間 | 合計に占める割合 |
|---|---:|---:|
| DAXクエリ | 140 ms | 約10.7% |
| ビジュアル表示 | 26 ms | 約2.0% |
| その他 | 1,147 ms | 約87.4% |

これは、Fabric側の集計処理が遅いというより、Power BIレポート側の待機時間、内部処理、他ビジュアルとの処理順序などが影響している可能性があります。

ただし、Fabric経路の合計時間は約1.3秒で安定しているため、現時点では大きな問題とは見なさず、今後の確認観点として扱うのがよいと考えます。

### 4.7 直接クエリの表示有無に差がある

Kudu経路とIceberg経路では「直接クエリ」の時間が表示されています。  
一方、Fabric経路では「直接クエリ」は表示されていません。

このため、今回のPerformance Analyzer上では、Kudu/Iceberg経路は外部問い合わせ時間が明確に出ており、Fabric経路では同じ形では表示されていないと考えられます。

ただし、この結果だけでFabric経路の接続方式を断定することはできません。  
今後の確認では、各経路のPower BI接続方式やストレージモードも併せて記録しておく必要があります。

## 5. 注意点

### 5.1 約330万件全体の性能比較ではない

今回の比較では、Kudu経路とIceberg経路についても、Fabric経路の約3.5万件に合わせた比較用テーブルを使用しています。

そのため、今回の結果をもって、約330万件全体でも同じ傾向になるとは言えません。  
今回言えるのは、**同程度のデータ件数に揃えた条件ではFabric経路が速かった**ということです。

### 5.2 対象は1テーブル・4カードビジュアルである

今回の対象は、1つのテーブルと4つのカードビジュアルです。

明細表、複数グラフ、複雑なスライサー、ページ切替、クロスフィルタなどを含む帳票では、結果が変わる可能性があります。

そのため、今回の結果は、本格的な全帳票評価ではなく、初回検証としての結果です。

### 5.3 測定回数によるばらつき確認は必要

今回の結果が単発測定である場合、キャッシュ状態や一時的な環境負荷の影響を受けている可能性があります。

今後、より確度を上げるには、同じ条件で複数回測定し、平均値、最大値、ばらつきを確認する必要があります。

### 5.4 DAXクエリと直接クエリの時間は単純加算しない

Performance Analyzerでは、DAXクエリ、直接クエリ、ビジュアル表示、その他などが表示されます。

ただし、DAXクエリ時間と直接クエリ時間を単純に足し合わせて合計時間として解釈するのは適切ではありません。  
DirectQuery構成では、DAXクエリの中で外部データソースへの問い合わせが発生するため、両者は関連して表示されます。

今回の結果は、  
**Kudu/IcebergではDAXクエリおよび直接クエリが支配的**  
という見方に留めるのが適切です。

### 5.5 Fabric Capacityの評価は別途必要

今回の測定は、Power BIのパフォーマンスアナライザー結果です。

そのため、Fabric Capacityの使用率、CU消費量、スロットリング有無までは判断できません。  
Fabric経路の本格評価に進む場合は、Power BIの表示性能だけでなく、Fabric Capacity側のメトリクスも確認する必要があります。

## 6. まとめ

今回の検証では、Power BIのパフォーマンスアナライザーを用いて、Kudu経路、Iceberg経路、Fabric経路の3経路で、同一内容のカードビジュアル表示性能を比較しました。

結果として、Fabric経路は4つすべてのビジュアルで最も速く、平均合計時間は約1.3秒でした。  
一方、Kudu経路とIceberg経路は平均で約11.4秒であり、特にレコード件数と販社数では約20秒かかっていました。

Kudu経路とIceberg経路は非常に近い結果であり、今回の範囲ではほぼ同等と見てよいと考えます。  
両経路では、ビジュアル表示時間は短く、DAXクエリおよび直接クエリに時間が集中しているため、主な遅延要因はPower BIの描画ではなく、データ取得・集計処理側にある可能性が高いです。

Fabric経路では、DAXクエリがすべて140 msと短く、今回の条件ではデータ取得・集計処理が軽く見えます。  
一方で、「その他」が約1.1秒ありますが、合計時間自体は約1.3秒で安定しているため、現時点では大きな問題とは見なさず、今後の確認観点として扱うのが妥当です。

今回の結論は以下です。

- 同程度のデータ件数に揃えた今回の条件では、Fabric経路が最も高速だった。
- Kudu経路とIceberg経路は、今回の範囲ではほぼ同等だった。
- Kudu/Iceberg経路の遅延要因は、Power BIのカード描画ではなく、DAXクエリ・直接クエリ側にある可能性が高い。
- レコード件数と販社数が特に重く、必要に応じてDAXやデータソース側SQLの確認余地がある。
- 今回の結果は、1テーブル・4カードビジュアル・約3.5万件規模に条件を揃えた初回検証として扱うのが適切である。
- 次の段階では、複数回測定、接続方式の確認、DAX/SQL確認、Fabric Capacity確認を行うことで、より確度の高い評価につなげられる。

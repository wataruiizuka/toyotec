# Fabric 初回検証報告  
## Kudu / Iceberg / Fabric 経路別 Power BI 表示性能比較

---

## 0. この資料で伝えたいこと

今回の検証では、Power BIの同じ帳票を、以下の3つの経路で表示したときに、どの経路が速く表示できるかを確認しました。

- Kudu経路
- Iceberg経路
- Fabric経路

結論として、**今回の条件ではFabric経路が最も速い結果**でした。

特に、レコード件数と販社数のカードでは、Kudu経路・Iceberg経路が約20秒かかったのに対し、Fabric経路は約1.3秒で表示されました。

ただし、今回の検証はあくまで初回検証です。  
対象は1つのテーブル、4つのカードビジュアル、約3.5万件規模に条件を揃えた比較です。  
そのため、今回の結果だけで、すべての帳票や全データ量でも同じ結果になるとは判断しません。

---

## 1. 目的

今回の目的は、Power BI帳票画面で同じ内容の集計を表示したときに、データの参照経路によって表示時間にどの程度の差が出るかを確認することです。

Power BIは、画面に数値やグラフを表示する前に、裏側でデータを取得したり、件数を数えたり、ユニーク数を計算したりします。  
今回の検証では、その処理にどれくらい時間がかかっているかを、Power BIのパフォーマンスアナライザーで確認しました。

比較対象の経路は以下の3つです。

| 経路 | 対象テーブル | 内容 |
|---|---|---|
| Kudu経路 | `ai21rep_ve_stg_kudu.tbba001g_fabric_test` | Kudu側の比較用テーブルをPower BIで表示する経路 |
| Iceberg経路 | `ai21rep_ve_stg.tbba001g_fabric_test_iceberg` | Iceberg側の比較用テーブルをPower BIで表示する経路 |
| Fabric経路 | `bronze_tbba001g` | Fabric側のテーブルをPower BIで表示する経路 |

今回は初回検証として、対象テーブルを1つに絞って確認しています。

なお、Kudu経路とIceberg経路の元テーブルは約330万件規模ですが、Fabric経路のテーブルは約3.5万件です。  
データ件数が大きく違うまま比較すると、公平な性能比較になりません。

そのため、今回の比較では、Kudu経路とIceberg経路についても、Fabric経路のデータ件数に合わせた比較用テーブルを使用しています。

つまり、今回の検証は、約330万件の元テーブル全体を対象にした性能比較ではありません。  
**約3.5万件程度に条件を揃えたうえで、Power BI帳票画面上の表示性能を比較する検証**です。

## 2. 測定に使ったもの

今回の測定には、Power BIの**パフォーマンスアナライザー**を使用しました。

パフォーマンスアナライザーとは、Power BIの画面上で、各ビジュアルの表示にどれくらい時間がかかったかを確認する機能です。

今回の結果では、主に以下の項目を確認しています。

| 項目 | 初級者向けの意味 |
|---|---|
| 合計時間 | そのカードが表示されるまでにかかった全体の時間 |
| DAXクエリ | Power BIが裏側で行う計算や問い合わせにかかった時間 |
| 直接クエリ | Power BIが外部のデータソースへ問い合わせた時間 |
| ビジュアル表示 | カードやグラフを画面に描くための時間 |
| その他 | 待機時間やPower BI内部処理など、上記以外の時間 |

今回のビジュアルはカード形式です。  
カード形式とは、件数や人数のような1つの数値を表示するシンプルな画面部品です。

そのため、もし表示が遅い場合は、カードを画面に描く処理よりも、裏側のデータ取得や集計処理に時間がかかっている可能性があります。

---

## 3. 帳票の集計式

今回測定したPower BI帳票は、以下の4つのカードビジュアルで構成されています。

| 帳票/対象ビジュアル | 集計式 | 初級者向けの意味 |
|---|---|---|
| レコード件数 | `COUNTROWS('各テーブル名')` | テーブルに何行あるかを数える |
| 販社数 | `DISTINCTCOUNT('各テーブル名'[cd_hansya])` | 販社コードが何種類あるかを数える |
| 会社数 | `DISTINCTCOUNT('各テーブル名'[cd_kaisya])` | 会社コードが何種類あるかを数える |
| 担当者数 | `DISTINCTCOUNT('各テーブル名'[cd_hanstaff])` | 担当者コードが何種類あるかを数える |

補足すると、`COUNTROWS` は行数を数える関数です。  
`DISTINCTCOUNT` は、重複を除いて何種類あるかを数える関数です。

たとえば、同じ販社コードが何回も出てきても、`DISTINCTCOUNT` では1種類として数えます。

利用者の体感としても、Fabric経路の方がかなり速く表示される結果です。

## 5. 結果からの考察

### 5.1 今回の条件ではFabric経路が最も速い

今回の測定では、Fabric経路が4つすべてのビジュアルで最も速い結果となりました。

特に、レコード件数と販社数では、Kudu経路・Iceberg経路が約20秒かかっているのに対し、Fabric経路は約1.3秒です。

この差は、利用者が画面を開いたときにも分かりやすい差になると考えられます。

そのため、今回の検証の範囲では、  
**同程度のデータ件数に揃えた場合、Power BI帳票表示性能はFabric経路が最も良い結果だった**  
と整理できます。

### 5.2 Kudu経路とIceberg経路はほぼ同等

Kudu経路とIceberg経路の結果は、全体的に非常に近い値でした。

| 帳票/対象ビジュアル | Kudu | Iceberg | 差分 |
|---|---:|---:|---:|
| レコード件数 | 20,183 ms | 20,151 ms | Icebergが32 ms速い |
| 会社数 | 2,656 ms | 2,675 ms | Kuduが19 ms速い |
| 販社数 | 20,258 ms | 20,190 ms | Icebergが68 ms速い |
| 担当者数 | 2,656 ms | 2,596 ms | Icebergが60 ms速い |

差分は数十ミリ秒程度です。  
これは人間の体感ではほとんど分からない差です。

そのため、今回の結果だけで「Kuduの方が速い」「Icebergの方が速い」と判断するのは適切ではないと考えます。

今回の範囲では、**Kudu経路とIceberg経路はほぼ同等の性能**と見るのが妥当です。

### 5.3 Kudu/Iceberg経路はデータ取得・集計処理に時間がかかっている

Kudu経路とIceberg経路では、合計時間の大部分をDAXクエリおよび直接クエリが占めています。

例として、Kudu経路のレコード件数は以下の通りです。

| 項目 | 時間 |
|---|---:|
| 合計時間 | 20,183 ms |
| DAXクエリ | 19,996 ms |
| 直接クエリ | 19,716 ms |
| ビジュアル表示 | 43 ms |
| その他 | 144 ms |

この結果を見ると、ビジュアル表示自体は43 msと非常に短いです。

つまり、Power BIのカードを画面に描く処理が遅いわけではありません。  
主に時間がかかっているのは、Power BIが裏側でデータを取得したり、集計したりする処理だと考えられます。

初級者向けに言うと、  
**画面の部品を描くのが遅いのではなく、表示する数値を取りに行く処理が遅い可能性が高い**  
ということです。

### 5.4 レコード件数と販社数が特に重い

Kudu経路・Iceberg経路では、以下の2つが約20秒かかっています。

- レコード件数：`COUNTROWS('各テーブル名')`
- 販社数：`DISTINCTCOUNT('各テーブル名'[cd_hansya])`

一方で、以下の2つは約2.6秒です。

- 会社数：`DISTINCTCOUNT('各テーブル名'[cd_kaisya])`
- 担当者数：`DISTINCTCOUNT('各テーブル名'[cd_hanstaff])`

ここで重要なのは、単純に「DISTINCTCOUNTだから遅い」とは言えない点です。

販社数、会社数、担当者数はいずれも`DISTINCTCOUNT`を使っています。  
しかし、販社数だけが約20秒で、会社数と担当者数は約2.6秒です。

そのため、関数の種類だけではなく、対象列のデータの持ち方や、Power BIからデータソースへ発行される問い合わせ内容が影響している可能性があります。

考えられる要因は以下です。

- レコード件数では、データソース側で全件確認に近い処理が発生している可能性
- 販社コード `cd_hansya` のユニーク数集計が、重い処理になっている可能性
- 対象列のデータ分布や統計情報が影響している可能性
- Power BIから発行されるクエリの内容が、ビジュアルごとに異なっている可能性
- Kudu/Iceberg側での読み取りや集計処理に時間がかかっている可能性

ただし、現時点では原因を断定できません。  
原因をより正確に確認するには、実際に発行されたDAXやデータソース側のSQLを確認する必要があります。

### 5.5 Fabric経路はDAXクエリが非常に短い

Fabric経路では、4つすべてのビジュアルでDAXクエリが140 msでした。

| 帳票/対象ビジュアル | Fabric 合計時間 | Fabric DAXクエリ |
|---|---:|---:|
| レコード件数 | 1,351 ms | 140 ms |
| 会社数 | 1,295 ms | 140 ms |
| 販社数 | 1,300 ms | 140 ms |
| 担当者数 | 1,304 ms | 140 ms |

Kudu/Iceberg経路ではDAXクエリが約2.4秒〜20秒かかっています。  
それに対して、Fabric経路ではすべて140 msです。

今回の表示性能差は、Fabric経路でDAXクエリ時間が短く収まっていることが大きく影響していると考えられます。

### 5.6 Fabric経路では「その他」の割合が大きい

Fabric経路では、DAXクエリは短い一方で、「その他」が約1.1秒あります。

| Fabric平均 | 時間 | 合計に占める割合 |
|---|---:|---:|
| DAXクエリ | 140 ms | 約10.7% |
| ビジュアル表示 | 26 ms | 約2.0% |
| その他 | 1,147 ms | 約87.4% |

「その他」は、Power BI内部の待ち時間や処理順序、他のビジュアルとの関係などを含む可能性があります。

ただし、Fabric経路の合計時間は約1.3秒で安定しています。  
そのため、現時点では大きな問題とは見なさず、今後の確認観点として扱うのがよいと考えます。

### 5.7 直接クエリの表示有無に差がある

Kudu経路とIceberg経路では「直接クエリ」の時間が表示されています。  
一方、Fabric経路では「直接クエリ」は表示されていません。

これは、Power BIから見た処理の見え方が経路ごとに違っている可能性を示しています。

ただし、この結果だけでFabric経路の接続方式を断定することはできません。  
今後の確認では、各経路のPower BI接続方式やストレージモードも併せて記録しておく必要があります。

## 6. 注意点

### 6.1 約330万件全体の性能比較ではない

今回の比較では、Kudu経路とIceberg経路についても、Fabric経路の約3.5万件に合わせた比較用テーブルを使用しています。

そのため、今回の結果をもって、約330万件全体でも同じ傾向になるとは言えません。

今回言えるのは、  
**同程度のデータ件数に揃えた条件ではFabric経路が速かった**  
ということです。

### 6.2 対象は1テーブル・4カードビジュアルである

今回の対象は、1つのテーブルと4つのカードビジュアルです。

明細表、複数グラフ、複雑なスライサー、ページ切替、クロスフィルタなどを含む帳票では、結果が変わる可能性があります。

そのため、今回の結果は、本格的な全帳票評価ではなく、初回検証としての結果です。

### 6.3 DAXクエリと直接クエリの時間は単純加算しない

Performance Analyzerでは、DAXクエリ、直接クエリ、ビジュアル表示、その他などが表示されます。

ただし、DAXクエリ時間と直接クエリ時間を単純に足し合わせて、合計時間として解釈するのは適切ではありません。

DirectQuery構成では、DAXクエリの中で外部データソースへの問い合わせが発生するため、両者は関連して表示されます。

今回の結果は、  
**Kudu/IcebergではDAXクエリおよび直接クエリが大きな割合を占めている**  
という見方に留めるのが適切です。

### 6.4 Fabric Capacityの評価は別途必要

今回の測定は、Power BIのパフォーマンスアナライザー結果です。

そのため、Fabric Capacityの使用率、CU消費量、スロットリング有無までは判断できません。

Fabric経路の本格評価に進む場合は、Power BIの表示性能だけでなく、Fabric Capacity側のメトリクスも確認する必要があります。

## 7. まとめ

今回の検証では、Power BIのパフォーマンスアナライザーを用いて、Kudu経路、Iceberg経路、Fabric経路の3経路で、同一内容のカードビジュアル表示性能を比較しました。

結果として、Fabric経路は4つすべてのビジュアルで最も速く、平均合計時間は約1.3秒でした。  
一方、Kudu経路とIceberg経路は平均で約11.4秒であり、特にレコード件数と販社数では約20秒かかっていました。

Kudu経路とIceberg経路は非常に近い結果であり、今回の範囲ではほぼ同等と見てよいと考えます。

Kudu/Iceberg経路では、ビジュアル表示時間は短く、DAXクエリおよび直接クエリに時間が集中していました。  
そのため、主な遅延要因はPower BIの画面描画ではなく、データ取得・集計処理側にある可能性が高いです。

Fabric経路では、DAXクエリがすべて140 msと短く、今回の条件ではデータ取得・集計処理が軽く見えます。  
一方で、「その他」が約1.1秒ありますが、合計時間自体は約1.3秒で安定しているため、現時点では大きな問題とは見なさず、今後の確認観点として扱うのが妥当です。

今回の結論は以下です。

- 同程度のデータ件数に揃えた今回の条件では、Fabric経路が最も高速だった。
- Kudu経路とIceberg経路は、今回の範囲ではほぼ同等だった。
- Kudu/Iceberg経路の遅延要因は、Power BIのカード描画ではなく、DAXクエリ・直接クエリ側にある可能性が高い。
- レコード件数と販社数が特に重く、必要に応じてDAXやデータソース側SQLの確認余地がある。
- 今回の結果は、1テーブル・4カードビジュアル・約3.5万件規模に条件を揃えた初回検証として扱うのが適切である。
- 次の段階では、複数回測定、接続方式の確認、DAX/SQL確認、Fabric Capacity確認を行うことで、より確度の高い評価につなげられる。

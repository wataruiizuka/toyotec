# 性能試験設計書（テキストにおける案）
## Kudu / Iceberg / Fabric 経路別 BI性能比較

> 本資料は、Fabricを実装する際の性能試験設計書(比較表)に関するたたき台です。  
> 前提が未確定の項目は「要確認」または「想定」として記載しています。  
---

## 1. 目的

本性能試験では、BIで同一帳票・同一条件のデータを参照した場合に、以下3つのデータ参照経路でどの程度の性能差が出るかを比較する。

1. 現行：Kudu → ゴールドテーブル → BI
2. 現行：Iceberg → ゴールドテーブル → BI
3. 新規：Iceberg → Fabric → BI

本試験の主目的は、各経路におけるBI表示性能、クエリ応答時間、安定性、データ反映時間を実測し、現行方式とFabric経由方式の差分を定量的に把握することである。

また、単純な表示速度だけでなく、運用負荷や今後の拡張性も含めて整理し、Fabric経由の有効性を判断するための材料を作成する。

## 2. 対象範囲

### 2.1 試験対象

| No | 比較対象フロー | 試験対象に含める内容 | 主な確認ポイント |
|---:|---|---|---|
| 1 | 現行：Kudu → ゴールドテーブル → BI | 現行Kudu由来のGoldテーブルをBIから参照する経路 | 現行方式の基準性能を確認する(実行済みであれば不要) |
| 2 | 現行：Iceberg → ゴールドテーブル → BI | 現行Iceberg由来のGoldテーブルをBIから参照する経路 | Iceberg由来の現行Gold参照性能を確認する |
| 3 | 新規：Iceberg → Fabric → BI | IcebergデータをFabric経由でBIから参照する経路 | Fabric利用時の速度・安定性・運用性を確認する |

### 2.2 対象外

| 項目 | 対象外とする理由 |
|---|---|
| 全帳票の網羅検証 | 初期PoCでは対象帳票を絞った方が比較しやすいため |
| Webアプリ全体の性能改善検証 | 今回の主対象はBI参照経路の性能比較であるため |
| 本番移行可否の最終判断 | 本試験は判断材料を作る位置づけであり、最終判断は別途必要なため |
| セキュリティ設計の詳細化 | Fabric側の権限設計は重要だが、今回の性能試験とは分けて扱うため |
| データモデル全体の再設計 | 性能比較の条件が変わるため、初期試験では既存構造を基準にする |

## 3. 前提条件・制約

### 3.1 現時点の前提

| 項目 | 内容 |
|---|---|
| Kuduの位置づけ | 直近・カレントデータを扱う現行基盤の一部として扱う |
| Icebergの位置づけ | 履歴・トランザクションデータを扱う基盤として扱う |
| Goldテーブルの位置づけ | BI表示に利用しやすいように加工・集計済みのテーブルとして扱う |
| Fabricの位置づけ | Icebergデータを参照・加工・BI連携する新規経路として扱う |
| BI帳票 | 同一帳票を使って3経路を比較する想定 |
| 測定方法 | 体感ではなく、秒数・成功率・エラー有無などの数値で比較する |

### 3.2 要確認事項

| No | 要確認事項 | 確認理由 |
|---:|---|---|
| 1 | 試験対象とするBI帳票名 | 帳票によってクエリ、データ量、表示速度が変わるため |
| 2 | 対象テーブル一覧 | 3経路で同じ意味のデータを比較するため |
| 3 | 対象期間 | データ量を揃えるため |
| 4 | 各経路のデータ件数・容量 | 性能結果の前提として必要なため |
| 5 | BIの接続方式 | Import、DirectQuery、Direct Lakeなどで性能が変わるため |
| 6 | Fabricの接続方式 | Shortcut、Lakehouse、Warehouse、Direct Lakeなどの違いを明確にするため |
| 7 | Fabric Capacity | Fabric側の性能は利用容量に依存するため |
| 8 | キャッシュ有無 | 初回表示と2回目以降で速度が変わるため |
| 9 | 同時利用者数 | 単独利用と複数利用で性能が変わるため |
| 10 | 合格基準 | 何秒以内なら合格か、現時点では未定のため |

## 4. 性能要件

現時点では具体的な合格基準が未確定のため、以下は**設計時の評価観点案**として扱います。

| 評価項目 | 測定内容 | 判定の考え方 |
|---|---|---|
| BI初回表示時間 | レポートを開いてから初回表示されるまでの時間 | 3経路で比較し、現行比で改善または同等かを確認する |
| フィルタ変更時間 | 条件変更後に画面が再描画されるまでの時間 | 利用者の操作感に影響するため重点的に測定する |
| ページ切替時間 | BI内のページを切り替えた時の表示時間 | 帳票利用時の体感速度として確認する |
| クエリ応答時間 | BIから発行されるクエリの実行時間 | どの経路がボトルネックか切り分ける |
| エラー率 | タイムアウト、クエリエラー、表示失敗の発生有無 | 原則としてエラーなしを目標とする |
| データ正確性 | 各経路で表示される集計値の一致 | 性能比較前に必ず一致確認する |
| リソース使用量 | CPU、メモリ、I/O、Fabric Capacity使用率など | 性能差の原因分析に利用する |
| データ反映時間 | 元データ更新後、BIに反映されるまでの時間 | 鮮度要件がある場合に測定する |

## 5. 試験方針

今回の性能試験は、以下の方針で実施します。

| 方針 | 内容 |
|---|---|
| 同一帳票で比較する | 帳票差による性能差を避けるため、同じBI帳票または同等条件の帳票で比較する |
| 同一条件で比較する | 対象期間、フィルタ条件、集計粒度、表示項目を揃える |
| 現行方式を基準にする | Kudu経由、Iceberg経由の現行性能を基準値として取得する |
| Fabric経由を新規方式として評価する | Fabric導入時の速度・安定性・運用性を現行方式と比較する |
| キャッシュ影響を分ける | 初回実行と2回目以降を分けて測定する |
| 複数回測定する | 1回だけの結果ではなく、複数回実行して平均・最大・p95などで見る |
| 数値一致を先に確認する | 性能を測る前に、3経路でBI上の数値が一致するか確認する |

## 6. 試験観点

| 観点 | 確認内容 |
|---|---|
| 表示性能 | BI画面がどれくらい速く表示されるか |
| 操作性能 | フィルタ変更やページ切替がどれくらい速いか |
| クエリ性能 | BIからデータ基盤へ問い合わせた時の応答時間 |
| 安定性 | エラーやタイムアウトが発生しないか |
| 同時実行性 | 複数ユーザーが同時に操作しても性能が大きく落ちないか |
| データ正確性 | 3経路で同じ条件の集計結果が一致するか |
| データ鮮度 | データ更新後、BIへ反映されるまでの時間に差があるか |
| 運用性 | Fabric経由にした場合、監視・障害対応・データ反映の運用が現実的か |

## 7. 比較観点表

| 比較観点 | 現行：Kudu → Gold → BI | 現行：Iceberg → Gold → BI | 新規：Iceberg → Fabric → BI | 確認したいこと |
|---|---|---|---|---|
| データ起点 | Kudu | Iceberg | Iceberg | 起点データの違いによる性能差 |
| 主なデータ用途 | 直近・カレントデータ想定 | 履歴・トランザクションデータ想定 | 履歴・トランザクションデータ想定 | 用途差が比較結果に影響しないか |
| BI向け加工 | Goldテーブル | Goldテーブル | Fabric内で参照または加工 | Gold有無・加工場所の違いによる性能差 |
| BI接続方式 | 要確認 | 要確認 | 要確認 | 接続方式が性能に与える影響 |
| 比較上の役割 | 現行基準値 | 現行Iceberg基準値 | 新規方式 | Fabric導入効果を測る |
| 想定メリット | 既存運用で実績がある | 履歴データをGold化して参照できる | FabricによるBI連携・拡張性が期待できる | 速度・運用・拡張性の比較 |
| 想定リスク | Kudu依存、現行構成依存 | Gold作成処理の負荷・遅延 | Fabric Capacity、接続方式、権限設計の影響 | 性能悪化要因の切り分け |

## 8. 試験シナリオ・ケース

### 8.1 基本シナリオ

| ケースID | 試験ケース | 実施内容 | 対象フロー | 測定項目 |
|---|---|---|---|---|
| TC-01 | BI初回表示 | 対象帳票を初回表示する | 3経路すべて | 初回表示時間、エラー有無 |
| TC-02 | BI再表示 | 同一帳票を再度表示する | 3経路すべて | 再表示時間、キャッシュ影響 |
| TC-03 | フィルタ変更 | 対象期間、店舗、商品などの条件を変更する | 3経路すべて | 再描画時間、クエリ時間 |
| TC-04 | ページ切替 | BI内の別ページへ切り替える | 3経路すべて | ページ切替時間 |
| TC-05 | 重い集計処理 | 明細に近い粒度、または集計負荷が高い表示を行う | 3経路すべて | クエリ時間、タイムアウト有無 |
| TC-06 | 同時実行 | 複数ユーザーまたは複数セッションで同時に表示する | 3経路すべて | 応答時間、エラー率、リソース使用量 |
| TC-07 | データ反映 | 元データ更新後にBIへ反映されるまでを確認する | 必要に応じて実施 | 反映時間、差分有無 |
| TC-08 | 繰り返し実行 | 同一操作を複数回実行する | 3経路すべて | 平均、最大、p95、ばらつき |

### 8.2 優先度

| 優先度 | 対象ケース | 理由 |
|---|---|---|
| 高 | TC-01、TC-03、TC-05 | BI利用者の体感速度に直結するため |
| 中 | TC-02、TC-04、TC-08 | キャッシュ影響や安定性を見るため |
| 必要に応じて | TC-06、TC-07 | 同時利用者数や鮮度要件が明確になった場合に実施する |

## 11. 負荷条件・パターン

| パターン | 内容 | 実施目的 |
|---|---|---|
| 単独実行 | 1ユーザーまたは1セッションで操作 | 基本性能を確認する |
| 連続実行 | 同じ操作を複数回繰り返す | キャッシュ影響とばらつきを確認する |
| 同時実行 | 複数ユーザーで同時に操作 | ピーク時の劣化を確認する |
| 高負荷クエリ | 重い集計や広い期間を指定する | ボトルネックを確認する |
| データ更新後実行 | データ更新後にBIを表示する | 反映遅延と整合性を確認する |

| 中 | 5 | 少人数利用想定 |

## 12. 計測項目・判定基準

### 12.1 計測項目

| 分類 | 計測項目 | 取得元の例 |
|---|---|---|
| BI表示 | 初回表示時間、再表示時間、ページ切替時間 | Power BI Performance Analyzerなど |
| BI操作 | フィルタ変更後の再描画時間 | Power BI、ブラウザ計測 |
| クエリ | SQL実行時間、スキャン量、待機時間 | Clouderaログ、Fabricログ |
| Fabric | Capacity使用率、スロットリング有無、処理時間 | Fabric Capacity Metricsなど |
| Gold作成 | Goldテーブル作成時間、更新時間 | ジョブログ |
| 安定性 | エラー件数、タイムアウト件数 | BIログ、基盤ログ |
| 正確性 | 件数、主要集計値の一致 | SQL確認結果 |
| 鮮度 | データ更新からBI反映までの時間 | 更新ログ、BI確認結果 |

### 12.2 判定基準案

| 評価項目 | 判定基準案 |
|---|---|
| データ正確性 | 主要指標が3経路で一致すること。差分がある場合は理由を説明できること |
| 表示性能 | Fabric経由が現行方式に対して改善、または同等であること。劣化する場合は原因と改善案を整理すること |
| 安定性 | 試験中にタイムアウトや表示失敗が発生しないこと |
| 同時実行性 | 同時実行時に極端な劣化やエラーが発生しないこと |
| 運用性 | Fabric経由にした場合の監視・再実行・障害切り分けが現実的であること |
| コスト妥当性 | Fabric Capacityやストレージ増加が効果に見合うか判断できること |

## 14. 障害時の対応方針

| 事象 | 対応方針 |
|---|---|
| BIが表示されない | 接続方式、権限、データセット、Gateway有無を確認する |
| クエリがタイムアウトする | 対象期間、スキャン量、SQL、リソース使用量を確認する |
| 3経路で数値が合わない | 対象期間、フィルタ条件、集計ロジック、更新日時を確認する |
| Fabric側が遅い | Capacity使用率、スロットリング、Shortcut経路、ファイル構成を確認する |
| 現行側が遅い | Gold作成処理、既存クエリ、Cloudera側リソースを確認する |
| 試験中に環境負荷が高騰する | 試験を一時停止し、ログ取得後に再実施条件を見直す |

| 結果概要 | どの経路が最も速いか、安定しているか |
| 数値結果 | 初回表示、フィルタ変更、クエリ時間、エラー有無 |
| 差分理由 | 遅い・速い理由として考えられる要因 |
| 課題 | 本番化前に確認すべき点 |
| 判断材料 | Fabric経由を継続検討すべきか、追加検証が必要か |

## 18. PM報告用の説明文

今回の性能試験では、BIにデータを表示するまでの経路として、現行の「Kudu → Gold → BI」、現行の「Iceberg → Gold → BI」、新規の「Iceberg → Fabric → BI」の3パターンを比較します。

比較では、同じ帳票・同じ期間・同じフィルタ条件を使い、BIの初回表示時間、フィルタ変更時の応答時間、クエリ実行時間、エラー有無、データ反映時間を測定します。

まずは3経路で表示される件数や主要指標が一致することを確認し、そのうえで性能を測定します。数値が一致しない状態で速度だけ比較すると正しい判断ができないため、データ正確性の確認を先に行います。

Fabric経由については、Shortcut、Lakehouse、Warehouse、Direct Lakeなどの具体的な接続方式によって結果が変わる可能性があります。そのため、接続方式とFabric Capacityを明確にしたうえで測定します。

最終的には、単純に速い・遅いだけではなく、安定性、運用負荷、データ鮮度、コスト面も含めて、Fabric経由にするメリットがあるかを判断できる形で整理します。

## 20. まとめ

今回の性能試験設計では、3つのBI参照経路を同一条件で比較し、Fabric経由の有効性を判断できる材料を作ります。

最初に実施すべきことは、対象帳票を1つ選び、3経路で同じ条件のデータを参照できる状態にすることです。

そのうえで、以下の順番で進めます。

1. 対象帳票・対象テーブルを決める
2. 3経路のデータ件数・主要指標の一致を確認する
3. BI表示条件を固定する
4. 現行Kudu経由を測定する
5. 現行Iceberg経由を測定する
6. 新規Fabric経由を測定する
7. 結果を比較表にまとめる
8. 差分理由、課題、追加確認事項を整理する

これにより、PMが先方資料を作成する際に、性能試験の目的、対象、比較軸、測定項目、判定方法をそのまま資料化できる状態にします。

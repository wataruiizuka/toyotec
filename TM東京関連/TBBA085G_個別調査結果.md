## 2. テーブルの特徴

TBBA085Gは、抹消または退避された新車受注の情報を保持しているテーブルと考えられます。

TBBA001Gの247カラムをすべて持ち、さらに`dt_massyo`が追加された248カラム構成です。共通カラムのデータ型は一致していますが、カラムの並び順と主キー構成は異なります。

構造上の関係は次のとおりです。

```text

＋
dt_massyo
＝
TBBA085Gの248カラム
```

このため、TBBA001Gとの関連は非常に強いと考えられます。

ただし、現在値データの比較だけでは、次の処理が実際に行われているかは確認できません。

- TBBA001GからTBBA085Gへのコピー
- TBBA001GからTBBA085Gへの移動
- TBBA085G格納後のTBBA001G側レコード削除
- 両テーブルに同時に存在する期間
- TBBA085Gへ格納される具体的な条件

## 17. 完全な履歴テーブルかどうか

TBBA085Gは抹消・退避済みのデータを保持する可能性がありますが、受注の変更前後をすべて保存する完全な履歴テーブルではないと考えられます。

```text

valid_from
valid_to
is_current
operation_type
is_deleted
before_image
after_image
kafka_offset
source_scn
```

また、`dt_massyo`は主キーではありません。

そのため、同じ5カラム主キーに対して、抹消日時だけが異なる複数の変更版を保持できる構造ではありません。

現時点では、次のように捉えるのが適切です。

```text
受注の変更履歴をすべて保持するテーブル
ではなく、
抹消・退避時点の最終状態を保持するテーブル候補
```

## 21. 履歴方式の候補

TBBA085G自体が抹消・退避済み受注の最終断面を保持している場合、TBBA001Gと合わせて次のように利用する方法が考えられます。

```text
TBBA001G
現行受注の現在状態

TBBA085G
抹消・退避済み受注の最終状態
```

ただし、この2テーブルを合わせても、同じ受注が変更された全履歴を再現できるとは限りません。

将来的に完全な状態変化を保持する場合は、受注キーや5カラム主キーを基準に、履歴基盤側で次の管理項目を追加する方法が考えられます。

| 管理項目 | 用途 |
|---|---|
| `source_system` | 元システムを識別 |
| `source_table` | TBBA001G・TBBA085Gなどの出所を識別 |
| `loaded_at` | 履歴基盤への取込日時 |
| `source_updated_at` | 元テーブルの最終更新日時候補 |
| `record_hash` | 履歴対象項目の変更判定 |
| `valid_from` | その状態の有効開始日時 |
| `valid_to` | その状態の有効終了日時 |
| `is_current` | 現在有効な状態か |
| `is_deleted` | 削除・退避状態か |
| `deleted_at` | 削除・退避を検知した日時 |

既存の`csv_md5`と`kudu_md5`は全件に設定されていないため、必要に応じて履歴対象カラムから独自の`record_hash`を作成する必要があります。

## 23. 現時点での評価

TBBA085Gは、TBBA001Gとほぼ同じ受注情報を持ち、追加で`dt_massyo`を保持するアーカイブ系テーブル候補です。

売上日候補`dd_uriage`と`dt_massyo`は全件に設定されていますが、取消日候補`dd_torikesi`は全件NULLでした。このため、業務上の取消受注を保存するテーブルというよりも、売上済み受注を一定条件で抹消・退避した最終状態を保持している可能性があります。

`TRIM(no_frame)`はTBAZ017Mとの車両接続キーとして非常に有力であり、約98.15%のユニーク値が一致しています。ただし、`no_frame`は一意ではないため、物理主キーや履歴キーとして単独では使用できません。

また、受注キー4カラムだけでは441種類の重複があり、重複するすべてのキーで売上日、車両番号、抹消日時候補が異なっていました。そのため、TBBA085G内のレコード識別には`dd_uriage`を含む5カラム主キーが必要です。

TBBA085Gには変更前後を管理する一般的な履歴カラムがなく、`dt_massyo`も主キーには含まれていません。そのため、完全な変更履歴ではなく、抹消・退避時点の最終断面を保持するテーブルとして扱うのが現時点では最も自然です。

ただし、TBBA001Gとの物理的なコピー・移動・削除関係や、`dt_massyo`の正式な意味は確認できていないため、最終的な履歴方式は業務要件とテーブル仕様を確認したうえで決定する必要があります。

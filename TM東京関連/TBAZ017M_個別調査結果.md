## 2. テーブルの特徴

TBAZ017Mは、車両そのものの基本情報を保持しているテーブルと考えられます。

ただし、`no_frame`単位で複数行が存在するため、単純に「1車両につき1行」という構造ではありません。Kudu上では、7カラムの複合主キーによって各レコードが識別されています。

そのため、履歴テーブルを設計する場合は、次の2つを分けて考える必要があります。

- 元テーブルの1レコードを識別するための物理主キー
- 他テーブルと車両を結び付けるための車両接続キー

## 6. `no_frame`のデータ品質

確認時点の集計結果は次のとおりです。

| 項目 | 結果 |
|---|---:|
| NULL | 0件 |
| 空文字・空白のみ | 11,059件 |
| 有効値あり | 23,697,519件 |
| 前後空白あり | 23,706,506件 |
| `TRIM`後のユニーク数 | 約4,467,000件 |
| 有効値1種類当たりの平均行数 | 約5.30行 |

ほぼ全件の`no_frame`に前後空白が含まれているため、テーブル間で照合する際は、次のように前後空白を除去する必要があります。

```sql
TRIM(no_frame)
```

ただし、元の値を上書きするのではなく、履歴テーブルでは次の2項目を分けて保持するのが安全です。

| 項目 | 用途 |
|---|---|
| `no_frame_original` | 元テーブルの値をそのまま保持 |
| `no_frame_normalized` | `TRIM`などを行った照合用の値 |

## 7. `no_frame`の一意性

`no_frame`は車両を横断的に確認するキーとして有力ですが、一意ではありません。

約2,370万件に対し、`TRIM(no_frame)`のユニーク数は約447万件でした。このため、`no_frame`を単独で次の用途に使用することはできません。

- TBAZ017Mの主キー
- 履歴テーブルの主キー
- 1レコードの一意識別
- JOIN後の行数保証
- 履歴バージョンの識別

一方で、TBBA001GおよびTBBA085Gとの高い接続率が確認されているため、車両単位でテーブルを横断する際の接続キー候補としては有効です。

## 8. 更新日時候補

確認されている主な日時カラムは次のとおりです。

| カラム | 現時点での扱い |
|---|---|
| `dt_johosaku` | 情報作成日時候補 |
| `dt_kosin` | 更新日時候補 |

ただし、正式な日本語名称、値が設定されるタイミング、更新条件は確認できていません。

また、`dt_kosin`には通常の業務日時とは考えにくい未来日時が含まれていました。そのため、履歴処理で使用する場合は、通常の日時と特殊値を分けて扱う必要があります。

想定される特殊値の例は次のとおりです。

- 未確定を表す値
- 無期限を表す値
- 初期値
- 上流システムで設定された管理用の値

正式な仕様が分かるまでは、未来日時を単純に除外せず、元の値を保持する方針が安全です。

## 9. MD5カラムの状態

2026年7月16日21時41分時点の確認結果は次のとおりです。

| 項目 | 件数 |
|---|---:|
| 総レコード件数 | 23,713,645件 |
| `csv_md5`が空白 | 17,357,697件 |
| `kudu_md5`が空白 | 17,385,547件 |
| 両方のMD5が設定済み | 6,328,098件 |
| 両方が一致 | 6,328,098件 |
| 両方が不一致 | 0件 |
| 両方設定済みにおける一致率 | 100.0000% |
| `csv_md5`のみ設定 | 27,850件 |

両方のMD5が設定されているレコードでは、すべて同じ値でした。

ただし、両方のMD5がそろっているのは全体の約26.69%です。そのため、既存のMD5だけを使って全レコードの変更を判定することはできません。

また、MD5が一致していることから、過去に更新がなかったとは判断できません。更新時に`csv_md5`と`kudu_md5`が同時に再計算される仕様であれば、更新後も一致するためです。

## 13. 履歴方式の候補

現時点では、次の方式が候補です。

```text
最新テーブル
＋
変更が発生したレコードだけを保持する履歴テーブル
```

または、変更前後の有効期間を管理するSCD Type2方式が考えられます。

ただし、既存のMD5は全件に設定されていないため、履歴化する場合は対象カラムから独自の`record_hash`を作成する必要があります。

毎日約2,371万件をそのまま保存する日次フルスナップショット方式は、保存量が急増するため、現時点では第一候補にはしません。

最終的な方式を決めるには、次の内容を確認する必要があります。

- 履歴として残すカラム
- 車両単位で管理するか、物理主キー単位で管理するか
- 変更を検知する方法
- 削除データを検知する方法
- 履歴の保持期間
- Power BIで必要となる過去時点の粒度

## 15. 現時点での評価

TBAZ017Mは、車両の基本情報を保持するマスタ系・現在値系テーブル候補です。

`TRIM(no_frame)`は他テーブルとの車両接続キーとして有力ですが、物理主キーや履歴キーとして単独で使用することはできません。

また、データ量が約2,371万件と大きいため、日次フルスナップショットよりも、変更が発生したレコードだけを保持する方式を優先して検討するのが現実的です。

ただし、既存のMD5、更新日時、現在値データだけでは、すべての変更や削除を完全に判定できません。そのため、履歴方式は現時点では候補として整理し、業務要件や保持対象を確認したうえで最終決定する必要があります。

## 2. テーブルの特徴

TBBA001Gは、新車受注に関する現在の状態を保持しているテーブルと考えられます。

受注日、条件変更日、登録日、売上日、納車日、取消日など、受注から納車・取消までの各段階に関係する日付候補を多数持っています。そのため、単なる受注受付テーブルではなく、受注後の業務状態を横持ちで管理している可能性があります。

一方で、変更前の値、変更後の値、履歴番号、有効開始日時、有効終了日時といった一般的な履歴管理項目は確認できません。

このため、現在のTBBA001Gは、受注の状態変化をすべて時系列で保持する履歴テーブルではなく、受注ごとの現在状態を保持するテーブル候補と考えるのが適切です。

前後空白を含むデータがあるため、テーブル間で照合する場合は、次のように`TRIM`を使用する必要があります。

```sql
TRIM(no_frame)
```

履歴テーブルでは、元値と照合用の値を分けて保持するのが安全です。

| 項目 | 用途 |
|---|---|
| `no_frame_original` | 元テーブルの値をそのまま保持 |
| `no_frame_normalized` | `TRIM`などを行った照合用の値 |

### 7.2 TBAZ017Mとの接続率

この結果から、`TRIM(no_frame)`は車両基本テーブルとの横断接続キーとして有力です。

ただし、次の制約があります。

- `no_frame`が空白の受注は接続できない
- `no_frame`はTBBA001G内で一意ではない
- `no_frame`だけではTBBA001Gの1レコードを一意に識別できない
- JOIN後の行数が1対1になるとは限らない

そのため、`no_frame`は車両を探すための接続キーとして使用し、TBBA001Gの物理主キーとしては使用しません。

## 16. 履歴方式の候補

TBBA001Gは受注状態が変化するテーブルであるため、現時点では次の方式が有力です。

```text
最新の受注状態を保持するテーブル
＋
変更が発生した受注だけを追加する履歴テーブル
```

履歴テーブルでは、少なくとも次の単位を基準にする必要があります。

```text
cd_hansya
cd_kaisya
no_cyumon
no_cyumoned
＋
履歴版を識別する項目
```

既存のMD5は全件に設定されていないため、差分検知には独自の`record_hash`を作成する方法が考えられます。

また、受注の状態変化を分析する場合は、業務日付とデータ取込日時を分けて保持する必要があります。

| 項目 | 例 |
|---|---|
| 業務日付 | `dd_jucyu`、`dd_jokenhen`、`dd_uriage`、`dd_nosya`、`dd_torikesi` |
| データ更新日時 | `dt_saisinup` |
| 履歴取込日時 | 履歴基盤側で新規付与 |
| 有効開始日時 | 履歴基盤側で新規付与 |
| 有効終了日時 | 履歴基盤側で新規付与 |
| 現在行フラグ | 履歴基盤側で新規付与 |
| レコードハッシュ | 履歴対象カラムから新規生成 |

ただし、`dt_saisinup`が変更検知に必ず使用できるか、同じ日時で複数回更新される可能性がないかは、現在値だけでは判断できません。

## 18. 現時点での評価

TBBA001Gは、新車受注の現在状態を保持する受注・状態管理系テーブル候補です。

4カラムの複合主キーは、確認時点ではTBBA001G内の全レコードを一意に識別できていました。一方で、この受注キーはTBBA085Gとの横断接続には適していません。

`TRIM(no_frame)`はTBAZ017Mとの車両接続キーとして有力ですが、約49.3万件では有効な値が設定されていないため、すべての受注を車両単位で接続することはできません。

取消日や条件変更日など、受注状態を把握するための業務日付は豊富に存在します。ただし、現在値しか保持していないため、変更前後の値や変更回数を復元することはできません。

そのため、履歴化する場合は、受注キー4カラムを基準に、変更が発生したレコードだけを蓄積する「最新＋状態遷移履歴」方式を優先して検討するのが現実的です。

ただし、変更検知に使用する項目、削除の検知方法、履歴対象カラム、保持期間を確認したうえで、最終的な履歴方式を決定する必要があります。

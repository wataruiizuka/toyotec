# 5. Kuduテーブルであることの確認

DDLには、次の記載があります。

```sql
STORED AS KUDU
```

したがって、`TBBA001G`はKuduテーブルであることが確定しました。

Kuduは、Cloudera環境で次のような処理に適しています。

- 主キーを使った高速検索
- レコード単位の追加
- レコード単位の更新
- レコード単位の削除
- ImpalaからのSQL参照

ただし、Kuduが更新可能であることと、変更前の値が自動的に履歴として残ることは別です。

同じ主キーに対して`UPDATE`や`UPSERT`が行われた場合、最新値だけが残り、変更前の値が失われる可能性があります。

現時点では、実際にどの処理が行われているかは確認できていません。

確認対象：

```text
INSERT
UPDATE
UPSERT
DELETE
```

# 6. 外部テーブルであることの確認

DDLの先頭は次のとおりです。

```sql
CREATE EXTERNAL TABLE ai21rep_pr_kudu.tbba001g
```

したがって、Hive Metastore上では外部Kuduテーブルとして管理されています。

テーブルプロパティには次の設定があります。

```text
'TRANSLATED_TO_EXTERNAL'='TRUE'
'external.table.purge'='TRUE'
```

## 6.1 PR環境での重要な注意点

`external.table.purge = TRUE`が設定されているため、`DROP TABLE`を実行すると、Metastore上の定義だけでなく、Kudu側の実データも削除対象になる可能性があります。

そのため、PR環境では次のSQLを実行しません。

```sql
DROP TABLE ai21rep_pr_kudu.tbba001g;
```

調査中は、次の更新系処理も実行しません。

- `INSERT`
- `UPDATE`
- `UPSERT`
- `DELETE`
- `TRUNCATE TABLE`
- `ALTER TABLE`
- `DROP TABLE`

当面は、次の参照系SQLだけを使用します。

- `SELECT`
- `SHOW`
- `DESCRIBE`
- `EXPLAIN`

# 9. 主キーと車両IDは分けて考える

`TBBA001G`の複合主キーは、受注レコードを識別するキーである可能性が高いです。

```text
受注キー
＝ cd_hansya
＋ cd_kaisya
＋ no_cyumon
＋ no_cyumoned
```

このキーが、車両1台を長期間追跡する車両IDと同じとは限りません。

想定されるケース：

- 受注時点では車台番号が未確定
- 取消後に新しい受注番号が発行される
- 注文内容変更で枝番が変わる
- 車両割当が変更される
- 同じ車両に対して受注が再作成される
- 売上戻し後に再計上される

履歴設計では、少なくとも次の3種類を分ける必要があります。

| キー | 役割 |
|---|---|
| 受注キー | 新車受注レコードを識別 |
| 統合車両ID | 車両を販売後も追跡 |
| 履歴版キー | 同一受注の変更前後を識別 |

# 11. HASHパーティション

パーティションは次のように定義されています。

```sql
PARTITION BY HASH (
    cd_hansya,
    cd_kaisya,
    no_cyumon,
    no_cyumoned
)
PARTITIONS 12
```

4カラムの複合主キー全体からハッシュ値を計算し、データを12個のKuduパーティションへ分散しています。

```text
受注主キー
   ↓
ハッシュ値を計算
   ↓
12パーティションへ分散
```

確認できた内容：

- 主キー全体がHASH対象
- HASHパーティション数は12
- RANGEパーティションはない
- 日付パーティションはない

# 12. 履歴設計におけるパーティション上の注意

現在値テーブルとしては、受注キーで分散する構成は妥当と考えられます。

一方、新規履歴テーブルでは次の検索が増える可能性があります。

- 特定日時点の受注状態
- 月別の受注変更件数
- 一定期間内の取消
- 車両単位の変更履歴
- 古い履歴のアーカイブ

そのため、履歴テーブルでは次の組み合わせを検討する可能性があります。

```text
受注キーまたは統合車両IDによるHASH分散
＋
履歴日付によるRANGE分割
```

ただし、実際のレコード件数、更新率、保持期間を確認する前に、物理設計は確定しません。

# 13. RANGEパーティションがない

DDLには`RANGE`パーティションの定義がありません。

したがって、次のような期間別の物理分割はされていません。

```text
受注日別
更新日別
登録月別
売上月別
納車月別
```

この構造は、時系列に履歴を追加していくテーブルよりも、主キー単位で現在状態を管理するテーブルに適した構成です。

ただし、RANGEパーティションがないことだけで履歴が存在しないとは断定できません。

# 16. 受注初期には車両番号が未確定の可能性

新車受注では、受注作成時点で車台番号やフレーム番号がまだ確定していない可能性があります。

```text
受注作成
  ↓
車両番号は未確定
  ↓
メーカー・工場から車両割当
  ↓
車台番号／フレーム番号を設定
```

そのため、履歴設計を最初から`no_frame`だけで管理すると、受注初期の履歴を車両へ紐付けられない可能性があります。

現時点では、次の二段階で考えるのが安全です。

```text
受注前半：
受注キーで追跡

車両割当後：
受注キーと統合車両IDを関連付ける
```

どちらも`STRING`で、NULLを許可しています。

名称からは、次のような比較に使われている可能性があります。

```text
CSV側レコードのハッシュ
Kudu側レコードのハッシュ
```

または、

```text
取込前のハッシュ
取込後のハッシュ
```

ただし、仕様は未確認です。

履歴設計へ利用する前に確認する内容：

- ハッシュ対象カラム
- カラムの連結順
- NULLの表現
- 空文字の表現
- 日付の文字列化形式
- 更新時に必ず再計算されるか
- `csv_md5`と`kudu_md5`の一致率
- 主キーを含むか
- 技術カラムを含むか

MD5が業務カラム全体から作成されている場合、変更検知に利用できる可能性があります。

一方、連携確認専用の値である場合、そのまま履歴設計へ流用できない可能性があります。

# 24. 履歴管理専用カラムの有無

一般的な履歴管理で利用する次のようなカラムは確認できません。

```text
valid_from
valid_to
is_current
is_deleted
operation_type
event_id
source_offset
history_version
```

一方、受注・登録・売上・納車・取消等の業務日付は多数あります。

そのため、`TBBA001G`は次のような構造である可能性が高いです。

```text
1行の受注レコード
├─ 受注日
├─ 条件変更日
├─ 登録日
├─ 売上日
├─ 納車日
└─ 取消日
```

これは、イベントごとに新しい行を追加する縦持ち履歴ではなく、同じ受注行に各業務状態を横持ちする構造です。

同一受注の値が更新された場合、変更前の値が残らない可能性があります。

# 26. `TBBA085G`との関係

## パターンC：一部の抹消データだけを持つ

# 34. 最終評価

`TBBA001G`は、**新車受注のライフサイクルを確認する中心テーブルとして適切**です。

受注、登録、売上、納車、取消などに関係する多数の項目があり、現在の新車受注状態を把握する主要ソースとして利用できる可能性があります。

一方、このテーブルだけで完全な履歴を実現できるとは判断できません。

理由：

1. 主キーは受注単位であり、車両IDではない  
2. 変更前後を管理する履歴専用カラムがない  
3. Kafka・CDCの操作情報がない  
4. 多数の日付はあるが、変更前の値を版管理している構造ではない可能性が高い  
5. 取消後のデータ移動先が未確認  
6. 更新日時とMD5の仕様が未確認  

現段階の位置づけ：

> **`TBBA001G`は、新車受注の最新状態および業務マイルストーンを取得する主要ソーステーブル候補です。完全な履歴を実現するには、`TBBA085G`、Kafka／NiFiのCDC情報、または新規履歴蓄積方式との組み合わせが必要です。**

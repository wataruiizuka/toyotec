# TOYOTEC 新アーキテクチャレビュー

## 1. 結論サマリー

この案は、方向性としてはかなり良いです。特に `OGG -> Kafka -> Flink -> Bronze(Iceberg) + Silver(Kudu) -> Gold(Iceberg/Star Schema)` の分離、Silver vs Oracle の完全性チェック、cutover をクリーン基線でやり直す判断は、現実のデータ基盤案件として筋が通っています。

ただし、無条件で賛成はしません。現時点で一番危ないのは、`Silver=Kudu` が設計の中核になっている一方で、基本設計書自身が `R1: 顧客SLA次第では Silver も Iceberg で統一可能` と書いている点です。つまり、最重要のアーキテクチャ決定がまだ未確定です。ここを決めずに詳細設計・見積・SOWへ進むと、構成が重くなりすぎるか、逆に要件を満たせない構成になります。

私の判断は「条件付き採用」です。MVPでは Kudu 前提で全部作り切るのではなく、まず latency SLA、OGG設定、完全性チェック、cutover、Gold検収条件をゲート化して、成立確認後に全242テーブルへ広げるべきです。

## 3. 良い点

1. **Bronze / Silver / Gold の役割分担が明確**
   Bronzeをappend-only履歴、SilverをOracle現状ミラー、Goldを業務ready層に分ける方針は妥当です。CDC履歴、最新状態、BI集計を同じテーブルに背負わせていない点は良いです。

2. **Silver vs Oracle の完全性チェックを正面から扱っている**
   SCN、Kudu MVCC snapshot、Oracle Flashback、Tiered hash比較まで踏み込んでいます。PoC止まりの基盤で一番揉める「本当にOracleと一致しているのか」に対し、設計上の回答を持っているのは強いです。

3. **論理削除・PK変更・TRUNCATEを逃げていない**
   `__is_deleted`、Dイベントのbefore->after正規化、TRUNCATEの販社単位DELETEなど、CDC案件で後から事故になりやすい論点を先に扱っています。ここは評価できます。

4. **cutoverで既存NiFi Silverを継承しない判断は現実的**
   クリーン基線を作り、差分発生時の責任をFlink/OGG側へ限定する設計は、検収と責任分界の観点で正しいです。

5. **Goldを旧114帳票のワイドテーブル再現にしない判断は良い**
   Star Schema化は、長期的な保守性・BI利用・共通ディメンションの観点で妥当です。旧SQLをそのまま再現しようとすると、技術負債を新基盤へ移植するだけになりがちです。

6. **スコープ外管理が別紙で存在する**
   BI移行、PII masking、NiFi廃止、FinOps、EOL対応、BCPなどを別管理にしている点は良いです。案件スコープを守る意識があります。

## 4. 悪い点・危ない点

| 優先 | 論点 | 問題 | 推奨 |

## 5. 推奨アーキテクチャ判断

私なら、現時点では次の条件で進めます。

**MVP構成**

- Bronze: Iceberg append-only。これは採用でよい。
- Silver: いったんKuduで代表テーブル検証。ただし本採用は latency SLA 合意後。
- Gold: Iceberg + Star Schema。旧114 SQL再現ではなく、業務質問に答えるモデルとして設計する。
- Streaming: Stage-1 Router + Stage-2 Flink SQL。方向性は良いが、242 job固定運用はMVP後に見直す。
- Reconciliation: Silver vs Oracleのみを正式な完全性チェック対象にする。ただしGoldには別の業務品質テストを必須化する。
- Orchestration: cronは避けたい。Cloudera環境でCDE/Airflow相当が使えるならそちらを優先。最低でもDAG、再実行、履歴、アラートを持つ。

**Kuduを採用する条件**

- Oracle更新からSilver参照まで p95 15秒程度が本当に必要。
- 利用クエリがPK point lookupまたは限定的な範囲参照に収まる。
- Kudu tablet設計、rebalance、監視、障害復旧を運用できる。
- BIの主要参照先はGoldで、Silver直参照は例外運用として統制できる。

**Kuduを採用しない、または縮小する条件**

- 業務側が1時間程度の鮮度でよい。
- Silver直参照のユースケースが明文化できない。
- 運用チームがKudu tablet運用・障害対応を持てない。
- Gold更新頻度を短縮すれば業務要求を満たせる。

## 7. 次アクション

1. **まずR1を潰す**
   顧客に「Silverの15秒級鮮度がどの業務で必要か」を確認してください。ここが曖昧なら、Kudu採用は過剰設計になる可能性があります。

2. **代表テーブルMVPを切る**
   5〜10テーブルで十分です。高TPS、TRUNCATEあり、PK変更あり、大容量、Gold主要Fact、重要マスタを混ぜて、OGG->Kafka->Flink->Kudu/Iceberg->Reconciliationまで通してください。

3. **SOW前にGoldの期待値を合わせる**
   旧114 SQLの完全再現を期待されると、Star Schema設計は炎上します。「業務質問ベースに再設計する」「検収サンプルで確認する」と先に握るべきです。

4. **セキュリティをPH1最低要件に引き戻す**
   全PII maskingをPH1でやる必要はありません。ただし、PII候補一覧、Ranger role matrix、監査ログ、秘密管理は最低限入れるべきです。

5. **運用テーブルを最初から作る**
   `ops.cdc_job_status`、`ops.gold_etl_checkpoint`、`ops.reconciliation_result`、`ops.schema_change_request` は最初から設計した方がよいです。あとから運用可視化を足すと、障害時に根拠が残りません。
